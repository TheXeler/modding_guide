添加作物

在 stonehearth mod 中，有生产可食用物品的作物、树苗（可放置物品）、草药和纤维资源（用作制作配方的原料）。但如果您愿意，您可以让它们生产其他类型的物品。

创建作物

  1. 首先让我们从 stonehearth mod 复制一个现有作物到我们的 mod。对于此示例，我们将复制芜菁作物。我们可以在 stonehearth/entities/crops 目录中找到所有作物。

  2. 现在像往常一样重命名我们复制的文件和目录。我们将为此示例制作一个番茄作物。

  3. 然后我们修改文件。作物的 JSON 文件定义了一个实体，并且必须具有以下组件：

    • "model_variants" : {} -- 这里的每个条目描述了在作物存在阶段使用的模型。它们可以范围从微小到完全生长，再到枯萎。您在此处为变体使用的名称将在文件的后面部分使用。我们还定义了一个"default"变体，可以被不关心作物阶段的其他脚本使用。

    • "mob" : {} -- mob 组件，这里我们定义 "model_origin" 以将模型居中于休耕田地上方，以及 "mob_collision_type" 为"clutter"，以便炉民可以穿过它们。

    • "stonehearth:crop" -- 这是作物组件。它需要以下数据：

      • "resource_pairings" : {} -- 获取阶段名称并将它们映射到如果在该阶段收获作物时的输出。我们将完全生长作物的模型变体名称映射到输出物品的别名，并将腐烂变体的名称映射到空字符串，以便在收获时它只是被销毁。
      • "harvest_threshhold" -- 定义作物在农民自动收获之前必须达到的阶段。
      • "plant_overlay_effect" -- 已弃用的字段，作物组件不再使用，可以忽略。

    • "stonehearth:growing" -- 这定义了生长发生的频率（将作物从一个阶段移动到下一个阶段）以及在每个阶段要更改的描述/模型。

      • "growth_period" -- 生长阶段之间必须经过的时间量。由 Xn 定义，其中 X 是数字，n 是 m 表示分钟，h 表示小时，d 表示天（游戏内时间）。
      • "preferred_seasons" : {} -- 每个生物群落的作物首选生长季节列表。键是生物群落别名，值是包含我们作物在该生物群落中的首选季节名称的字符串数组。在非首选季节种植作物会使它们生长得更慢。
      • "growth_stages" -- 一个 JSON 对象数组，每个对象包含描述每个生长阶段的 3 个属性："model_name"（引用上面定义的模型变体）、"name" 和 "description"（当我们点击作物模型时将在单位框架中出现的本地化字符串）。
        注意成熟阶段和腐烂阶段之间的空对象（{ }）。它们用于在成熟阶段等待 N 次 * growth_period，因此您有足够的时间在它们腐烂之前收获它们。您可以根据需要添加/移除任意数量的它们。
      • "growth_check_script" -- 定义一个可选文件（例如："file(corn.lua)"），其中包含"growth_check"函数，可以在达到每个生长阶段时调用该函数以查看我们是否能够进入下一个阶段。只需创建一个包含具有此签名的函数的类的 Lua 脚本 growth_check(entity, target_growth_stage, growth_attempts) 并返回一个表示阶段的数字（检查 growing 组件的 _grow 函数以了解如何使用）。
        如今，游戏中没有作物使用此字段，因为有许多其他地方可以影响作物生长（天气/季节、城镇奖励...），因此此选项增加了微管理并且难以平衡，但它可以用于实现诸如土壤肥力（生产更好的作物、添加另一个 resource_pairing）、只在夜间生长的作物等。

    对于"entity_data"，我们将保留 "net_worth" 不变，因为此文件代表作物在农田中时（不可"购买"/"出售"），并且 "value_in_gold" 将计入我们城镇净值的农业分数，因此我们将其保留为 1（否则净值可能会增加得太快，因为每个作物都会增加分数）。

    对于目录，我们将确保名称和本地化字符串的描述指向我们的 mod（这是在选择农场田地时出现在农民作物列表中的名称和描述），并且材质标签包括"crop"。

  4. 我们根据希望作物拥有的阶段数量编辑我们的 QB 模型，并为图标截图，该图标将显示在 UI 的作物列表中。

  5. 最后，我们在清单中为我们的作物添加别名：

    ...
"aliases": {
   "crops:tomato_crop": "file(entities/crops/tomato_crop)"
},
...

创建成熟产物

作物在收获时会根据"resource_pairings"映射中指定的内容生产不同的物品。

由于我们在示例中添加的是番茄作物，现在我们将创建其产物，一篮番茄。

  1. 我们将复制芜菁食物物品。所有可食用物品都在 stonehearth/entities/food 内。

  2. 像往常一样，我们将重命名文件，编辑 QB 模型并为 UI 图标截图。

  3. 让我们检查 tomato_basket.json 文件：

    • 它有一个 mixin（"mixins": "stonehearth:mixins:food_decay"）。此 mixin 内的属性将被 food_decay 服务使用，使篮子在一段时间后腐烂（并最终消失）。

    • 它有两个 model_variants，一个用于正常篮子，另一个用于篮子腐烂时。如果需要调整模型原点，可以选择性地添加 "mob" 组件。

    • 它有一个 stacks 组件：

      "stonehearth:stacks": {
   "max_stacks": 10
}

      这是在篮子准备好被携带到堆放处供消费之前必须放入篮子的作物数量。在这种情况下，炉民将在放下篮子之前收获 10 个番茄作物（如果可能）。
      堆叠数不适用于制作。在制作原料中，重要的是物品的数量。

    • 在 entity_data 中，我们像往常一样有 net_worth 数据，以及中性的 appeal（0）。

    • catalog 包括 "is_item"、一个 "category"、许多 "material_tags" 以确保正确标记和补货，以及一个额外的键 "subject_override"。

      此键指向一个将在炉民交谈时使用的别名。它将使他们不是谈论一篮番茄，而是谈论番茄作物（思想气泡中将出现番茄作物的图片，而不是番茄篮子的图片）。

      我们编辑本地化键、材质标签和 subject override 以指向我们的 mod。

    • 然后我们有 "stonehearth:food_container" 信息，因为此物品是一个食物容器：

      • "effect" 指向从篮子中检索物品时将播放的效果文件的名称。
      • "food" 指向 serving 的 JSON 文件的别名（见下文）。
      • "stacks_per_serving" 是每当我们从中获取一份食物食用时将从篮子中减少的堆叠数。如果剩余堆叠数为 0 或更少，篮子将消失。

  4. 最后，不要忘记在清单中为我们的番茄篮子添加别名：

    "aliases": {
   "crops:tomato_crop": "file(entities/crops/tomato_crop)",
   "food:tomato:tomato_basket" : "file(entities/food/tomato/tomato_basket)"
},

    别名必须与我们在作物 json 文件中为 resource_pairings 添加的内容匹配。

创建一份食物

通常，作物生产的食物篮子随后用作餐点的服务容器。 任何可以食用的食物物品（不仅仅是烹饪原料）必须有一个与之关联的 serving 实体（在 entity_data 的"stonehearth:food_container"部分中定义）。

  1. 当我们复制 stonehearth/entities/food/turnip 文件夹时，我们已经复制了 serving 文件。现在我们将进行通常的重命名和编辑。

    tomato_serving.json 文件没有很多属性，这些是在 "entity_data" 下之前未解释过的新属性：

    • "stonehearth:item" -- 此数据在几个地方使用，如库存控制器、资源调用处理程序和 carry block 组件。它有几个属性：
      • "category" 字段实际上并未使用，因为"category"现在是目录数据的一部分。
      • "destroy_on_uncarry" 标志，确定如果此物品掉落到地面将被销毁。
    • "stonehearth:food" -- 这里我们有 serving 的只读数据：
      • "stonehearth:sitting_on_chair" 和 "default" -- 在它们内部，我们有一个 "satisfaction" 的数字，这是当炉民坐在椅子上和坐在地上时，serving 将填满炉民胃部的程度（卡路里）。
      • "eating_effect" -- 可选字段。我们可以在此声明自定义效果（如果我们不这样做，炉民将默认使用"eat"效果）。如果我们想定义不同的动画，例如用于饮料，则很有用。

  2. 我们在清单中为我们的 serving 添加别名，以便我们可以在篮子的 JSON 文件中使用它。现在 aliases 部分看起来像这样：

    "aliases": {
   "crops:tomato_crop": "file(entities/crops/tomato_crop)",
   "food:tomato:tomato_basket" : "file(entities/food/tomato/tomato_basket)",
   "food:tomato:tomato_serving": "file(entities/food/tomato/tomato_serving)"
},

将我们的作物添加到可用作物列表

  1. 对 all_crops.json 文件添加 mixinto：

    "mixintos": {
   "stonehearth/services/server/farming/data/all_crops.json" : "file(services/server/farming/data/all_crops.json)"
}

    mixinto 看起来像这样：

    {
   "crops": {
      "tomato": {
         "crop_type": "more_crops:crops:tomato_crop",
         "ordinal": 1,
         "level_requirement": 1
      }
   }
}

    我们添加新作物的别名、作物列表的 ordinal，以及农民解锁我们作物所需的等级要求。在此示例中，我们使用了等级要求 1，因此它将默认解锁。

  2. 我们对 services/server/farming/data/initial_crops.json 添加 mixinto：

    "mixintos": {
   "stonehearth:farmer:all_crops" : "file(services/server/farming/data/all_crops.json)",
   "stonehearth:farmer:initial_crops" : "file(services/server/farming/data/initial_crops.json)"
}

    mixinto 看起来像这样：

    {
   "crops_by_kingdom": {
      "stonehearth:kingdoms:ascendancy": {
         "tomato": true
      },
      "rayyas_children:kingdoms:rayyas_children": {
         "tomato": true
      },
      "northern_alliance:kingdoms:northern_alliance": {
         "tomato": true
      },
      "another_kingdom:kingdoms:another_kingdom": {
         "tomato": true
      }
   }
}

    我们可以为我们想要的任何王国添加它，即使玩家可能没有安装添加它们的 mod。它将在选择王国时应用，因此在这里我们可以决定我们的作物是否默认在作物列表中，或者该王国是否必须用种子解锁它等。

    如果我们不添加此 mixinto，我们将必须提供其他方式让玩家解锁我们的作物。

  3. 最后，让我们测试我们的作物是否可以种植，它是否生长，以及是否可以正确收获（如果是可食用作物，还可以食用）。

    为了更快的测试，我们可以选择一个炉民并在默认控制台中使用 promote_to farmer 命令。

    如果我们启用了 debugtools mod，我们还可以 shift+点击炉民并点击"Make Hungry"，以便他们尝试从我们的篮子中吃东西。我们还可以通过按住 shift 并点击作物，然后点击"grow"来强制作物生长。

解锁作物

当我们不将作物添加到 initial_crops 列表时，有几种方法可以手动解锁作物，尽管底层使用的代码基本相同。

  • 通过遭遇战解锁作物。示例将是 Amberstone 战役中的传家宝作物。它是一个类型为"script"的遭遇战，我们在该脚本中解锁作物。如果需要，您可以重用它。代码只是检索农民的 job info controller 并调用其 manually_unlock_crop 函数：

    local farmer_job = stonehearth.job:get_job_info(player_id, "stonehearth:jobs:farmer")
farmer_job:manually_unlock_crop("crop_id_from_all_crops_list")

  • 通过消耗品解锁作物。这些消耗品在单位框架中有一个命令来使用它们。在作物种子的情况下，它们会将作物添加到作物列表中，种子盒将消失。命令用于解锁作物的代码与上述遭遇战使用的代码相同，但参数不同，因为它也是一个为其他类型的消耗品调用的函数，因此它必须共享相同的名称和参数。

    我们可以通过从 stonehearth/entities/consumables/seeds 复制一个并修改文件来创建一盒种子。在根实体的 JSON 文件中，"entity_data"中有这部分，我们可以声明要解锁的作物：

    "stonehearth:consumable": {
   "script": "stonehearth:consumables:scripts:unlock_crop",
   "crop": "tomato"
}

    然后我们将通过遭遇战、战利品或为其制作配方等方式将种子盒交给玩家。

    如果我们想将它们添加到外国作物商人的可能奖励中，我们可以对 stonehearth/data/gm/campaigns/trader/arcs/encounters/returning_crop_trader_encounter.json 添加 mixinto（此文件在 rayyas_children mod 中是不同的，因此我们也可以对其添加另一个 mixinto）。

  • 通过在自定义 Lua 脚本中调用 farming 服务中的 add_crop_type 函数（传递会话和作物的别名作为参数）来解锁作物。这种强制添加作物的方式不推荐。游戏不使用它，它仅用于自动测试以进行快速测试。

使用 SHED 创建作物

步骤与克隆任何其他实体时相同。您需要在 SHED 中分别克隆作物、篮子和 serving 实体，然后手动添加 mixinto 到作物列表。
