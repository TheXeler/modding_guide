管线和上下文

在 Stonehearth 中，我们使用一个名为 Horde3D 的渲染器，管线和上下文是该渲染器的构造。管线是一个 XML 文件，描述了在一帧中何时渲染什么。

它通过文件中命令的顺序隐式指示何时渲染某物，并通过在给定命令的属性中指定上下文来指示什么被渲染。

您可以在 stonehearth/data/horde/pipelines 中找到它们。

例如，这是战争迷雾管线（fow.pipeline.xml）：

  <Pipeline name="FOW">
     <Setup>
        <GlobalRenderTarget id="FogOfWarRT" />
     </Setup>
     <CommandQueue>
        <Stage id="FogOfWar_RT">
           <SwitchTarget target="FogOfWarRT" />
           <ClearTarget colBuf0="true" depthBuf="true" col_A="0.33" />
           <DrawGeometry scene="0" context="HEIGHT" forceLodLevel="0" cached="false"/>
           <DrawGeometry context="FOW_RT_EXPLORED" forceLodLevel="0" cached="false"/>
        </Stage>
    </CommandQueue>
  </Pipeline>

让我们从中选取一个命令作为示例：

  <DrawGeometry context="FOW_RT_EXPLORED" forceLodLevel="0" cached="false"/>

这里，DrawGeometry 是命令，FOW_RT_EXPLORED 是上下文。其他属性是传递给 DrawGeometry 命令的参数。

上下文本质上是一个标签，用于告诉渲染器要绘制什么。在这种情况下，所有具有带有 FOW_RT_EXPLORED 上下文的材质的对象都将被渲染，并且将使用该上下文中为该材质指定的着色器进行渲染。

游戏中有两个主要管线，forward 和 forward_postprocess。forward_postprocess 管线是高质量渲染器。您可以通过覆盖 .xml 文件将自己的上下文添加到管线中（如果其他人也覆盖这些相同的文件，这可能会对其他 mod 的兼容性产生不良影响），或者只是在您的着色器中实现一个或多个现有上下文。

有关 Horde3D 管线的更多信息请参见这里。（请注意，Stonehearth 的 Horde3D 版本略有不同，但这 mostly 是正确的。）

渲染材质

在 stonehearth/data/horde/materials 内，我们有一堆以".material.json"结尾的 JSON 文件。这只是一个命名约定，就像我们的 cubemitter 文件（粒子效果）以".cubemitter.json"结尾一样。

这些材质定义了如何渲染游戏中的 3D 模型。

让我们看看 glowy_voxel.material.json：

  {
     "materials" : {
        "VOXEL" : {
           "name" : "materials/voxel.material.json",
           "inputs" : [
              {
                 "name" : "glossy",
                 "bind_to_material_input" : "glossy"
              }
           ]
        }
     },
     "shaders" : {
        "EMISSIVE" : {
           "state" : "states/voxel/replace.state",
           "shader" : "shaders/color_only.shader"
        },
        "EMISSIVE_ADD" : {
           "state" : "states/voxel/add.state",
           "shader" : "shaders/color_only.shader"
        }
     }
  }

这里有相对路径。如果我们将此文件复制到我们的 mod 并用于我们的某个实体，除非我们将它们更改为引用我们 mod 文件夹的绝对路径，否则这些路径将默认从 stonehearth/data/horde 目录中读取。

材质是可组合的。材质可以选择在 "materials" 条目内指定要包含在其中的"子材质"。所有子材质都被包含并扁平化（递归地）以创建新材质。

包含其他材质的材质仍然可以指定着色器，这些上下文将覆盖所包含的材质。

"shaders" 条目指向状态和着色器。所有 .shader 文件都用 GLSL 编写，仅包含一个顶点程序和一个片段程序。游戏中使用的着色器可以在 stonehearth/data/horde/shaders 中找到。

着色器状态（混合等）已与着色器声明分离，因此也可以重用（您可以在 stonehearth/data/horde/states 内找到它们）。

因此，材质声明正在实现的上下文，并绑顶着色器、着色器状态和统一输入以创建上下文。它还可以声明材质的统一输入，并将它们绑定到着色器统一变量。如果您愿意，这些材质级输入构成了材质的"接口"。来自 starfield.material.json 的示例：

  {
     "shaders" : {
        "STARFIELD" : {
           "state" : "states/voxel/stars.state",
           "shader" : "shaders/starfield.shader",
           "inputs" : [
              {
                 "name" : "brightness",
                 "default" : [1.0, 1.0, 1.0, 1.0],
                 "bind_to_material_input" : "brightness"
              }
           ]
        },
        "STARFIELD_DEFERRED" : {
           "state" : "states/voxel/stars_deferred.state",
           "shader" : "shaders/starfield.shader",
           "inputs" : [
              {
                 "name" : "brightness",
                 "default" : [1.0, 1.0, 1.0, 1.0],
                 "bind_to_material_input" : "brightness"
              }
           ]
        }
     },
     "inputs" : [
        {
           "name" : "brightness",
           "default" : [1.0, 1.0, 1.0, 1.0]
        }
     ]
  }

没有尝试处理来自不同材质的多个冲突着色器；获胜者是最终胜出的那个。

此外，您不能调整所包含材质的着色器统一变量；您只能调整材质的输入。将着色器统一变量默认值视为私有变量，将材质统一变量视为公共可覆盖变量。

让我们以 ghost_item.json 为例。这里，transparent_with_depth.material.json 在其"inputs"部分公开了"alpha"统一变量。我们将其重新布线以绑定到我们自己的"alpha"输入：

  {
     "materials" : {
        "SELECTABLE" : {
           "name" : "materials/selected.material.json"
        },
        "TRANSPARENT" : {
           "name" : "materials/transparent_with_depth.material.json",
           "inputs" : [
             {
               "name" : "alpha",
               "bind_to_material_input" : "alpha"
             }
           ]
         }
     },
     "inputs" : [
        {
           "name" : "alpha",
           "default" : [0, 0, 0, 0.3]
        }
     ]
  }

GLSL，着色器语言

GLSL 有很多内容，外面也有很多好的资源，所以我们不会在这里全面介绍。不过，我们将介绍基础知识，并指出您可以进一步了解的地方。

着色器是在 GPU 上运行以渲染三角形的程序。如上所述，Stonehearth 使用两种类型的着色器程序：顶点程序和片段程序。

顶点程序一次接受模型的一个顶点作为输入，并转换它们以供片段程序使用。典型的第一步是将顶点的位置转换到相机空间，以便它出现在屏幕上的正确位置。

color_only.shader 中的顶点程序如下所示：

  #include "shaders/utilityLib/vertCommon.glsl"

  uniform mat4 viewProjMat;

  attribute vec3 vertPos;
  attribute vec4 color;

  varying vec4 outColor;

  void main() {
     outColor = color;
     gl_Position = viewProjMat * calcWorldPos(vec4(vertPos, 1.0));
  }

它从每个顶点获取颜色并将其放入 outColor 以供片段着色器使用，并通过首先使用从 vertCommon.glsl 导入的辅助函数计算世界位置，然后乘以视图投影矩阵，将 vertPos 转换到相机空间。

片段着色器计算每个片段的颜色，其中片段基本上是屏幕上的一个像素。顶点着色器中的 varying 值在构成正在渲染的三角形面的片段上进行插值。因此，如果我们有 varying float x，并且一个顶点的 x 值为 1.0，而同一三角形中另一个顶点的 x 值为 0.0，则两者之间的片段将自动获得 x = 0.5。

color_only.shader 中的片段程序如下所示：

  varying vec4 outColor;

  void main() {
     gl_FragColor = outColor.rgba;
  }

这里我们说每个插值颜色都被分配给输出颜色，没有任何变化。

Stonehearth 中的 .shader 文件有三个部分：

  • [[FX]] 用于声明着色器的输入。
  • [[VS]] 用于声明顶点着色器。
  • [[FS]] 用于声明片段着色器。

可以在着色器中导入单独的 .glsl 文件，这对于在着色器之间共享 GLSL 代码很有用。

完整的 color_only.shader 文件如下所示：

  [[FX]]

  [[VS]]
  #include "shaders/utilityLib/vertCommon.glsl"

  uniform mat4 viewProjMat;

  attribute vec3 vertPos;
  attribute vec4 color;

  varying vec4 outColor;

  void main() {
     outColor = color;
     gl_Position = viewProjMat * calcWorldPos(vec4(vertPos, 1.0));
  }

  [[FS]]

  varying vec4 outColor;

  void main() {
     gl_FragColor = outColor.rgba;
  }

技术说明：在 Stonehearth 中，我们的着色器使用稍旧版本的 GLSL 编写，这意味着我们有一些已弃用的关键字，如"varying"和"attribute"。它们在兼容性上下文中仍然有效，更多信息可以在这里找到。

有关 GLSL 的更多信息，您可以深入阅读他们的文档这里。

测试着色器

您可以在 user_settings.json 中启用此属性，以便能够在不需要重启游戏的情况下测试对着色器和 cubemitter 文件的更改：

  "enable_renderer_file_watcher" : true

它应该与您的 user_id 处于同一级别，确保在需要的地方添加逗号。

此选项可能并不总是按预期工作。在某些情况下，您可能需要重新加载游戏才能使更改生效。注意由于渲染相关文件中的错误文件路径或无效值导致的崩溃。

制作新的渲染材质和着色器非常复杂，但如果您觉得冒险，可以随时复制现有着色器并进行修改。玩家可能拥有不支持某些属性的旧显卡，因此在创建着色器时要小心。

作为参考，Stonehearth 使用 OpenGL 4.1 进行高质量渲染，但低质量渲染器适用于较旧的驱动程序，因此您应该使用旧式 GLSL。您可以使用 #version pragma 在着色器顶部指定要编译的程序版本。请参阅此页面作为参考。但是，请注意，如果您使用 4.1 等最新版本独有的语法和函数，您的着色器可能在由旧 GPU 驱动程序编译时崩溃，因为它们不支持这些函数。

有时着色器会导致视觉故障，如果将受影响的体素分割到 QB 模型中的单独矩阵或层中，可以避免这种情况（对于仅将材质应用于其部分颜色的模型）。

在当前版本的游戏中，已知具有渲染材质的实体在尝试选择它们时不会显示轮廓。具有自定义渲染材质的模型部分将显示不正确的选择颜色。
