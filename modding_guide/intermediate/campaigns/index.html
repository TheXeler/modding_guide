什么是战役、弧线和遭遇？

战役是叙事引导的游戏事件（例如 goblin 战役、trader 战役等）。

它们被分为弧线，每个弧线都实现戏剧性/游戏性目的，并由遭遇组成。

遭遇是针对特定战役的通用、模块化游戏片段的实例。一个通用遭遇（如 spawn camp 或 show dialog）可以重新皮肤化或以特定战役为目标实例化，并与其他遭遇组合在一起，创建许多可能的叙事连贯的游戏挑战。

例如，goblin 战役向玩家抛出越来越多的敌人，由三个弧线（开始、游击和 finale）以及几个基本通用遭遇的多个 goblin 特定实例（spawn camp、wait、city raid 等）组成。

游戏主持者树

所有遭遇、弧线和战役都是节点（它们继承自 stonehearth/services/server/game_master/controllers/node.lua）并存在于树中，以 game master 控制器为根节点。你可以在 debugtools 的Campaign browser中可视化它们：

游戏主持者有战役（作为子节点），战役有弧线（Trigger、Challenge、Climax），弧线有遭遇。遭遇本身实际上使用可重用脚本（例如：create_camp_encounter.lua）和列在弧线索引文件中的特定节点的 JSON 数据（例如：ambient_threats_arc.json）来运行内容，以自定义遭遇参数。

所有现有的战役、弧线和遭遇都可以在 stonehearth/data/gm 的子目录中找到。

入边和出边

每个遭遇节点都需要有我们称之为 "in_edge" 的内容，并且可以有零个、一个或多个 "out_edge"。

这些用于塑造定义战役流的图或树。in_edge 是一个自定义标识符，通常与遭遇文件同名，以便不同的节点可以触发相同的遭遇。来自 daily_report_encounter.json 的示例：

  "in_edge": "daily_report_encounter",

对于每个弧线的第一个遭遇，"in_edge" 必须是 "start"。每个弧线只能有一个具有此 in_edge 的遭遇。

对于 out_edges，语法会有所不同。示例：

  • Trigger one -- 在此遭遇执行完成后，它将仅触发一个遭遇（在这种情况下，in_edge 为 "create_camps" 的遭遇）：

    "out_edge" : "create_camps",

    另一个遭遇将从数组中的遭遇中随机选择一个触发：

    "out_edge": {
   "type": "trigger_one",
   "out_edges": [
      "00_sighting_intro",
      "00_sighting_intro_rc_na",
      "01_strange_trading_intro",
      "02a_hello_friend_intro",
      "11_shook_down_intro",
      "12_heirbloom_entry",
      "14_chase_scene_intro",
      "15_crystal_charging_intro",
      "21_residents_intro",
      "23_borrow_crafter_entry"
   ],
   "fallback": "quest_finish"
}

    "fallback" 键指向如果数组中的任何遭遇都无法触发（它们的 "can_start" 检查全部失败）时将触发的遭遇。它是可选的。

    这将执行相同的操作，但边将有权重，以便我们可以控制触发每个边的机会：

    "out_edge": {
   "type": "trigger_one",
      "out_edges": [
         {
            "type": "weighted_edge",
            "out_edge": "do_nothing",
            "weight": 5
         },
         {
            "type": "weighted_edge",
            "out_edge": "spawn_friendly_entities",
            "weight": 1
         }
      ],
      "fallback": "quest_finish"
}

  • Trigger many -- 此遭遇在执行完成后将同时触发数组中的所有遭遇：

    "out_edge": {
   "type": "trigger_many",
   "out_edges": [
      "check_honor_token",
      "wait_for_mountain_event",
      "wait_for_ogo_death"
   ]
}

  • Trigger sequence -- 这将触发一系列边，每次触发此出边规范时触发一个（所以通常你会在生成器遭遇或可以多次触发的遭遇中使用它）：

    "out_edge": {
   "type": "trigger_sequence",
   "out_edges": [
      {
         "out_edge": "encounter_1",
         "repeats": 6
      },
      "encounter_2"
   ],
   "repeat_last": false
}

    数组内的条目可以是出边的名称，如 "trigger_many" 类型，或者是具有 "out_edge" 和 "repeats" 字段的对象。这将使遭遇在序列中的下一个遭遇被触发之前重复 N 次。我们还可以选择在数组外定义 "repeat_last" 字段（默认为 false）。如果我们将其设置为 true，最后一个遭遇将在完成序列后每次此遭遇触发 out edge 时无限重复，否则序列将在完成最后一个遭遇后停止。

  • Destroy tree -- 这将销毁战役树中的一个分支（用于使已完成的任务不会使树变得杂乱）：

    "out_edge": {
   "type": "destroy_tree",
   "root": "generator_for_chance",
   "destroy_root": true
}

    我们将销毁从 "root" 中指定的边开始的路径，包括它取决于 "destroy_root" 字段。你可能会在游戏中看到类型为 "none" 的遭遇，它们只有一个这种类型的出边，用于在完成一系列遭遇后进行清理。使用此功能时要小心，你可能会删除一些你仍想保留的节点。

我们也可以混合类型，如下所示：

  "out_edge" : {
     "type" : "trigger_many",
     "out_edges" : [
        "trigger this edge always",
        {
           "type" : "trigger_one",
           "out_edges" : [
              "this edge has weight 1 (20% chance)",
              {
                 "type" : "weighted_edge",
                 "out_edge" : "this edge has weight 4 (80% chance)",
                 "weight" : 4
              }
           ]
        }
     ]
  }

为了表示当前弧线的结束，允许战役开始另一个弧线，我们使用 "arc:finish"：

  "out_edge" : "arc:finish"

此弧线的其他遭遇可能仍然存在，但它们应该自行解决。

某些遭遇将在其 info 部分中定义它们的出边，以便它们可以根据遭遇如何结束来控制将生成哪个遭遇，因此它们在 JSON 的根级别不会有 "out_edge" 属性。

有条件地启动遭遇

有一些遭遇用于检查条件本身，以便在满足条件时触发它们的出边。

但我们也可以在任何遭遇 JSON 文件中添加检查，以允许仅在特定条件下启动遭遇。

这些添加在 "can_start" : {} 部分中，通常在入/出边下方。例如：

  "can_start": {
     "test_1": {
        "game_mode_check": {
           "type": "deny_if",
           "item": "game_mode",
           "value": "stonehearth:game_mode:peaceful"
        }
     }
  }

这里，"test_1" 是测试集的自定义标识符。如果我们有多个集，当任何一个通过时，遭遇将启动。我们可以在一个测试中有多个检查，在这种情况下，只有当所有检查都通过时，遭遇才会启动。

我们可以为检查使用任何名称（上面示例中的 "game_mode_check"），因为重要的是内部的属性：

  • "type" -- 以下之一：

    "deny_if" / "deny_if_not" / "deny_if_less_than" / "deny_if_greater_than" / "deny_if_between" / "deny_if_not_between"

  • "item" -- 我们正在比较的东西。它可以是以下之一：

    • "kingdom" -- 我们将比较玩家王国的 URI（"value" 将是王国的 URI）。
    • "net_worth" -- 我们将比较玩家的净资产值。这已弃用，请改用 "score"。
    • "score" -- 我们将比较玩家的某个分数字段。在这种情况下，我们将添加一个 "score_type" 字段，以指定分数名称（例如 "military_strength" / "net_worth"）。
    • "num_citizens" -- 我们将比较玩家当前的市民数量。
    • "days_elapsed" -- 我们将比较自游戏开始以来经过的天数。
    • "reached_citizen_cap" -- 我们将检查玩家是否已达到其市民上限（"value" 将是 true 或 false）。
    • "biome" -- 我们将比较当前生物群系的 URI（"value" 将是生物群系的 URI）。
    • "game_mode" -- 我们将比较当前游戏模式的 URI（"value" 将是 "stonehearth:game_mode:peaceful"、"stonehearth:game_mode:normal" 或 "stonehearth:game_mode:hard"，假设是原版游戏）。
    • "hostility" -- 玩家是否对此节点的上下文中的 "npc_player_id" 敌对。"npc_player_id" 是在某些类型的遭遇（例如 create_camp）中使用的字段，并将成为在同一分支上的节点之间共享的上下文的一部分。
    • "time_of_day" -- 我们将比较当前时间（"value" 将是大于或等于 0 且小于 24 的数字）。
    • "exists_in_world" -- 我们将检查物品是否放置在世界中并为玩家所有。对于此类型，我们需要添加一个 "uri" 字段以指定物品的 URI，"value" 将是 true 或 false。
    • "campaign_completed" -- 我们将检查玩家是否完成了给定的战役。我们需要添加一个 "campaign_name" 字段，其中包含在 gm_index.json 文件中定义的战役名称（"value" 将是 true 或 false）。
    • "script" -- 运行指定 Lua 脚本的 start() 函数（我们将添加一个额外的 "script" 字段指向它）并将其返回值与指定的 "value" 字段进行比较。
    • "number_active" -- 我们将检查给定节点的多少个实例当前处于活动状态（即，"start_node" 被触发的次数减去 "end_node" 触发的次数，如果我们不指定 "start_node"，它将默认为 "in_edge" 的值）。
    • "number_spawned" -- 检查给定节点被触发了多少次（我们将添加一个 "node_name" 字段，其中包含在弧线的 JSON 文件中定义的节点 ID）。
    • "city_tier" -- 我们将比较玩家城镇的城市等级（对于原版游戏，"value" 通常是 1、2 或 3）。
    • "highest_job_level" -- 检查具有特定职业的 Hearthlings（我们将添加一个 "job_alias" 字段以指向职业 URI）并返回他们的最高等级，以与 "value" 进行比较。
    • "config" -- 我们将比较任意配置变量的值。这些可以在 stonehearth.json、user_settings.json 或命令行中指定。我们将添加一个 "key" 字段，其中包含配置变量的名称。
    • "counter" -- 比较任意计数器变量的值。计数器变量是战役范围的，并使用 set_counters 遭遇设置。我们将为此处的计数器变量名称添加一个 "key"。
    • "weather" -- 我们将比较当前天气的 URI（"value" 将是天气的 URI）。
    • "and" -- 如果所有子测试都通过，则返回 true。我们将添加一个包含检查的 "tests" : {} 字段。
    • "or" -- 与 "and" 相同，但如果任何子测试通过则返回 true。
    • "none" -- 这用于测试目的，检查将始终通过。

  • "value" -- 我们将使用 "type" 中的条件将 "item" 返回的值与此处的值进行比较。如果值应该是数字，我们还可以指定一个范围（例如，对于 "deny_if_between" 检查），如下所示：

    "value" : { "min" : 0, "max" : 10 }

我们为检查使用的自定义名称可以用于游戏模式调整覆盖。如果你创建了自定义遭遇类型，你也可以在其中有一个自定义的 can_start 函数。

规划你的战役

你可以制作许多类型的遭遇，从随着玩家选择而演变的故事情节，到随机环境遭遇，如 stonehearth 模组中的 ambient threats。

重要的是你要仔细计划你想要添加到战役中的内容并逐步处理，因为根据遭遇类型，创建遭遇链可能涉及许多文件。

一旦你知道你的战役将如何发展，想想可用的不同遭遇类型中哪一个最适合你的遭遇，如果没有一个有用，创建你自己的可重用遭遇类型，或者使用script encounter代替。

重用现有的遭遇类型允许我们仅使用 JSON 文件创建战役，这很容易。对于 script encounters 或自定义控制器，你需要使用 Lua。

如何创建战役

要创建战役，你可以复制 stonehearth/data/gm/gm_index.json 文件并编辑它。

确保将此别名添加到你的清单中：

  "aliases" : {
     "data:gm_index" : "file(data/gm/gm_index.json)"
  }

然后向 stonehearth 模组的 gm index 添加 mixinto：

  "mixintos" : {
     "stonehearth:data:gm_index" : "my_mod:data:gm_index"
  }

如果你想从 stonehearth 模组中移除战役，以便在你的模组启用时它们不运行，你可以使用 mixintypes，但请记住，它可能会影响假设那些战役将运行的其他模组。

你的 gm_index.json 文件将看起来像这样（来自 northern_alliance 模组的示例）：

  {
     "type": "index",
     "campaigns": {
        "hunting": {
           "animal_camps": "file(campaigns/hunting/animal_campaign.json)"
        },
        "town_progression": {
           "na_town_progression": "file(campaigns/town_progression/town_progression_campaign.json)"
        }
     }
  }

"campaigns" 下的每个键都是一个自定义标识符（战役子类型），并包含一个指向战役索引文件路径的战役节点名称。

如何创建弧线

创建战役文件后，是时候编辑它了。继续上面的示例，这是来自 animal_campaign.json：

  {
     "type": "campaign",
     "rarity": "common",
     "can_start": {
        "test_1": {
           "kingdom_check": {
              "type": "deny_if_not",
              "item": "kingdom",
              "value": "northern_alliance:kingdoms:northern_alliance"
           }
        }
     },
     "arcs": {
        "trigger": {
           "migratory_animals": "file(arcs/migratory_animals_arc.json)"
        },
        "challenge": {},
        "climax": {}
     }
  }

我们可以有 can_start 检查以防止战役在此处启动。"rarity" 实际上并未实现。

然后我们有一个 "arcs" 部分。弧线从 "trigger" 进展到 "challenge" 再到 "climax"。只有 "trigger" 弧线是必需的。许多战役，如 ambient threats 和 trader 战役，只使用 trigger 弧线。

我们为每个弧线定义一个弧线文件，带有一个用于调试的标识符名称，指向弧线索引 JSON 文件。它应包含的内容示例（来自 kitties 战役）：

  {
     "type": "arc",
     "rarity": "common",
     "encounters": {
        "kitty_generator": "file(encounters/kitty_generator.json)",
        "spawn_koda": "file(encounters/spawn_koda.json)",
        "spawn_jacko": "file(encounters/spawn_jacko.json)",
        "do_nothing": "file(encounters/do_nothing.json)"
     }
  }

"type" 是 "arc"。在 "encounters" 内，我们列出我们将在战役中使用的所有遭遇 JSON 文件（在创建它们时添加它们）。按照惯例，遭遇的标识符只有小写字母、数字和下划线。尽量使用简短的描述性名称。

如何创建遭遇

遭遇只是 JSON 文件，如战役和弧线索引。我们可以将 GM 文件组织在子文件夹中，以便每个文件夹包含不同弧线的遭遇等。

一般来说，遭遇文件将具有以下內容：

  {
     "type": "encounter",
     "encounter_type": "create_camp",
     "rarity": "common",
     "in_edge": "spawn_friendly_entities",
     "out_edge": {
        "type": "trigger_many",
        "out_edges": [
           "spawn_enemy_entities",
           "wait_for_duration"
        ]
     },
     "create_camp_info": {
     }
  }

如前所述，"rarity" 实际上并未使用。

  • "type" : "encounter"

  • "encounter_type" -- 这是现有的类型之一，或者如果你创建了自定义类型。

  • "in_edge" -- 必需字段，这是此遭遇的标识符，通常与弧线索引中此文件的标识符匹配。

    对于每个弧线，你应该有一个 in edge 名为 "start" 的遭遇。这将是弧线开始运行时触发的遭遇。

  • "out_edge" -- 可选字段，如之前解释的出边。

  • "unique" -- 可选字段（布尔值）。当为 true 时，一次只能运行一个此类型的遭遇。

  • "can_start" : {} -- 可选字段。如之前解释的，这些是防止运行此遭遇的标志。

  • "mixins" -- 可选字段。Mixin 的 URI 或 URI 数组，以便你可以在遭遇中重用信息。如果你为遭遇文件嵌套 mixins，请小心，有时最好将所有信息放在眼前。

  • "encounter_info" -- 此字段的名称会有所不同，通常是遭遇类型，末尾附加 _info。将包含此遭遇类型所需的所有数据。

如何创建你自己的遭遇类型

你可以在你的模组中创建一种新的可重用遭遇类型（与script遭遇非常相似，但更有组织）。要做到这一点：

  • 创建一个控制器并将其添加到清单的 "controllers" 部分下
  • 控制器的键必须命名为 "game_master:encounters:your_custom_type"
  • 在遭遇的 JSON 文件中使用它时，你必须添加一个名为 "mod_root" 的额外键，与 "encounter_type" 处于同一级别，其值应为你模组的命名空间。

所有遭遇（以及从遭遇启动的所有脚本）都是控制器。更多关于控制器的内容在指南的此部分中。所以如果需要，你可以使用生命周期函数。这些控制器需要有 start(ctx,info) 函数，也可能有其他函数，如 stop()、suspend()/continue() 或 can_start(ctx)，除了你创建的任何自定义函数。

你可以在 stonehearth/services/server/game_master/controllers/encounters 中找到现有遭遇类型的控制器。stonehearth/services/server/game_master/controllers 的其他子文件夹中也有用于遭遇的其他脚本。

如何使用 debugtools 测试你的战役

请参阅关于Campaign Browser的部分以了解如何使用它。

使用 SHED 编辑战役

SHED 有一个很好的工具用于创建遭遇。在编写本指南时，你必须在使用它之前准备 gm_index.json、战役和弧线 JSON 文件。

  1. 在你的模组中添加一个指向你的 gm_index 的别名。

    "aliases": {
   "data:gm_index" : "file(data/gm/gm_index.json)"
}

    别名必须完全是 "data:gm_index"，否则 SHED 将无法找到它。另外，记得向 stonehearth 的 gm_index 添加 mixinto，以便你可以在游戏中测试你的战役。

  2. 打开 SHED 并点击 "Encounter Designer" 选项卡。

    你应该看到一个列出你文件夹中准备好的所有模组的战役的树。点击其中一个战役，你应该在右侧看到一个显示战役样子的图表：

    你可以使用鼠标滚轮放大/缩小图表，并可以使用第三个鼠标按钮在图表周围平移（按下鼠标滚轮，按住它，然后移动鼠标）。当你将鼠标悬停在子树上时，你可以看到它们在图表中被突出显示。

    每当你更改节点时，记得保存更改并点击右上角的 "Refresh Encounters" 按钮，以便图表更新（例如，假设你创建了一个新遭遇/节点，它将未附加到图表。但如果你编辑另一个节点使其指向你的新节点作为 out_edge，那将不会反映在图表中，直到你重新加载/刷新）。有时它会自动刷新，例如删除节点时。

  3. 要创建遭遇：在树/图表的空白部分右键单击，转到 'Add New Node->' 并选择你要添加的遭遇类型（仅支持默认遭遇类型，自定义类型不会出现）。

    然后，将出现一个对话框。转到你选择的遭遇文件夹（它可能指向错误的弧线）并为你的新 JSON 文件写一个名称。建议文件名使用小写字母且没有空格或符号。不过，它可以包含下划线和数字。

    点击 'Save' 后，JSON 文件将被创建，遭遇将被添加到遭遇列表中（如果你有多个弧线，它不会，但在图表中右键单击节点将显示一个选项，将其父级化到你选择的弧线）。如果你没有看到它被添加，你可能需要刷新遭遇。

    创建后，选择其节点并更改一些默认值以消除文件中的错误，如入/出边等。如果这是你的第一个遭遇，确保给它一个 "start" 的入边，以便它在刷新后父级化到弧线节点并开始塑造树。

    你也可以通过上下文菜单操作（右键单击遭遇节点）克隆或删除遭遇。它将自动向弧线 JSON 文件添加/移除遭遇。删除时，它将真正删除 JSON 文件（与在 Manifest 选项卡中移除别名不同），所以确保你真的想删除它。

你应该在 JSON 编辑器上方的栏中看到几个选项。第一个是在默认文本编辑器中打开当前文件（对于 JSON 文件），第二个是打开此文件的包含文件夹，第三个是保存，最后一个是本地化文件（记住，有时本地化可能不起作用）。如果你添加了 mixins，带有 mixins 的预览按钮也应该出现在最后。

保存时，如果你有一些不良缩进，JSON 将被很好地格式化。

SHED 中的 JSON 编辑器支持遭遇的自动完成和提示，这非常有用。如果你没有得到属性的下拉列表，尝试删除直到属性名的所有内容，然后输入 ':' 看看它是否出现。使用箭头键浏览选项并按 Enter 或双击确认选项：

如果文件左侧的颜色是绿色，一切正常。当它是红色时，SHED 不会让你保存文件（语法错误）。将鼠标悬停在左侧的箭头上将显示描述问题的工具提示。如果是橙色，可能缺少某些字段或 SHED 无法识别的一些数据，但会让你继续。当它是灰色时，意味着 SHED 没有检查此文件中的错误，因为它不知道它应该包含什么，所以你要对你写的内容负责。

这些是你可以在图表中看到的不同节点类型及其含义：

  • 蓝色节点：代表战役本身。点击一个，你应该看到相应的 campaign.json 文件，并且你将能够编辑它。如果需要，你可以在 Manifest 选项卡中编辑 gm_index 文件。
  • 青色节点：代表弧线。点击一个，你应该看到相应的 arc.json 文件，其中列出了该弧线中的所有遭遇。
  • 灰色节点：代表弧线起始或结束节点，以及一些遭遇的出边。它们没有关联的文件，它们只是指示选择或命名出边。
  • 绿色节点：遭遇节点。这些可以是任何类型。它们左侧有一个小图标，指示遭遇的类型。
  • 橙色节点：这些是generator遭遇。它们用于根据特定条件循环生成相同的出边。
  • 带有红色边框的节点：这些要么在其 JSON 中有错误，要么未在遭遇列表中指定。
  • 带有条纹背景的节点：它们有未保存的更改。
