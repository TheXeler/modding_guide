等待类遭遇战

等待

一种在触发输出边之前等待指定游戏时间长度的遭遇战。

  • "encounter_type" : "wait"

  • "wait_info" : {}

    • "duration" -- 一个时间表达式，表示我们想要等待的游戏内时间长度。
    • "game_mode_tuning" -- 可选字段。一个布尔值，用于为此遭遇战的 "duration" 参数启用调优覆盖。您可以在 stonehearth/data/game_mode/hard/hard.json 的 "encounter_tuning_overrides" 部分中查看示例（该字段在那里称为 "delay"）。在这种情况下，当玩家选择困难模式时会应用此修改器。
    • "destroy_on_trigger" -- 可选字段。一个布尔值，用于在触发输出边后销毁此节点。
    • "continue_on_disconnect" -- 可选字段。一个布尔值，用于在多人游戏中玩家（客户端）断开连接时继续运行此节点。默认为 false（暂停此遭遇战/计时器，并在玩家重新连接后恢复）。

等待特定时间

一种在触发输出边之前等待到给定游戏时间的遭遇战。

  • "encounter_type" : "wait_for_time_of_day"

  • "wait_for_time_of_day_info" : {}

    • "time" -- 带有可选随机元素的游戏内一天中的时间。例如："15:30" 表示'下午 3:30'。"5:00+1h30m" 表示'上午 5:00 到 6:30 之间'。
    • "continue_on_disconnect" -- 可选字段。一个布尔值，用于在多人游戏中玩家（客户端）断开连接时继续运行此节点。默认为 false（暂停此遭遇战/计时器，并在玩家重新连接后恢复）。

等待事件

一种在实体上等待事件发生后触发输出边的遭遇战。

  • "encounter_type" : "wait_for_event"

  • "wait_for_event_info" : {}

    • "source" -- 将监听事件的对象。通常是来自先前遭遇战并已注册到共享上下文中的实体。例如："wolf_pack_raid.wolves"。

      此属性也可以是数组，例如：

         [
      "spawn_enemy_entities.wolf_skeleton",
      "spawn_enemy_entities.skeletons",
      "spawn_enemy_entities.skeleton_giant"
   ]

      当所有来源都触发了事件时，输出边才会触发。请注意，这些条目也可以引用多个实体（例如，在之前的遭遇战中，我们创建了随机数量的骷髅兵，并将对每个骷髅兵的引用存储在 "spawn_enemy_entities.skeletons" 中）。

    • "event" -- 要等待的事件名称。例如："stonehearth:kill_event"。

    • "repeatable" -- 可选字段。布尔值，默认为 false。如果为 true，则在此节点被销毁之前持续监听并触发输出边。

    • "trigger_on_each" -- 可选字段。布尔值，默认为 false。如果为 true，且 "source" 引用多个实体，则当任何一个条目触发事件时就会触发输出边，而不是等待所有来源都触发事件。通常与 "repeatable" : true 配合使用。

等待净资产

一种在玩家达到指定净资产之前等待触发输出边的遭遇战。

  • "encounter_type" : "wait_for_net_worth"

  • "wait_for_net_worth_info" : {}

    • "threshold" -- 当净资产高于阈值时，将触发输出边。它可以是整数（不含小数）或范围，如果是范围则这样写：

      "threshold": {
   "min": 3700,
   "max": 4000
}

      将在该范围内随机选择一个数字，并将净资产与之比较。

    • "interval" -- 可选字段（默认为 "1h"）。一个时间表达式，表示检查净资产值以尝试触发输出边的频率。较低的检查频率对性能更有利。

    • "continue_on_disconnect" -- 可选字段。一个布尔值，用于在多人游戏中玩家（客户端）断开连接时继续运行此节点。默认为 false（暂停此遭遇战/计时器，并在玩家重新连接后恢复）。

等待条件满足

一种在一组条件满足之前等待触发输出边的遭遇战。

  • "encounter_type" : "wait_for_requirements_met"

  • "wait_for_requirements_met_info" : {}

    • "requirements" : {} -- 条件列表。定义时使用与 "can_start" 测试 内部相同的结构和字段。例如：

      "requirements" : {
   "city_tier": {
      "type": "deny_if_less_than",
      "item": "city_tier",
      "value": 2
   },
   "kingdom_check": {
      "type": "deny_if_not",
      "item": "kingdom",
      "value": "stonehearth:kingdoms:ascendancy"
   }
}

    • "recheck_delay" -- 可选字段，一个时间表达式。尝试触发输出边之间等待的游戏内时间，默认为 "1h"。

    • "max_checks" -- 可选字段。最大检查次数。例如，如果将此字段设置为 5，则在第六次尝试检查条件时节点将被销毁。

    • "cancellation_event" -- 可选字段。将在玩家的派系上监听的事件名称。如果在此节点仍处于活动状态时触发了该事件，节点将被销毁。

    • "continue_on_disconnect" -- 可选字段。一个布尔值，用于在多人游戏中玩家（客户端）断开连接时继续运行此节点。默认为 false（暂停此遭遇战/计时器，并在玩家重新连接后恢复）。

门控类遭遇战

生成器

一种在延迟后触发边的生成器，可选择循环重复。

  • "encounter_type" : "generator"

  • "generator_info" : {}

    • "delay" -- 一个时间表达式。触发 "spawn_edge" 之前等待的游戏内时间。每次触发边时，延迟计时器将重新启动。

    • "spawn_edge" -- 延迟后要生成的边的名称（例如，"city_raid"）。也可以引用多条边（使用与输出边的值相同的语法）。

    • "source_entity" -- 可选字段。在上下文中注册的对象的名称（例如，"goblin_raiding_camp_1.boss"）。如果源实体被销毁或不存在，此遭遇战将停止生成边。

    • "min_spawns" -- 可选字段。我们希望生成器触发边的最小次数（最小值为 1）。

    • "max_spawns" -- 可选字段。我们希望生成器触发边的最大次数（最小值为 1）。

      如果我们同时指定 "min_spawns" 和 "max_spawns"，将在它们之间随机选择一个数字作为生成限制。

      如果我们不包含其中任何一个，生成器将无限期地继续生成边（或者直到源实体被销毁，如果我们定义了该字段）。

      如果此生成器遭遇战也有输出边，它将在达到最大生成次数时触发（如果我们只定义了最小值则为最小值，如果我们同时定义了两者则为随机数）。

    • "game_mode_tuning" -- 可选字段。一个布尔值，用于为此遭遇战的 "delay" 参数启用调优覆盖。您可以在 stonehearth/data/game_mode/hard/hard.json 的 "encounter_tuning_overrides" 部分中查看示例。在这种情况下，当玩家选择困难模式时会应用此修改器。

    • "continue_on_disconnect" -- 可选字段。一个布尔值，用于在多人游戏中玩家（客户端）断开连接时继续运行此节点。默认为 false（暂停此遭遇战/计时器，并在玩家重新连接后恢复）。

物品阈值

一种在阈值满足之前等待触发输出边的遭遇战。只要这些物品符合某些材料或 URI，阈值将与玩家仓库中的物品数量进行比较。

  • "encounter_type" : "item_threshold"

  • "item_threshold_info" : {}

    • "threshold" -- 要比较的数量。也可以是范围，如下所示：

      "threshold" {
   "min": 800,
   "max": 1200
}

      在这种情况下，将在 "min" 和 "max" 之间随机选择一个数字进行比较。

    • "check" -- "less_than" 或 "greater_than"。用于比较阈值的条件。在内部，它们被视为"小于或等于"和"大于或等于"。

    • "materials" : [] -- 材料标签数组。如果我们定义此字段，当拥有此数组中所有材料标签的物品达到阈值时，遭遇战将触发其输出边。

    • "uris" : [] -- URI 数组。如果物品的 URI 在此数组中，则它们将匹配阈值。

    我们必须定义 "materials" 数组或 "uris" 数组。同时定义两者时只会考虑 "uris"。

标记类遭遇战

计数器

一种计算运行次数并将此信息保存在用户指定的 ctx.ctx_registration_variable 中的遭遇战。

如果它能够运行 X 次，则生成成功输出边。否则，生成失败输出边。两者都在 "counter_info" 的 out_edges 字段中指定。

  • "encounter_type" : "counter"

  • "counter_info": {}

    • "out_edges" : {} -- 包含将触发的输出边。例如：

      "out_edges": {
   "fail": "wait_for_next_shakedown",
   "success": "unlock_red_token_recipe"
}

      通常 "fail" 输出边最终会回到此节点的祖先节点，以便它可以再次运行，但也有其他方式可以再次运行同一节点。

    • "times" -- 在触发 "success" 边之前计数到此数量。在此之前，触发 "fail" 边。

    • "ctx_registration_variable" -- 用于存储当前计数的变量名称。例如："blue_tier_shakedown_success_counter"。

    • "optional_reference_entity" -- 可选字段，对之前在共享上下文中注册的女体的引用（例如，"create_blue_tier_camp.entities.loot_chest"）。仅当引用实体仍然存在且与上次相同时，遭遇战才会增加计数。如果它不存在，则遭遇战不执行任何操作。如果它存在，但与上次存储在上下文中的实体不同，计数器将重新开始。

设置计数器

更新战役范围内的计数器值，这些计数器用于通过 "counter" can_start 检查 来限制遭遇战。

  • "encounter_type" : "set_counters"

  • "set_counters_info" : {}

    • "counters" : {} -- 一个或多个键值对，其中键是我们想要创建或重用的计数器名称，值是数字或要计算的公式。公式是包含任何 Lua 表达式的字符串，其中 'x' 是计数器的当前值（默认为 0）。例如：

      "counters" : {
   "foo": 42,
   "bar": "x / 2 + 1",
   "quux": "stonehearth.calendar:get_seconds_since_last_midnight()"
}

不要将这些计数器与 "counter" 遭遇战混淆。它们有不同的用途。

工具类遭遇战

销毁实体

一种销毁特定实体的遭遇战。

  • "encounter_type" : "destroy_entity"

  • "destroy_entity_info" : {}

    • "target_entities" : [] -- 包含应被销毁的实体的注册路径（例如，"golem_raid.golems"）的数组。

    • "effect" -- 可选字段。实体被销毁时将播放的效果的 URI（例如："stonehearth:effects:poof_effect:grey"）。

    • "random_delay" : {} -- 可选字段。毫秒范围（"min" 和 "max" 值）。将在它们之间随机选择一个数量，实体将在等待该时间后消失。这样做的目的是让实体不会同时消失，也不会同时开始效果（如果指定了）。通常我们使用一个很小的值，小于一秒。

    • "continue_always" -- 可选字段（布尔值）。当为 true 时，即使没有销毁任何内容也会触发下一个遭遇战。当为 false 时，如果没有销毁任何内容则不会继续。

      如果实体可能在此遭遇战运行之前通过其他方式被销毁（例如，炉民可能已经杀死的 ambient threats 中的怪物），您通常希望将此设置为 true。

    • "destroy_all_registered_entities" -- 可选字段。布尔值，当为 true 时，遭遇战将销毁 ctx.entity_registration_paths 中的所有实体，而不是 "target_entities" 中指定的实体（因此如果您想销毁所有内容，可以使用此字段代替该字段）。这对于一段时间后清理怪物很有用，以节省计算机资源，因为新实体会在地图中生成。

    • "script" -- 可选字段。在销毁实体之前立即运行的脚本的 URI。脚本必须至少有一个 "start" 函数。例如：

      local GoblinCampDepartsScript = class()

-- When the camp departs, no matter how the player interacted with
-- these goblins, set amenity for all goblins back to hostile.

function GoblinCampDepartsScript:start(ctx)
   stonehearth.player:set_neutral_to_everyone('goblins', false)
end

return GoblinCampDepartsScript

      我们可以使用 ctx 参数访问遭遇战的上下文。请记住，如果您想使用别名而不是路径来引用脚本，必须在清单的 "controllers" 下添加它。

脚本

一种只运行关联脚本（作为控制器创建）的遭遇战。

  • "encounter_type" : "script"

  • "script_info" : {}

    • "script" -- 要运行的脚本的 URI。类似于上面的示例脚本，它必须有一个 start(ctx, data) 函数。ctx 参数将具有节点的上下文，data 参数将包含来自遭遇战 JSON 文件中 "data" 字段的数据。请记住，如果您想使用别名而不是路径来引用脚本，必须在清单的 "controllers" 下添加它。

    • "data" : {} -- 可选字段。将传递给自定义脚本的数据。

    • "continue_on_disconnect" -- 可选字段。一个布尔值，用于在多人游戏中玩家（客户端）断开连接时继续运行此节点。默认为 false（暂停此遭遇战并在玩家重新连接后恢复）。

      如果您不将 "continue_on_disconnect" 设置为 true，可能需要在自定义脚本中实现 suspend() 和 continue() 函数，以便游戏知道是否必须停止某个计时器、稍后恢复它等。它们将在玩家断开连接/重新连接时被调用。

对话类遭遇战

公告

一种显示通知公告的遭遇战。

  • "encounter_type" : "bulletin"

  • "bulletin_info" : {}

    • "type" -- "info"、"alert" 或 "quest" 之一。这会改变游戏中公告的外观。如果类型为 "alert"，它将与普通公告分开显示，并且在显示另一个公告时不会自动隐藏。新的警报将显示在旧警报的顶部。
    • "title" -- 公告的本地化文本。当玩家点击公告时，它将消失。如果您希望在此之后在窗口中显示其他消息，请查看对话树遭遇战。
    • "sticky" -- 可选字段（布尔值，默认为 false）。粘性公告通知会保留在 UI 中，直到用户处理它们。非粘性公告会在几秒后淡出，但仍可在公告列表中访问。
    • "zoom_to_entity" -- 可选字段。对上下文注册路径的引用。当用户点击公告时，如果该实体存在，摄像机将聚焦于此实体。
    • "send_to_all_players" -- 可选字段（布尔值，默认为 false）。当您希望公告出现在所有玩家面前时使用。这通常不需要，因为战役为每个玩家独立运行，但有时很有用（例如，这在泰坦战役中使用，因为它影响地图的大部分区域，经常干扰其他玩家的城镇）。
    • "active_duration" -- 可选字段（一个时间表达式）。如果我们指定此字段，即使玩家未与其交互，公告也将在此持续时间后自动移除。有助于防止公告列表中堆积可能不再相关的通知。

对话树

一种显示对话树的遭遇战，其选择最终可能导致不同的输出边。

  • "encounter_type" : "dialog_tree"

  • "dialog_tree_info" : {}

    • "start_node" -- 我们要开始的对话节点的键（例如，"msg1"）。

    • "source_entity" -- 可选字段。对来自另一个节点的对象的引用（例如，"goblin_raiding_camp_1.boss"）。仅当此源实体存在时，对话树才会开始。

    • "i18n_data" : {} -- 可选字段。用于设置对话的 i18n 变量数据的字典。可以是对 ctx 路径的引用或字面值。例如：

        "i18n_data" : {
     "boss_display_name" : "Emma",
     "town_name" : "town_name"
  }

      在 en.json 文件中，我们会像这样使用它们，例如：

        "msg01" : "The enemy called [str(i18n_data.boss_display_name)] appears",
  "msg02" : "They came with a goblin called [name(i18n_data.goblin)]."

      [str()] 和 [name()] 的区别在于，当我们想要引用已在上下文中注册的实体的名称（具有自定义名称的实体 - 如炉民、怪物和 NPC）时使用 [name()]。在这种情况下，我们不需要将其添加到 "i18n_data" 中。

      有些遭遇战会将变量添加到上下文中，我们也可以在我们的 "i18n_data" 中使用这些值（例如，城镇的名称已在 town_name 中注册，因为许多对话都希望调用它）。

    • "script" -- 可选字段。在对话开始之前运行的脚本的 URI（请记住，如果您想使用别名而不是路径来引用脚本，必须在清单的 "controllers" 下添加它）。

      类似于上面的示例，它必须有一个 start(ctx, info) 函数（我们将传递给它上下文以及可以从遭遇战 JSON 文件的 "dialog_tree_info" 下访问的所有信息）。一个例子是在对话运行之前初始化自定义 i18n 数据，例如：

       "i18n_data" : {
    "town_serial_number": "town_serial_number",
    "english_town_ordinal_suffix": "english_town_ordinal_suffix"
 }

      我们在遭遇战的 JSON 文件中声明了上述数据，但它们的值尚未在任何地方初始化。因此，我们创建并引用以下脚本，它将在上下文中注册这些变量的值：

       local TownProgressionI18NDataScript = class()

 function TownProgressionI18NDataScript:start(ctx)
    ctx.town_serial_number = stonehearth.town:get_town(ctx.player_id)
                                             :get_town_serial_number()
    ctx.english_town_ordinal_suffix = english_number_ordinal_suffix(ctx.town_serial_number)
 end

 function english_number_ordinal_suffix(number)
   local n = ""..number
   local ordinal, digit = {"st", "nd", "rd"}, string.sub(n, -1)
   if tonumber(digit) > 0 and tonumber(digit) <= 3 and
        string.sub(n,-2) ~= 11 and string.sub(n,-2) ~= 12 and
        string.sub(n,-2) ~= 13 then
     return ordinal[tonumber(digit)]
   else
     return "th"
   end
 end

  return TownProgressionI18NDataScript

      然后我们可以在对话树的文本中使用 [str(i18n_data.town_serial_number)]。

    • "choose_ctx_portrait" : [] -- 可选字段。包含对话头像 URI 的数组。将从这些提供的选项中选择一个，并且可以被设置了 "use_ctx_portrait" 并具有头像字段的此对话及后续对话引用。

    • "use_ctx_portrait" -- 可选字段（布尔值）。是否使用（当前或之前）"chosen_ctx_portrait"。这样我们可以为每次游戏选择不同的头像以增加多样性，但在任务期间保持不变。

    • "nodes" : {} -- 对话的节点。每个节点都是一个具有自定义标识键的对象（尝试使用有意义的名称来保持数据有序）。在每个节点内，我们有 "bulletin" : {} 字段，具有以下属性：

      • "title" -- 通知的本地化标题。我们需要它为所有节点提供，因为每个对话也将是公告列表中的一个条目，即使通知只弹出第一个。此遭遇战类型的公告类型将始终为 "quest"。
      • "dialog_title" -- 对话的本地化标题。将出现在对话框窗口的顶部。它是必需的，但如果您不想要它，可以将其定义为空字符串。
      • "portrait" -- 可选字段。对话头像的 URI。如果您在同一遭遇战或之前的遭遇战中注册了头像，并将 "use_ctx_portrait" 设置为 true，则可以将头像设置为空字符串（在一些现有的遭遇战中，您也会看到 "$ctx"）。

      这将确保显示头像并重复使用保存在上下文中的头像。请注意，使用注册的头像将使遭遇战的所有对话节点都具有相同的头像，只要它们中存在该字段。如果您重用头像，将无法将特定的头像更改为其他头像。

      • "portrait_offset" -- 可选字段。头像从顶部偏移的像素数。

      • "message" -- 当前节点消息的本地化文本。它也可以是本地化文本的数组，这样将随机选择一个。您可以在本地化文本中包含 i18n_data，以及一些 HTML 标签来为其添加格式。

      • "choices" : {} -- 玩家推进对话到下一个节点的一个或多个选择。它们的键是按钮的本地化文本，但不包含 "i18n()" 包装。它们的值是对象且会有所不同。例如：

         "choices" : {
    "my_mod_namespace:campaigns.abc.encounters.dialog_1.msg1.choice_00": {
       "next_node": "msg2"
    },
    "my_mod_namespace:campaigns.abc.encounters.dialog_1.msg1.choice_01": {
       "next_node": "msg4"
    },
    "my_mod_namespace:campaigns.abc.encounters.dialog_1.msg1.choice_02": {
       "out_edge": "dialog_ends"
    }
 }

        我们可以有 "next_node"，指向对话的另一个节点的键，或者 "out_edge"，它将结束对话并触发指定的输出边。

        请注意，对于此遭遇战类型，我们通常没有在 JSON 文件中定义输出边，因为输出边是在对话选择中指定的，并且多个选择可以指向同一个输出边。

商店

一种显示带有给定待售物品集的商店菜单的遭遇战。

  • encounter_type" : "shop"

  • "shop_info" : {}

    • "name" -- 商店的本地化名称。将出现在商店窗口顶部。
    • "title" -- 商店通知的本地化文本。
    • "inventory" : {} -- 商店的内容。它们将匹配指定的过滤器集（稀有度、材料、URI...）。您可以在此处查看有关如何设置它们的更详细说明这里。

递送任务

一种表示可以完成或放弃的持久任务通知的遭遇战。

  • "encounter_type" : "delivery_quest"

  • "delivery_quest_info" : {}

    • "title" -- 公告通知的本地化文本。

    • "dialog_title" -- 任务对话框的本地化文本，将出现在窗口顶部。

    • "text" -- 可选字段。显示在要求列表上方的本地化文本。

    • "abandon_out_edge" -- 可选字段。如果玩家放弃此任务将触发的输出边（在城镇发展任务中，这指向前一个遭遇战，因此玩家可以在需要时再次选择此任务）。

    • "requirements" : [] -- 包含完成任务的要求的对象数组（只要满足这些要求，玩家可以随时完成任务。如果需要，他们也可以在任何时候放弃任务）。

      每个要求将有一个 "type" 字段和一个或多个取决于它的附加字段（为了可读性，它们在下面缩进）。以下是可能的值：

      • "type" : "give_item" -- 递送 N 个特定 URI 的物品。

        • "uri" -- 物品的 URI。
        • "count" -- 要递送的物品数量。
        • "keep_items" -- 可选字段（布尔值）。当为 true 时，完成任务时物品不会从玩家的库存中移除。

      • "type" : "give_material" -- 递送 N 个标记有给定材料的物品。

        • "material" -- 物品必须具有的材料标签才能满足此要求。
        • "count" -- 要递送的物品数量。
        • "keep_items" -- 可选字段（布尔值）。当为 true 时，完成任务时物品不会从玩家的库存中移除。

      • "type" : "job_level" -- 拥有给定职业的炉民达到给定最低等级。

        • "uri" -- 职业的别名。
        • "level" -- 市民必须在给定职业中至少达到此等级才能满足要求。

      • "type" : "happiness" -- 拥有 N 个市民达到给定最低幸福度等级。

        • "min_value" -- 表示满足要求的最低幸福度的数字。请记住，幸福度范围为 1 到 100。
        • "min_citizens" -- 表示必须至少达到该幸福度的市民数量的数字。除了数字外，您还可以使用 "all"，表示所有市民，无论有多少。

      • "type" : "gold" -- 拥有 N 枚金币（当前、花费或赚取）。

        • "subtype" -- "give"、"have"、"spent" 或 "earned" 之一。如果子类型为 "give"，完成任务时将从玩家的库存中扣除金币。"spent" 和 "earned" 类型指的是通过在商店买卖获得的金币。
        • "count" -- 所需的金币数量。

      • "type" : "have_item_quality" -- 拥有 N 个已部署或存储的具有给定最低品质的物品。

        • "quality" -- 表示物品品质的数字：1 表示标准品质，2 表示精良品质，3 表示卓越品质，4 表示杰作品质（请注意，最后一种品质仅在城镇发展任务的一个分支中解锁）。
        • "count" -- 必须至少具有给定品质的物品数量。

      • "type" : "net_worth" -- 拥有给定的最低净资产。

        • "value" -- 要拥有的最低净资产。

      • "type" : "placed_item" -- 在世界中放置了给定物品。

        • "uri" -- 物品的 URI。

收集任务

一种显示请求物品的对话框的遭遇战。

  • "encounter_type" : "collection_quest"

  • "collection_quest_info" : {}

    • "out_edges" : {} -- 包含 3 个字段（"refuse"、"fail" 和 "success"），每个字段根据任务结果指向要触发的输出边。

      请注意，通常我们不在此遭遇战的 JSON 文件中添加输出边，因为它将根据玩家的选择触发其中之一。

    • "script" -- 确定请求哪些物品的脚本控制器的 URI（例如："stonehearth:game_master:script:collection_quest_shakedown"）。如果您创建自己的脚本，请确保在清单的 "controllers" 下添加它。我们将在创建时将 "collection_quest_info" 下的数据传递给脚本。脚本必须有一个 get_tribute_demand 函数，返回一个包含物品 URI 指向四个字段（uri、count、icon 和 display_name）的对象（Lua 表）。查看 collection_quest_shakedown.lua 中的实现方式。

    • "filler_material" : {} -- 可选字段。包含生物群落 URI 指向资源 URI 的映射。根据生物群落，将选择不同的资源作为请求的填充材料。这在 "stonehearth:game_master:script:collection_quest_shakedown" 脚本中使用，因此如果您不重用它，可能需要添加自己的自定义数据。

    • "skip_return_trip" -- 可选字段（布尔值）。当为 true 时，遭遇战是一个简单的物品请求，因此将直接显示 "collection_due" 对话框。

      使用此选项时存在一个已知问题，即即使没有所有物品，玩家也可以确认对话框（在这种情况下，只有可用物品将从库存中移除）。这意味着无法触发 "fail" 输出边，只能触发 "success" 和 "refuse" 输出边。如果您想使用此选项，您可能希望改用递送任务遭遇战。

    • "source_entity" -- 可选字段。对上下文注册路径的引用。如果此实体不存在，遭遇战将不会开始，如果实体在任务激活期间死亡，遭遇战将停止。

    • "duration" -- 在收集请求的物品之前等待多长时间的时间表达式。

    • "dialog_view" -- 可选字段，如果为 "portrait"，将使用与显示需求相同的视图。否则，您可以使用自定义 UI 视图的名称。此对话框视图将仅用于介绍对话框。

    • "i18n_data" -- 可选字段。与对话树遭遇战相同。

    • "nodes" : {} -- 对话的节点列表。每个节点都是一个包含多个属性的键。键必须具有以下名称："introduction"、"shakedown"、"shakedown_refused"、"collection_progress"、"collection_due"、"collection_failed"、"collection_success"、"revenge"。

      只有 "collection_due"、"collection_failed" 和 "collection_success" 节点是必需的，其余的是可选的。

      假设全部定义，它们出现的顺序是："introduction"、"shakedown"，然后如果玩家接受则为 "collection_progress"，如果玩家拒绝则为 "shakedown_refused"，在这种情况下将触发 "refuse" 输出边并显示 "revenge" 对话框。

      如果玩家接受了，在等待 "duration" 后将出现 "collection_due" 对话框，然后如果他们有所需物品并接受给予，物品将从库存中移除，显示 "collection_success" 对话框并触发 "success" 输出边。如果玩家没有所需物品或拒绝，将显示 "collection_failed" 对话框，触发 "fail" 输出边并显示 "revenge" 对话框。它还将运行关联脚本的 revenge(requested_items, info) 函数。第二个参数将包含复仇公告数据、库存跟踪数据和复仇节点数据。

      这些对话节点中的每一个都包含一个 "bulletin" 字段，具有以下属性：

      • "title" -- 公告通知的本地化标题。
      • "dialog_title" -- 对话框窗口的本地化标题（此遭遇战必需，但如果您不想要它，请将其留为空字符串）。
      • "portrait" -- 对话头像图像的路径（此遭遇战必需）。
      • "message" -- 本地化文本或本地化文本数组（在这种情况下，将随机选择一个）。要在对话框中显示的消息。它们可以使用来自 "i18n_data" 字段的变量。

捐赠对话

一种生成物品并通过对话框给予玩家的遭遇战。

  • "donation_dialog_info" : {}

    • "loot_table" -- 一个战利品表，其中包含描述要捐赠给玩家的物品的条目。如果您只想捐赠金币，可以省略。

    • "donation_gold" -- 可选字段。要捐赠的金币数量（无小数）。如果我们在之前的遭遇战中在上下文中注册了 donation_gold，则可以省略此字段，而使用注册的变量。

    • "expiration_timeout" -- 可选字段（一个时间表达式）。如果我们指定此字段，即使玩家未与其交互，对话框也将在此持续时间后自动移除。有助于防止公告列表中堆积可能不再相关的通知。

    • "choose_ctx_portrait" : [] -- 可选字段。与对话树遭遇战相同。

    • "use_ctx_portrait" -- 可选字段。与对话树遭遇战相同。

    • "i18n_data" : {} -- 可选字段。与对话树遭遇战相同。

    • "nodes": {} -- 包含 "simple_message" 字段，其中有 "bulletin" 字段，具有以下属性：

      • "title" -- 与对话树遭遇战相同。

      • "dialog_title" -- 与对话树遭遇战相同。

      • "portrait" -- 可选字段。与对话树遭遇战相同。

      • "message" -- 与对话树遭遇战相同。

      • "choices" : {} -- 与对话树遭遇战相同，除了它们只能包含一个值为 "accept" 或 "reject" 的 "result" 键。如果用户点击拒绝选项，他们将不会获得物品。

捐赠

城镇发展遭遇战

城镇升级选择

一种显示选择城镇升级并可选择城镇名称的对话框的遭遇战。

  • "encounter_type" : "town_upgrade_choice"

  • "town_upgrade_choice_info" : {}

    • "includes_town_naming" -- 可选字段。一个布尔值，用于确定是否显示文本框让玩家更改城镇名称。

    • "choices" : {} -- 升级城镇旗帜/炉心等的选择列表。每个选择都是一个自定义标识符，包含以下属性：

      • "name" -- 旗帜/炉心等类型的本地化名称。

      • "description" -- 相关城镇奖励的本地化描述。

      • "icon" -- 相应旗帜/炉心等的图像路径。

      • "sort_order" -- 用于在 UI 中对不同选择进行排序的序号。

      • "out_edge" -- 当玩家选择此选项时要触发的输出边。它最终应该通向城镇升级类型的遭遇战，这将向玩家的城镇授予描述中提到的奖励。

        请注意，通常我们不在此遭遇战的 JSON 文件中添加输出边，因为它将根据玩家的选择触发不同的输出边。

城镇升级

一种授予城镇奖励和/或替换旗帜或炉心的遭遇战。炉民（和一些 NPC）将聚集在要升级的物品周围，它将被更改（授予城镇奖励），城镇等级将提升，然后炉民和 NPC 将庆祝一段时间（使用他们声音常量中的 "celebration" 音轨）。NPC 将在庆祝结束后离开并消失。

大多数字段都是可选的，因此这也可用于围绕物品发出庆祝活动。

  • "encounter_type" : "town_upgrade"

  • "town_upgrade_info" : {}

    • "tier_achieved" -- 可选字段。一个数字，表示我们要将玩家城镇升级到的等级。

    • "celebration_duration" -- 表示庆祝持续时间的时间表达式。请确保定义此字段。

    • "old_facility_uri" -- 我们正在升级的物品的 URI（旗帜、炉心等）。一旦放置此物品，升级和庆祝将开始（例如，如果此遭遇战在移动物品时触发）。炉民将聚集在它周围。

      如果未定义旧设施，将选择城镇的炉心。如果它不存在，将改用城镇的旗帜。

      如果旧设施不是唯一实体，我们将从玩家的库存中随机选择一个，优先选择已经放置在世界的物品。

    • "new_facility_uri" -- 可选字段。将替换旧旗帜/炉心等的物品的 URI。

    • "facility_upgrade_effect" -- 升级旧设施时播放的 VFX 的 URI。通常会在某个时刻覆盖它以隐藏从旧到新的突然变化。

    • "town_bonus" -- 可选字段。具有相关城镇奖励的控制器的 URI。请记住在清单的 "controllers" 下添加它。一个什么都不做的城镇奖励示例：

      local MyTownBonus = class()

function MyTownBonus:initialize()
   self._sv.player_id = nil
   self._sv.display_name = 'i18n(my_mod:data.gm.campaigns.town_upgrades.hearth_choice.bonus.name)'
   self._sv.description = 'i18n(my_mod:data.gm.campaigns.town_upgrades.hearth_choice.bonus.description)'
end

function MyTownBonus:create(player_id)
   self._sv.player_id = player_id
end

return MyTownBonus

      上述函数将确保奖励出现在城镇概览 UI 的奖励选项卡中。您可以在 stonehearth/data/town_bonuses 中查看更多奖励控制器示例。在游戏的某些部分，我们会像这样检查特定奖励：

      local town = stonehearth.town:get_town(self._sv._entity:get_player_id())
town:get_town_bonus('stonehearth:town_bonus:strength')

      这允许根据当前城镇奖励执行一些代码，我们还可以调用城镇奖励控制器的函数，例如：

      local town = stonehearth.town:get_town(session.player_id)
local glory_bonus = town:get_town_bonus('stonehearth:town_bonus:glory')
if glory_bonus then
   return glory_bonus:spawn_next_wave(source_entity)
end
return false

    • "finish_event" -- 可选字段。庆祝结束后将在玩家的派系上触发的事件名称。

    • "conversation_subjects" : [] -- 可选字段。要为庆祝注入的主题 URI 数组。它们都将具有积极的情感。例如，我们在原版游戏中为升级到力量旗帜注入 "stonehearth:subjects:town_status"、"stonehearth:food:cake" 和 "stonehearth:banner:strength" 主题。炉民将在庆祝期间有机会谈论它们。

    • "thought" -- 可选字段。要注入给所有炉民的与当前升级相关的想法的 URI（例如，"stonehearth:thoughts:town:progression:outpost"）。

    • "has_fireworks" -- 可选字段。一个布尔值，用于确定是否运行 "stonehearth:effects:fireworks" 效果。请注意，此效果将从新设施播放。

    • "dialog" : {} -- 可选字段。在聚集庆祝之前弹出的对话框的数据。它包含以下字段：

      • "title" -- 公告通知的本地化文本。

      • "dialog_title" -- 对话框标题的本地化文本，将出现在窗口顶部。

      • "tier_title" -- 已达成的等级的本地化文本（前哨站、定居点等）。

      • "reward_message" -- 宣告消息第一部分的本地化文本（例如："经市民一致同意，我们宣布："）。

      • "reward_message_ctd" -- 宣告消息第二部分的本地化文本（例如："成为所有人安全避风港的前哨站。"）。

        奖励/宣告消息将出现在窗口中间。它分为两部分，因为城镇名称将出现在它们之间。

      • "achievements" : [] -- 对象数组。这是一个描述与城镇等级/奖励相关的成就的列表，显示在对话框底部。每个条目包含以下属性：

        • "id" -- 此条目的唯一标识符（例如："strength_bonus_1"）。
        • "name" -- 描述成就的本地化短文本（"随机每日增益"、"2 级背景音乐"等）。
        • "icon" -- 在此成就的对话框中显示的图标路径。在原版游戏中它是丝带，但您可以为每个奖励使用不同的图标。
        • "track" -- 当成就出现在 UI 中时播放的音轨的 URI（成就会逐个出现，就像解锁一样）。

    • "npcs" : [] -- 可选字段。要为庆祝生成的 NPC 数组。每个条目都是一个包含这些属性的对象：

添加市民

向城镇添加一个新市民，可以是玩家派系的或其他种族/派系的。他们将在城镇附近生成并首先走向旗帜。

请注意，一旦添加到玩家的城镇，您将无法通过代码检索市民的原始派系。他们将被视为与玩家的城镇具有相同的派系/阵营，即使他们的外观不同（例如，兔子、兽人）。

  • "encounter_type" : "add_citizen"

  • "add_citizen_info" : {}

    • "population" -- 可选字段。新市民的王国 URI（例如："stonehearth:kingdoms:amberstone"）。如果省略，将使用与玩家城镇相同的王国。

    • "role" -- 可选字段。要从派系数据中使用的角色。如果省略，将使用 "default" 角色。

    • "gender" -- 可选字段。新市民的性别。如果省略，将使用随机性别。性别在 stonehearth/data/constants.json 中指定，并与不同角色内的条目匹配。

    • "job" -- 新市民默认将拥有的职业的 URI。虽然这被视为可选字段，但请确保设置它，以便新市民拥有适当的服装并可以晋升到其他职业。不保证会使用默认职业。

    • "level" -- 可选字段。新市民在指定职业上的默认等级。

    • "allowed_jobs" : [] -- 可选字段。包含职业 URI 的数组。这限制了新市民可以晋升的职业。当市民来自另一个派系且没有玩家派系职业索引中某些职业的服装或动画时很有用。

    • "equipment" : [] -- 可选字段。装备部件的 URI 数组，将自动装备给新市民（注意关于职业的装备角色）。

    • "notification_title" -- 可选字段（如果省略此字段，将不显示通知）。公告通知的本地化文本。当玩家点击公告时，摄像机将聚焦于新市民。如果愿意，您可以在本地化文本中使用 i18n_data.citizen_custom_name、i18n_data.citizen_display_name 和 i18n_data.town_name。

    • "attribute_distribution_override" : {} -- 可选字段。如果包含，它将覆盖此市民的派系默认属性。

      • "allocated_points" : {} -- 包含 "min" 和 "max" 属性。将在它们之间随机选择点数。
      • "point_limits" : {} -- 包含条目，其键是主要属性的名称（"mind"、"body"、"spirit"），其值也是 "min" 和 "max"，用于确定可以给某个属性多少点数。属性影响其余的属性。

重新登船

一种生成由最多 3 名市民和最多 10 件物品组成的登船队伍的遭遇战，可在新游戏中重用（如果玩家选择城镇发展物品，它将携带城镇奖励）。

它还将包括已解锁的作物（用于 "stonehearth:jobs:farmer" 职业）和来自任何制作职业的已解锁制作配方，但这些在新游戏中解锁有条件（如果您选择来自不同王国的重新登船队伍，配方必须已存在于当前王国的配方索引中，并且工匠的职业 URI 也必须匹配，否则无法解锁）。

市民将保留他们的职业、等级和装备，但请注意，带着高等级战斗单位登船可能会增加早期游戏的战斗难度。

重新登船可以推迟（保留在公告列表中或不保留），但一旦玩家点击"出发"，队伍将被保存到文件中，并围绕旗帜开始庆祝。当庆祝结束时，所选市民将永远离开城镇，所选物品也将从库存中消失。如果玩家不希望这些市民离开，可以随时返回到之前的存档文件。

  • "encounter_type" : "reembarkation"

  • "reembarkation_info" : {}

    • "reject_out_edge" -- 如果玩家推迟对话框（"也许下个月"选项）将触发的输出边。在原版游戏中，这被设置为销毁当前遭遇战节点，并且重新登船生成器在后台以 30 个游戏日的延迟继续运行。

创建营地

一种创建由怪物和/或物品组成的 NPC 营地的遭遇战。

  • "encounter_type" : "create_camp"

  • "create_camp_info" : {}

    • "npc_player_id" -- （必需字段）玩家或 NPC 派系名称（派系文件的 "kingdom_id"）。可以是空字符串（将被视为未指定）。

    • "amenity_with_player" -- 可选字段（"hostile" 或 "neutral"）。我们将设置此营地的怪物对玩家的友好度。不要与 "neutral_to_everyone" 字段同时使用。

    • "neutral_to_everyone" -- 可选字段（布尔值）。当为 true 时，任何人都无法攻击此营地的怪物，他们对所有其他派系都是中立的。请记住稍后取消设置，否则指定派系的怪物在未来的遭遇战中仍将保持中立。

    • "ctx_entity_registration_path" -- 可选字段（例如："goblin_raiding_camp_1"）。上下文注册路径的名称。我们将在此路径中存储来自 "citizens" 的实体引用（实际上在 <your_custom_ctx_path>.citizens 内）。

    • "use_previous_enemy_location" -- 可选字段。一个布尔值，用于在之前注册的敌人位置（ctx.enemy_location）生成营地。

    • "use_entity_location" -- 可选字段。引用已放置实体的上下文注册路径。营地将在实体所在的位置生成。

    • "belongs_to_player" -- 可选字段。一个布尔值，使营地块属于玩家（默认情况下它们将属于 "npc_player_id" 的派系）。

    • "radius" -- （必需字段）营地的半径，以方块数为单位。

    • "keep_grass" -- 可选字段（布尔值）。默认情况下，如果营地在草地上生成，草层将被移除，在地形中形成凹陷。使用此字段保留草地。无论如何，营地将在其生成的位置摧毁任何实体（如树木或不属于玩家的任何东西）。

    • "spawn_range" : {} -- （必需字段）。包含 "min" 和 "max" 属性，用于确定在城镇周围（以方块数为单位）生成营地的范围。

    • "script" -- 可选字段。在生成营地并注册其所有实体后运行的脚本的 URI，用于进一步定制。必须有一个 start(ctx, info) 函数，它将接收上下文和 "create_camp_info" 内的信息。如果您制作自定义脚本，请记住在清单的 "controllers" 下添加它。

    • "combat_bulletin" : {} -- 可选字段。包含具有本地化文本的 "title" 字段，当玩家的市民遇到来自此营地的敌人时将其显示为通知公告。

    • "searcher_delay" -- 可选字段。一个时间表达式。如果营地找不到合适的生成位置（给定城镇周围范围内的足够空间），我们将等待此持续时间并再次尝试搜索生成位置（默认为 "3d"）。这是一种安全措施，以防止遭遇战停滞。

    • "on_searcher_failure" : {} -- 可选字段。包含 "retry"（用于重试的布尔值或最大重试次数的数字）或 "destroy_root_node"（指向我们要销毁的节点的输入边）。用于搜索器未能找到生成营地的位置的情况。

    • "continue_on_disconnect" -- 可选字段。一个布尔值，用于在多人游戏中玩家（客户端）断开连接时继续运行此节点。默认为 false（暂停此遭遇战/计时器，并在玩家重新连接后恢复）。

      对于此遭遇战类型，如果营地已经生成，无论此选项如何，它将继续存在。但是，如果在玩家断开连接时正在运行寻找位置的计时器，它将停止尝试并在玩家回来后恢复（因此只有在玩家连接时才允许营地生成）。

    • "landmark_data" : {} -- 可选字段。包含 "brush" 和 "landmark_block_types" 等属性，用于为营地生成地标。检查 stonehearth/lib/landmark/landmark_lib.lua 以获取可能的字段（create_cube_as_terrain 和 create_qb_as_terrain 函数的选项）。

    • "pieces" : {} -- 营地的组成部分（必需字段）。每个条目都是一个自定义标识符，具有以下属性：

      • "entity_uri" -- 物品的 URI。代替此字段，我们可以使用指向描述营地块的 JSON 文件的 "info" 字段（允许使用更多属性）。查看 stonehearth/data/gm/campaigns/goblin_war/arcs/trigger/raidcamp/encounters/create_camp 文件夹以获取示例。
      • "position" : {} -- 包含从上方看到的相对于营地中心的 "x" 和 "y" 坐标（营地中心位于 (0,0)）。
      • "npc_player_id" -- 可选字段。将拥有营地块的派系名称。
      • "rotation" -- 可选字段。营地块的旋转角度（以度为单位）。
      • "movable" -- 可选字段（布尔值）。此物品是否应具有移动/取消部署命令以允许玩家移动它。

    • "citizens" : {} -- 可选字段。要在营地中生成的生物实体（如怪物或 NPC）列表。每个条目都是一个自定义标识符，包含以下属性：

      • "from_population" : {} -- 使用此字段意味着我们想要生成在 "npc_player_id" 派系文件中定义的实体。以下是 "from_population" 内的属性：

        • "role" -- 我们希望此实体在派系中担任的角色。

        • "gender" -- 可选字段。此条目中实体的性别。如果我们没有此字段，将根据 stonehearth/data/constants.json 中 population.base_genders 数组中的性别随机选择性别。

        • "location" : {} -- 可选字段。包含 "x" / "y" / "z" 属性，用于将此实体放置在营地内。

        • "range" -- 可选字段。一个表示方块/世界单位的数字，用于在 "location" 周围的范围内随机化实体的生成位置。

        • "min" -- 可选字段。为此条目生成的实体的最小数量。

        • "max" -- 可选字段。为此条目生成的实体的最大数量。将在 "min" 和 "max" 之间随机选择实体数量。如果未指定，它们将默认为 1。

        • "scale_with_run" : {} -- 可选字段。用于根据此遭遇战触发的次数增加我们生成的实体数量。包含这些属性（定义时，"min" 和 "max" 将被它们修改，或者如果留空则仅由运行次数修改）：

          • "encounter_cap" -- 可选字段。当按运行次数缩放时，这将作为上限，以防止数字变得太高。
          • "scale_factor" -- 可选字段。包含时，"min" 和 "max" 将乘以（并四舍五入）此值的（遭遇战运行次数减一）次方。

        代替 "from_population" 字段，我们也可以使用 "from_ctx"，指向之前在上下文中注册的实体。以下属性在 "from_population" 之外：

      • "attributes" : {} -- 可选字段。包含要应用于实体的属性名称/值对。

      • "combat_leash_range" -- 可选字段。一个大于 0 的数字，表示拴绳范围。这是实体允许远离其生成区域的方块数（防止它们被风筝离家园位置太远）。

      • "equipment" : {} -- 可选字段。包含指向装备部件 URI 或 URI 数组的自定义标识符。它们将装备给此条目的所有实体。如果使用数组，将从中随机选择一个装备部件，因此如果您想装备多个部件，必须将它们作为单独的 URI 条目添加。

      • "equipment_scale" -- 可选字段。一个表示装备部件缩放的数字（以便大怪物不使用小武器）。

      • "job" -- 可选字段。要将此条目的实体晋升到的职业的 URI。

      • "loot_drops" : {} -- 可选字段。一个战利品表，包含要应用到此条目实体的战利品。

      • "render_info" : {} -- 可选字段。包含指向颜色/材质映射 URI 的 "color_map" 和/或 "material_maps" 字段，以应用到此条目的实体。

      • "scale" -- 可选字段。要应用到此条目实体的缩放。

      • "tuning" -- 可选字段。要应用于实体的怪物调优文件的 URI（例如："stonehearth:monster_tuning:goblins:default"）（可以有更多附加属性，这是在遭遇战之间重用信息的好方法）。您可以在 stonehearth/data/monster_tuning 中找到现有的调优文件。它们是 JSON 文件，因此如果您创建新文件，可能希望在清单中为其添加别名。

      • "unit_info" : {} -- 可选字段。包含要应用于实体的 unit_info 组件的属性（"display_name"、"custom_name" 和 "description"）。

      "citizens" 将作为数组存储在上下文中，因此我们可以通过索引单独引用它们，例如 "create_scout_camp.citizens.wolf_cage.tame_wolf[1]"（请记住 Lua 索引从 1 开始）。

    • "boss" : {} -- 可选字段。与 "citizens" 列表中的 "from_population" / "from_ctx" 字段相同的属性。代表此营地的首领。如果您指定了 "ctx_entity_registration_path"，它将作为 boss 注册在其下，其自定义名称和显示名称将作为 boss_custom_name 和 boss_display_name 注册在上下文中（以便您可以在对话中使用它们）。如果它引用多个实体，只有一个将被注册为首领。

一种启动由 NPC 执行的任务的遭遇战。例如，四处游荡或掠夺。

  • "encounter_type" : "create_mission"

  • "create_mission_info" : {}

    • "spawn_range" : {} -- 可选字段。具有 "min" 和 "max" 属性，用于确定我们想要生成怪物的城镇周围范围（以方块数为单位）。

    • "continue_on_disconnect" -- 可选字段。一个布尔值，用于在多人游戏中玩家（客户端）断开连接时继续运行此节点。默认为 false（暂停此遭遇战/计时器，并在玩家重新连接后恢复）。

      对于此遭遇战类型，如果怪物已经生成，无论此选项如何，它们将继续存在。但是，如果在玩家断开连接时正在运行寻找生成位置的计时器，它将停止尝试并在玩家回来后恢复。

    • "use_previous_enemy_location" -- 可选字段（布尔值）。当为 true 时，怪物将在之前的敌人位置（营地等）生成。

    • "on_searcher_failure" : {} -- 可选字段。包含 "retry"（用于重试的布尔值）或 "destroy_tree" : {}（带有 "destroy_root" : true 和 "root" : "in_edge_of_encounter_node_to_destroy" 属性）。用于搜索器未能找到生成怪物的位置的情况。

    • "mission" : {} -- 必需字段。内部的字段将根据任务类型而变化，我们从常见的开始：

      • "ctx_entity_registration_path" -- 可选字段（例如："goblin_raiding_camp_1"）。上下文注册路径的名称。我们将在此存储来自 "members" 的实体引用。

      • "combat_bulletin" : {} -- 可选字段。包含 "title" 属性的本地化文本，当炉民遇到来自此任务的敌人时显示通知公告。此字段也可以称为 "sighted_bulletin" : {}。

      • "npc_player_id" -- 此任务 NPC 的派系名称。如果在之前的遭遇战中已定义，则可以省略此字段。

      • "members" : {} -- 必需字段。与创建营地遭遇战中的属性相同。

      • "offset" : {} -- 必需字段。包含 "x"、"y"、"z" 属性，用于在生成位置的偏移处生成怪物。

      • "use_wait_entity_location" -- 可选字段（布尔值）。当为 true 时，怪物将在之前注册的敌人位置生成。

      • "raid_timeout_minutes" -- 可选字段（默认为 4300，约 3 天）。在此数量的游戏分钟后，怪物将离开/消失（如果它们具有袭击超时观察器）。

      • "raid_timeout_variance_minutes" -- 可选字段（默认为 0）。怪物将在 "raid_timeout_minutes" 和 "raid_timeout_minutes" 加上 "raid_timeout_variance_minutes" 之间的某个时间消失（内部使用这些字段的数值作为分钟创建一个时间表达式）。

      • "role" -- 任务类型，必须是以下之一：

        • "raid_stockpiles" -- 怪物将从玩家的仓库中偷窃物品和/或摧毁它们（这取决于它们的 AI 行为，例如：哥布林工人和小偷知道如何偷窃，但掠夺者只会摧毁东西）。

          对于此任务类型，我们可以有一个附加字段："require_free_stockpile_space"。它是一个可选的布尔值。当为 true 时，如果怪物自己的仓库中没有足够的空间，他们将不会袭击仓库。如果玩家没有仓库，怪物将只是跑到城镇，希望在路上找到可以攻击的东西，尽管他们最终可能只是漫无目的地游荡。

        • "spawn_enemies" -- 简单地生成一些敌人。对于此任务类型，我们可以有一个附加字段："spawn_effect"。它是可选的（敌人生成时播放的效果的 URI）。

        • "raid_crops" -- 如果玩家没有农场，此任务将不会开始。这用于小动物来摧毁玩家的作物。对于此任务类型没有附加属性。

        • "pillage" -- 敌人将生成并前往某个地方进行攻击。如果他们在 3 个游戏小时后无法到达目的地，我们将尝试寻找新的目的地。

          对于此任务类型，我们有两个附加属性："pillage_radius" : {}，这是必需的，包含 "min" 和 "max" 字段，用于在城镇旗帜周围的该范围内找到攻击位置；以及 "target"，它是可选的，指向用于攻击位置的上下文注册路径（而不是旗帜）。

        • "wander" -- 怪物将生成并以方形模式开始游荡。对于此遭遇战类型，我们有两个附加字段："wander_radius" : {}，这是必需的，包含 "min" 和 "max" 属性，用于随机化方形的大小范围（以便他们不总是在同一个地方行走）；以及 "walk_interval"，它是可选的，是一个时间表达式，表示多久走到下一个位置（默认为 "30m"）。

城市袭击

一种袭击，只是在之前的敌人位置（由创建营地遭遇战或创建任务遭遇战设置）同时启动的一组任务。与创建任务遭遇战不同，此遭遇战不搜索生成位置。

  • "city_raid_info" : {}

    • "continue_on_disconnect" -- 可选字段。对于此遭遇战，它似乎不运行代码，因为我们不搜索生成袭击的位置。
    • "missions" : {} -- 包含具有自定义标识符的条目，每个条目都具有与创建任务遭遇战中的 "mission: {}" 字段相同的属性。
