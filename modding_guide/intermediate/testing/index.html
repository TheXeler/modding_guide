Microworld

Microworld 模组通过创建一个自定义测试世界来帮助我们更快地测试模组，我们可以使用调试工具生成模组化实体，而无需每次想要测试某些内容时都经过登船过程。

然而，对于测试生物群系，没有简单的方法，因为你将希望检查整个地图生成，而不仅仅是一个平坦的小世界。

设置 Microworld

Microworld 模组现在与游戏捆绑在一起（你也可以在 Github 上找到它）。

  1. 从主菜单进入模组菜单，在本地模组列中搜索它，并启用 "Microworld for Tests" 模组。

  2. 关闭游戏。

  3. 编辑你的 user_settings.json 文件。

    • 如果使用 Humble Bundle 安装程序安装游戏，它通常位于 C:\Program Files (x86)\Stonehearth。

    • 如果通过 Steam 安装，则位于 C:\Program Files (x86)\Steam\steamapps\common\Stonehearth。

    其中一些键可能已经存在，在这种情况下只添加缺少的部分。

    {
   "user_id": "your_user_id",
   "game" : {
      "main_mod" : "microworld"
   },
   "mods" : {
      "microworld" : {
         "world" : "harvest_test"
      }
   }
}

    "main_mod" 键告诉游戏应该加载哪个模组作为主模组。在这种情况下，我们将首先加载 Microworld 模组，这将跳过徽标/主菜单屏幕并直接加载一个小世界。确保在你想要再次玩游戏时使此键无效（"_main_mod" : "microworld"），或使用 Esc 菜单返回主菜单或加载存档文件。

    Microworld 模组设置中的 "world" 选项在这种情况下指向 "harvest_test"（默认情况下它将运行 mini_game 世界）。位于 microworld/worlds 目录中的所有测试世界，其名称都以 _world.lua 结尾。你必须遵循该命名约定才能使模组运行测试。例如，如果你的测试世界称为 mytest_world.lua，在 user_settings 中你将指定为 "mytest"。换句话说，文件必须以 _world 结尾，但 json 中的条目不能。

  4. 最后，启动游戏。

你也可以从命令行运行此模组（确保首先启用了模组！）：

  Stonehearth.exe --game.main_mod=microworld --mods.microworld.world=harvest_test

默认世界

Microworld 模组中存在 4 个测试世界（在此页面编写时）：data_driven_world.lua、harvest_test_world.lua、mini_game_world.lua 和 settlement_test_world.lua。

你可以尝试更改你的 user_settings 以查看每个世界的内容。对于 data_driven 世界，你需要添加一个额外的键：

  "mods" : {
     "microworld" : {
        "world" : "data_driven",
        "data_driven_index" : "microworld:data_driven:world:profession_test"
     }
  }

"data_driven_index" 键指向 microworld 清单中定义的别名。你可以为自定义世界在那里添加更多别名。

如何创建你自己的自定义世界

你可以在 microworld/micro_world.lua 中找到许多在自定义世界中创建对象的函数，你也可以使用来自游戏或其他模组的代码。

  • data_driven_world.lua 世界使用 data/data_driven_world 子文件夹中的 JSON 数据来生成自定义世界。如果你不想使用 Lua，可以尝试这种方法。

    如上所述，你需要向 microworld 的清单添加别名以添加你的数据驱动世界，在 user_settings 中使用 "data_driven" 世界以及你的自定义 JSON 世界的 "data_driven_index"。

    在 JSON 文件中，你可以指定世界的大小、你想要生成的市民及其位置、要生成的物品及其位置、地形块及其位置等。查看 data/data_driven_world 中的示例，以及 data_driven_world.lua 中的注释。

  • 其余世界是用 Lua 编写的。这是创建微世界所需的最少代码：

    local MicroWorld = require 'micro_world'
local SettlementTest = class(MicroWorld)

function SettlementTest:__init()
   self[MicroWorld]:__init(100)
   self:create_world()
   -- More code
end

return SettlementTest

    我们在第一行需要 micro_world.lua 文件，其中包含 Microworld 类。然后我们实例化 Microworld 类，并为我们的对象定义一个 __init() 函数以初始化我们的自定义世界。在其中，我们调用 Microworld 类的初始化函数，传递我们希望测试世界的直径（以方块数为单位），并调用 create_world() 函数（也来自 Microworld 类）以实际创建平坦地形。最后，我们返回我们的对象。

    如果需要，我们可以在类中创建更多函数。micro_world.lua 中已经有许多函数对准备测试世界很有用，你可以像调用 create_world() 函数一样调用它们，并传递你需要的任何参数。例如，create_world() 函数可以接收王国 URI 和生物群系 URI 作为参数，以使用它们生成测试世界。

    请注意，microworld 尝试模拟常规游戏，但它实际上并不经历与常规游戏相同的游戏流程。例如，我们必须手动初始化要在 MicroWorld:create_world 中使用的服务。因此，如果你遇到仅在测试中发生的 bug，这可能是由于测试本身而不是游戏。通常测试应该匹配游戏行为，但不能保证。

Stonehearth Tests

这些测试与上面解释的 Microworld 世界非常相似。stonehearth_tests 模组利用 microworld 模组生成测试世界，因此如果你计划使用 stonehearth_tests 模组，你需要在模组菜单中启用两者。

开发团队使用此模组在开发过程中手动隔离测试功能，或尝试在小环境中重现 bug。

你在此模组中有大量示例可以重用，所以如果你是编码新手，你可以在这里摆弄 Lua，直到你感觉足够舒适以在你的模组中尝试更复杂的事情。

上面为 microworld stated 的相同警告也适用于 stonehearth_tests 模组。例如，在运行测试时，你会注意到游戏大师服务尚未初始化（当你打开 Campaign Browser 调试工具时它看起来是空白的——幸运的是它有一个从 UI 启动它的按钮）。所以你需要手动初始化它们。此外，除非你执行 multiplayer_test.lua 中列出的操作，否则多人游戏不起作用。

设置 stonehearth_tests

stonehearth_tests 模组现在与游戏捆绑在一起。

  1. 从主菜单进入模组菜单，在本地模组列中找到 "Microworld for Tests" 和 "Stonehearth Unit Tests" 模组，并启用两者。

  2. 关闭游戏。

  3. 编辑你的 user_settings.json 文件。

    其中一些键可能已经存在，在这种情况下只添加缺少的部分。

    {
   "user_id": "your_user_id",
   "game" : {
      "main_mod" : "stonehearth_tests"
   },
   "mods" : {
      "stonehearth_tests" : {
         "test" : "pet_test"
      }
   }
}

    "main_mod" 键告诉游戏应该加载哪个模组作为主模组。在这种情况下，我们将首先加载 stonehearth_tests 模组，这将跳过徽标/主菜单屏幕并直接加载一个小测试世界。确保在你想要再次玩游戏时使此键无效（"_main_mod" : "stonehearth_tests"），或使用 Esc 菜单返回主菜单或加载存档文件。

    stonehearth_tests 模组设置中的 "test" 选项在这种情况下指向 "pet_test"。当你创建自己的自定义测试世界时，你可以将其更改为你的 Lua 文件的名称。按照惯例，位于 stonehearth_tests 目录中的所有测试，其名称都以 _test.lua 结尾。与 Microworld 模组不同，我们必须在 user_settings.json 的 "test" 字段中声明文件的完整名称（减去扩展名），如上面的示例所示。

  4. 最后，启动游戏。

你也可以从命令行运行此模组（确保首先启用了此模组和 Microworld 模组！）：

  Stonehearth.exe --game.main_mod=stonehearth_tests --mods.stonehearth_tests.test=pet_test

Stonehearth Autotests

我们可以创建自动化测试，将预期结果与实际结果进行比较，这样我们就不必手动测试功能。它们对于验证在更改某些代码后是否引入了回归非常有用。

请记住，我们可能无法自动化某些事情，并且 autotest 有时重现的问题在游戏中无法重现，反之亦然。尽量避免有时通过有时失败的不可靠测试。

要创建我们自己的自动化测试，我们有两个现在与游戏捆绑在一起的模组。stonehearth_autotest 模组需要 autotest_framework 模组才能工作，因此如果你计划运行 autotests，你必须在 user_settings.json 中启用两者：

  1. 从主菜单进入模组菜单，在本地模组列中搜索 "Autotest Framework" 和 "Stonehearth Autotests" 模组，并启用两者。

  2. 关闭游戏。

  3. 编辑你的 user_settings.json 文件。

    其中一些键可能已经存在，在这种情况下只添加缺少的部分。

    {
   "user_id": "your_user_id",
   "game" : {
      "main_mod" : "stonehearth_autotest"
   },
   "mods" : {
      "stonehearth_autotest" : {
         "options" : {
            "run_forever" : false,
            "num_times" : 10,
            "exit_on_complete" : false,
            "continue_on_failure" : true,
            "pause_on_error" : true,
            "script" : "/stonehearth_autotest/tests/end_to_end/harvest_autotests.lua",
            "function" : "long_*",
            "group" : "nightly"
         }
      }
   }
}

    "main_mod" 键告诉游戏应该加载哪个模组作为主模组。在这种情况下，我们将首先加载 stonehearth_autotest 模组（输入时要小心，它不以 's' 结尾）。确保在你想要再次玩游戏时使此键无效（"_main_mod" : "_stonehearth_autotest"），或使用 Esc 菜单返回主菜单或加载存档文件。

    然后我们有几个可以使用的 "options"：

    • "run_forever" -- 可选。如果我们设置为 true，它将永远运行指定的测试，直到我们关闭游戏。

    • "num_times" -- 可选。将每个测试运行指定的次数。

    • "exit_on_complete" -- 可选。当为 false 时，将在最后一个测试完成后保持游戏打开，而不是自动关闭游戏。

    • "continue_on_failure" -- 可选。当为 true 时，即使其中一个测试失败，测试也将继续执行。

    • "pause_on_error" -- 可选。当为 true 时，将在测试失败时暂停执行，以便能够检查游戏状态。

    • "script"、"function" 和 "group" -- 使用这些我们定义要运行哪些 autotests。如果我们不指定脚本或组，则将执行名为 "all" 的组。要查找可用组，请查看 stonehearth_autotest/tests/index.json 中的 "groups" 键。

      我们可以运行单个脚本而不是一组脚本。在这种情况下，在 "script" 选项中指定包含 autotests 的 Lua 文件的完整路径。按照惯例，我们以 _autotests.lua 结束我们的 autotests 文件。

      我们还可以从脚本运行特定测试。为此，在 "script" 选项中定义包含测试的脚本，然后在 "function" 选项中写入测试的名称。例如：

      "script" : "/stonehearth_autotest/tests/end_to_end/harvest_autotests.lua",
"function" : "clear_test"

      我们还可以为 "function" 选项使用几个伪正则表达式。例如，这将运行 death_autotests.lua 中以 regression_ 开头的所有测试：

      "script" : "/stonehearth_autotest/tests/end_to_end/death_autotests.lua",
"function" : "regression_*"

      这将运行 death_autotests.lua 中不以 regression_ 开头的所有测试：

      "script" : "/stonehearth_autotest/tests/end_to_end/death_autotests.lua",
"function" : "!regression_*"

      注意感叹号和星号。

  4. 最后，启动游戏。根据你的选项，窗口将在测试运行完成后自动关闭或不关闭。此外，你可能有兴趣增加 autotest_framework 模组的日志记录级别。框架将在 stonehearth.log 中打印正在执行的脚本和测试（以及哪些测试失败，连同消息）。通过的测试在完成时不会打印任何特殊消息。

你也可以从命令行运行此模组（确保首先启用了 autotest_framework 和 stonehearth_autotest 模组！）：

  Stonehearth.exe --game.main_mod=stonehearth_autotest --mods.stonehearth_autotest.options.script=stonehearth_autotest/tests/end_to_end/harvest_autotests.lua

如何创建你自己的自定义 autotests

大多数 autotests 位于 stonehearth_autotest/tests/end_to_end 内，你可以查看它们以了解它们的工作原理。Autotests 默认以速度 4 运行，其中许多依赖于游戏如此之快。与 microworld 模组不同，我们不需要手动创建地形，将使用默认大小。

一般来说，遵循以下步骤：

  1. 在（例如）stonehearth_autotest/tests/end_to_end 文件夹中创建一个 Lua 文件。例如 my_mod_autotests.lua。

  2. 将你的 Lua 文件添加到 stonehearth_autotest/tests/index.json 文件中某个组内的 "scripts" 数组中。

  3. 编辑你的 Lua 文件。最初它看起来像这样：

    local my_tests = {}

function my_tests.verify_this_and_that(autotest)

   autotest:success()
end

return my_tests

    你添加的所有测试必须在满足成功条件时调用 autotest:success()，否则它们将失败。然后，你可以通过调用 autotest:fail 并附带描述失败原因的说明性消息来使它们在满足失败条件时失败：

    autotest:fail('failed to complete this thing')

    一般来说，autotests 将具有以下结构：

    • 它们将首先通过创建一些实体来设置场景。

    • 然后开始监听表示测试成功（有时也是失败）的事件。

    • 然后通过点击 UI 或在 Lua 中调整内部值来启动一些任务，以产生成功条件。

    • 然后等待一段时间以给游戏时间执行上述操作。

    • 然后失败（只有在未及时满足成功条件时才会达到的超时）。

    例如：

    -- Verify that a hearthling can die when guts reach 0 and that the tombstone is generated
function death_tests.memorialize_death(autotest)
   local worker = autotest.env:create_person(5, 5, { job = 'worker', resources = { health = 0, guts = 8} })

   autotest.util:listen(radiant, 'radiant:entity:post_create', function (e)
         if e.entity:get_uri() == 'stonehearth:tombstone' then
            autotest:success()
         end
      end)

   autotest:sleep(5000)
   autotest:fail('failed to die')
end

    为了找到 autotests 中使用的辅助函数，你需要查看 autotest_framework 文件。例如，在上面的示例中，我们调用 autotest.env.create_person()，这是来自 autotest_framework/lib/server/autotest_environment.lua 的函数。

    函数 autotest.util.listen() 是来自 autotest_framework/lib/server/autotest_util_server.lua 的包装函数（你可以在那里找到其他辅助函数，如 fail_if_expired(timeout)）。

    任何时候你在 autotests 中监听事件时，你必须使用其中定义的包装器之一，以便 autotest 框架可以正确清理它们，无论测试通过还是失败（否则，当下一个测试执行时，一些监听器可能仍然存在，并可能在错误的时间触发，抛出错误）。

    autotest:success()、autotest:fail() 和 autotest:sleep() 函数在 autotest_framework/lib/server/autotest_instance.lua 中定义，它控制 autotests 的实际执行。sleep() 函数的参数是一个表示毫秒的数字。

    测试中使用的其他常见函数位于 autotest_framework/lib/server/autotest_ui_server.lua 中。当然，你可以使用来自游戏的代码（如检索组件等）。

    要小心，因为如果你不重置/清理某些内容，它们可能会从一个测试保留到另一个测试，如打开的 UI 菜单。

Lua 分析器脚本

使用这些脚本，你可以分析模组的 Lua 性能。你可以在这里找到它们。

设置说明：

  1. 将以下内容放在你的 user_settings.json 中：

    "lua" : {
    "enable_cpu_profiler" : true,
    "enable_memory_profiler" : true,
    "cpu_profiler_method" : "sampling", // or "time_accumulation"
    "profiler_instruction_sampling_rate" : 1,
    "max_profile_length": 10000 //optional, how long in ms we can run the profiler
},
"simulation" : {
    "initial_speed_override" : 0,
    "long_profile_tick_threshold" : 500
}

    粘贴到你的用户设置后删除注释，并根据需要调整配置值。

  2. 通过导航到你下载 lua_profiler 脚本文件夹的文件夹，然后使用以下参数运行 Python 脚本来构建 lua 文件映射：

    collect_lua_file_map.py [PATH_TO_MODS_FOLDER] lua_file_map.js

  3. 将 [PATH_TO_MODS_FOLDER] 替换为你的 stonehearth 模组文件夹的路径。

    只有模组文件夹中提取的模组将被分析。其他文件夹中的模组或 .smod 格式的模组将被跳过。

  4. 加载你想要分析的游戏。

  5. 等待 UI 出现，打开Performance Monitor 调试工具。

  6. 准备好后（点击速度 1 按钮并等待初始脚本赶上后），点击分析按钮（播放按钮）。如果你正在寻找导致突然卡顿的原因，请选中 Long Ticks Only 复选框。

  7. 按下停止按钮以完成分析。

  8. 打开 Chrome 并导航到 lua_profiler 文件夹中的 lua_profiler.html 文件。

  9. 点击 "Choose Files" 按钮并导航到 Stonehearth\profiler_output\DATE_AND_TIME_OF_PROFILE_CAPTURE 下的分析转储。profiler_output 文件夹可以在与 mods 文件夹相同的文件夹中找到。

  10. 选择该文件夹下的所有文件并点击打开。

  11. 当分析器完成加载文件时，它将用每个方法名称、总时间、调用次数、Lua 消耗百分比（这个数字很重要）和文件路径填充行。

  12. 查看花费最多时间的函数，并找出是否有方法使其更快。

  13. 再次分析！

重要：每次需要加载新分析时，启动一个新的 lua_profiler/lua_profiler.html 标签页。

注意：如果你在 ai::CreateAction 中分析时遇到游戏崩溃，请在前一行添加对 ai.get() 的检查。
