添加自定义建筑资源

我们可以向建筑编辑器添加自定义建筑资源，以便炉民可以使用它们进行建造。

  1. 首先，我们需要创建一个通用资源并将其添加到 stonehearth/data/resource_constants.json。我们在这里解释了与建筑编辑器相关的字段。我们的示例如下所示：

    {
   "resources": {
      "sugar resource": {
         "name": "i18n(candyland:ui.data.constants.sugar_resource_name)",
         "icon": "/candyland/entities/construction/sugar/sugar.png",
         "stacks": 60,
         "default_resource": "candyland:resources:sugar:sugar_cube",
         "auto_queue_crafter_job": "stonehearth:jobs:mason",
         "builder_icon": "/candyland/entities/resources/sugar/builder_sugar.png",
         "builder_icon_hover": "/candyland/entities/resources/candy/builder_sugar_hover.png",
         "builder_sound_uri": "stonehearth:sounds:building_wall_stone",
         "tooltip": "candyland:ui.game.build_mode2.tooltips.sugar_resource"
      }
   }
}

  2. 我们需要将建筑颜色与我们的建筑资源关联。这些颜色不能已经为其他建筑资源定义。我们的 building_brushes.json 文件如下所示：

    {
   "always_available": {
      "sugar resource": {
         "colors": {
            "#E52F1B": {
               "display_name": "i18n(candyland:data.build.building_brushes.colors.#E52F1B)",
                "ordinal": 1
            },
            "#01A712": {
               "display_name": "i18n(candyland:data.build.building_brushes.colors.#01A712)",
               "ordinal": 2
            },
            "#EBEBE8": {
               "display_name": "i18n(candyland:data.build.building_brushes.colors.#EBEBE8)",
               "ordinal": 3
            },
            "#DAB869": {
               "display_name": "i18n(candyland:data.build.building_brushes.colors.#DAB869)",
               "ordinal": 4
            },
            "#A77C3F": {
               "display_name": "i18n(candyland:data.build.building_brushes.colors.#A77C3F)",
               "ordinal": 5
            }
         }
      }
   }
}

    所有这些颜色将可用于我们的糖资源的房间/墙壁、柱子、地板、道路和楼梯。

  3. 现在，我们通过清单中的 mixinto 将我们的通用资源和建筑颜色添加到 stonehearth mod 的相应索引中：

    "mixintos": {
   "stonehearth:data:resource_constants" : "file(data/resource_constants.json)",
   "stonehearth:build:brushes" : "file(data/build/building_brushes.json)"
}

  4. 我们需要使材料标签中包含"sugar"和"resource"的物品在游戏中可获得，可以通过采矿、采集、制作等方式。

添加建筑颜色

如何为现有建筑资源添加自定义颜色已在自定义指南中解释。您也可以在上文中查看自定义资源的示例。

添加地板图案

地板、墙壁和屋顶图案通过 mixinto 添加到 stonehearth/data/build/building_brushes.json。地板图案也将自动适用于道路。

    "mixintos" : {
       "stonehearth:build:building_brushes" : "file(data/build/building_brushes.json)"
    }

例如，我们的 mixinto 如下所示：

    {
       "pattern": {
          "sugar resource": {
             "patterns": {
                "candyland:build:brushes:pattern:green_cane_diagonal": {
                   "icon": "/candyland/ui/game/modes/build_mode/building_designer_2/images/brushes/green_cane_diagonal.png",
                   "display_name": "i18n(candyland:data.build.building_brushes.patterns.green_cane_diagonal)"
                }
             }
          }
       }
    }

地板图案列在 "pattern" 条目内，按建筑资源分组。在 "patterns" 键下，我们有图案的 URI 列表（URI 链接到 QB 模型），我们为 UI 定义 "icon"，为工具提示定义 "display_name"。

与颜色一样，图案不能在不同资源之间重复，因此请确保没有为同一建筑资源或在不同资源之间重复的图案 URI。

导入现有图案的 QB 模型以查看它们的设置方式。您可以在 stonehearth/data/build/brushes/pattern 中找到它们。地板图案不需要完全正方形，您可以制作矩形形状，甚至带有孔洞的形状。但它们必须只有 1 个体素高，因为地板和道路只有那么高。

还建议将您在图案中使用的颜色添加到相应资源的可用颜色列表中（如果尚未存在）。否则，它们在建筑成本列表中将被默认为木材。

不要忘记将图案的 URI 添加到您的清单中！

    "aliases" : {
       "build:brushes:pattern:green_cane_diagonal" : "file(data/build/brushes/pattern/green_cane_diagonal/green_cane_diagonal.qb)"
    }

添加墙壁图案

与添加地板图案相同，但 mixinto 将它们包含在 "wall" 条目内：

    {
       "wall": {
          "sugar resource": {
             "patterns": {
                "candyland:build:brushes:wall:cookie_wall": {
                   "icon": "/candyland/ui/game/modes/build_mode/building_designer_2/images/brushes/cookie_wall.png",
                   "display_name": "i18n(candyland:data.build.building_brushes.patterns.cookie_wall)"
                }
             }
          }
       }
    }

不要忘记为您的 .qb 模型添加别名：

    "aliases" : {
       "build:brushes:wall:cookie_wall" : "file(data/build/brushes/wall/cookie_wall/cookie_wall.qb)"
    }

与地板图案非常相似，您可以为图案制作不同的形状，但墙壁应该只有 1 个体素厚，否则在使用建造者中的房间/墙壁工具时会出现意外结果。

墙壁的 .qb 模型通常以 (0,0,0) 为中心，但您也可以将其制作在第一象限。如果您导入现有的墙壁图案（它们在 stonehearth/data/build/brushes/wall 内），您将看到可以使它们更短或更高，它们不需要具有游戏中的默认墙壁高度（6 个方块），建造者知道如何拉伸图案。它们也可以有不同的长度。

确保您的图案的颜色也在相应资源的可用颜色列表中（注意不要添加重复项）。

添加屋顶图案

屋顶图案像地板/墙壁图案一样添加到 building_brushes.json 文件中：

    {
       "roof": {
          "sugar resource": {
             "patterns": {
                "candyland:build:brushes:roof:cookie_castle_roof": {
                   "icon": "/candyland/ui/game/modes/build_mode/building_designer_2/images/brushes/cookie_castle_roof.png",
                   "display_name": "i18n(candyland:data.build.building_brushes.patterns.cookie_castle_roof)",
                   "type": "flat"
                },
                "candyland:build:brushes:roof:cotton_candy_roof": {
                   "icon": "/candyland/ui/game/modes/build_mode/building_designer_2/images/brushes/cotton_candy_roof.png",
                   "display_name": "i18n(candyland:data.build.building_brushes.patterns.cotton_candy_roof)",
                   "type": "peaked"
                }
             }
          }
       }
    }

请注意，我们在这里有一个额外的字段用于屋顶类型。如果是 "peaked"，屋顶模型将被视为 stonehearth mod 中的尖顶屋顶：我们可以使它们更高或更矮，并决定哪些侧面倾斜，哪些侧面应该放下墙壁来填补空隙。

如果是 "flat"，屋顶模型将被视为 rayyas_children 屋顶（或城堡屋顶）：它们将保持平坦，我们只能改变它们面向的方向。

屋顶只能作为图案添加，因为它们的 .qb 模型必须以特殊方式设置，以表示其不同部分，如内角/外角、侧面和中间。但玩家可以自由使用建筑编辑器中的方块工具创建更详细的屋顶或具有不同倾斜度的屋顶（例如，northern_alliance mod 中的屋顶就是使用方块工具制作的）。

让我们看看屋顶的 QB 模型（您可以在 stonehearth/data/build/brushes/roof 中找到更多示例）。尖顶屋顶的设置为 9 网格：

每个部分在不同的矩阵/层中。这些矩阵/层的名称是数字，必须与上图中的相同。由于部分在游戏中的放置和旋转方式，图案将根据倾斜度、大小等看起来不同，因此这是游戏中尖顶屋顶模型常用的大小。

平顶屋顶的设置为 13 网格（也考虑内角）：

请注意，内角的矩阵/层名称以 c 结尾，并基于其他角。我们需要在层中使用这些确切的名称。

对于平顶屋顶，如果我们愿意，可以使它们更高一些（例如，用于城堡屋顶的城垛），并增加中间部分的大小以使其具有图案。还要注意粘土板条屋顶的设置方式。屋顶的边缘与 X/Z 轴对齐，而木制部分向外突出。

确保将您使用的颜色添加到相应资源的建筑颜色列表中（如果尚未存在）。并且不要忘记将 .qb 模型的别名添加到清单中：

    "aliases" : {
       "build:brushes:roof:cotton_candy_roof" : "file(data/build/brushes/roof/cotton_candy/cotton_candy_roof.qb)",
       "build:brushes:roof:cookie_castle_roof" : "file(data/build/brushes/roof/cookie_castle/cookie_castle_roof.qb)"
    }

添加窗户

窗户和门有一个"切割器"，当您在设计建筑时将它们放置在墙壁中时，会自动为它们创建一个孔洞。

我们可以在 stonehearth/entities/construction 中找到窗户。它们像任何其他实体一样创建，我们可以通过配方、战利品等使它们可用。复制现有的窗户将确保我们使用现有的类别和材料标签，以及任何可能需要的 mixin（例如"stonehearth:mixins:placed_object"）。

窗户的 .qb 模型在 Qubicle 中以 0,0,0 为中心（并与地板对齐）。

以下是与通常不同的组件。来自 wooden_window_frame_ghost.json：

    "mob": {
       "align_to_grid": [
          "x"
        ]
    }

这是 mob 组件。"align_to_grid" 数组可以包含 "x"、"z" 或两者。在这种情况下，我们希望确保窗户 qb 模型的中心与 X 轴上最近的整数值对齐，以便它在墙壁中正确对齐。对齐一开始可能不明显（因为位置会根据模型/区域原点和 Qubicle 中的原始位置而变化），但如果未正确设置，当我们尝试将窗户放置在建筑设计中时，窗户将会偏移。

    "stonehearth:fixture": {
       "cursor": "stonehearth:cursors:create_window",
       "margin": {
          "left": 0,
          "right": 0,
          "top": 1,
          "bottom": 1
       },
       "bounds": {
          "min": { "x": -1, "y": 0 },
          "max": { "x": 1, "y": 2 }
       }
    }

这是 fixture 组件。"cursor" 字段不再使用。我们现在为建筑编辑器中的每个夹具类别都有不同的光标。"margin" 部分也不再使用，可以省略。现在我们允许在墙壁中彼此相邻地放置窗户。

然而，"bounds" 区域是必需的。窗户和门需要此区域（请注意，它仅用 X 和 Y 点定义，因为窗户和门旨在切割 1 个方块宽的墙壁）。它是我们在建造者中可以触摸以拖动窗户/门的区域（包括孔洞）。

对于我们可以在建造者中使用的其余可放置实体，我们需要确保它们至少定义了 1 个碰撞区域（无论其碰撞类型如何），否则建造者将不知道如何在其计算中正确使用它们。

    "stonehearth:portal": {
       "cutter": [
          {
             "min": { "x": -1, "y": 0 },
             "max": { "x": 1, "y": 2 }
          }
       ]
    }

这是 portal 组件。在这里我们指定窗户的切割器区域，也仅用 X 和 Y 点定义。当我们设计建筑时，当我们将窗户放置在墙壁上时，此区域将自动从墙壁中切割出来。该区域基于整个单位。我们不能切割建筑方块的分数。通常它与 fixture 组件的边界大小相同，但也可以是不同的大小。

窗户通常具有 "mob_collision_type" 为 "clutter"，以便炉民可以站在离它们很近的地方，但较大的窗户也可以定义碰撞区域。

较大的窗户还具有 "movement_modifier_shape"。这样做的目的是，如果您将它们放置在靠近地面或地面上，炉民会避免穿过它们。由于炉民喜欢较短的路径，有时他们会穿过高大的窗户而不是使用门。这有助于解决这个问题。示例：

  "movement_modifier_shape": {
     "modifier": -0.5,
     "nav_preference_modifier": -0.5,
     "region": [
        {
           "min": { "x": -1, "y": 0, "z": -1 },
           "max": { "x": 1, "y": 3, "z": 2 }
        }
     ]
  }

"modifier" 控制实体在区域内移动时的速度。"nav_preference_modifier" 影响实体在进行寻路时如何感知该形状中的速度。"region" 是应用修饰符的区域，它通常对应于窗户的切割器（请注意它具有 3 个维度）。

这些属性也用于道路等，以便炉民在上面走得更快。

添加门

门与窗户类似，它们也位于 stonehearth/entities/construction 下。大多数门具有相同的 mixins（来自 stonehearth/mixins/doors）以重用相同的动画表和其他常见属性。

在它们的 ghost JSON 文件中，fixture 组件中有一个名为"valign"的额外属性，但此属性也不再使用，就像 cursor 和 margins 一样。现在我们可以在墙壁中的任何位置放置门。

portal 组件的功能与窗户相同，如上所述。我们还使用 "align_to_grid" : [ "x" ] 以便它们可以适应墙壁内部。

通常，在它们的 ghost 文件的组件中（通过 mixin 或直接在 JSON 中），我们指示动画表和默认效果，例如：

  "render_info": {
     "animation_table": "file(/data/rigs/entities/construction/double_door/double_door.json)"
  },
  "effect_list": {
     "default": "file(/data/rigs/entities/construction/double_door/effects/closed.json)",
     "initial_pose_animation": "closed.json"
  }

"animation_table" 指向门的 rig（骨架）。效果列表中的 "default" 效果将是我们在放置门后门播放的默认动画，"initial_pose_animation" 是门处于 ghost 形式时定位门的动画效果的名称。

通常门在 Qubicle 中不以 (0,0,0) 为中心。它们在第一象限与地面对齐。它们的矩阵有名称，以便它们可以重用现有的 rig 进行动画。请务必检查 stonehearth mod 中的示例。

在实体的 JSON 文件（不是 ghost）中，我们还可以找到以下其他组件：

  • 在"stonehearth:entity_forms"组件内，我们可以看到 "placeable_on_walls" 和 "placeable_on_ground" 标志。这些使放置成为可能。对于只应放置在墙壁上的物品，我们使用第一个，对于只应放置在地面上的物品，我们使用第二个。

    对于门，我们同时使用两者，因为我们也可以为隧道入口放置门。

  • 与窗户一样，"region_collision_type" 为"none"。

  • 它具有 "sensor_list" 组件，其中包含传感器列表。这些是具有自定义标识符的条目，定义世界单位中的半径（忽略"type"字段，它未被使用）。

  • "stonehearth:door" 组件只有一个字段，指向我们在传感器列表中声明的传感器的标识符。然后，在门组件代码中，我们检索传感器并跟踪它。当实体进入/离开传感器半径时，传感器触发一些我们可以自定义的代码。在门的情况下，它将播放打开/关闭动画（如果炉民站在门附近则保持打开）。

  • 在 entity_data 中，我们有 "stonehearth:entity_radius"。这也以世界单位定义。它用于确定战斗等所需的距离（范围）（它是静态的只读数据，因此如果需要，我们可以在代码中的任何地方查询它）。

装饰和家具

这些像任何其他可放置的物品一样创建。为了使它们出现在建筑编辑器选项卡中，它们需要在目录数据中属于以下类别之一：decoration、construction、door、furniture、workshops、storage、window；并且要么您已经在库存中有它们，要么可以制作它们的工匠已经解锁了这些配方。

确保可悬挂的物品已正确设置所有属性。地毯/地板砖在 ghost 文件的 entity_data 中具有 "stonehearth:floor_deco" : {}，用于允许玩家在其上放置战斗命令的旗帜。

装饰、家具等不需要 fixture 组件，而是定义它们自己的碰撞区域。对于那些可以在建筑编辑器中使用的物品，请确保它们至少定义了一个区域，即使它们的碰撞类型是"clutter"/"tiny"/"none"，以便我们可以在蓝图中拖动它们，并且建造者知道如何在其计算中使用它们。
