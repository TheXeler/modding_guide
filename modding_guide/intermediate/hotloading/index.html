什么是热加载？

在 Stonehearth 的情况下，它意味着在运行时激活某些内容，即在游戏已经初始化所有内容之后。

何时应该热加载模组？

嗯，你可能创建了一个自定义王国或生物群系，并且希望仅在选择该王国/生物群系时启用某些战役或其他功能。例如，当我们选择使用 Rayya's children 模组游玩时，我们会覆盖一些图像，使 UI 看起来不同，当我们选择在沙漠生物群系中游玩时，我们会覆盖一些 goblins 和其他动物的模型，使它们看起来更像沙漠风格。

过去我们称之为"热加载模组"，但现在实际上应该称为"应用清单"，因为我们在这里不是激活整个模组。

热加载清单仅允许在世界生成完成之前 -或- 对于来自客户端 Lua 的 'client_only' 清单。

并非应用清单中的所有内容都会起作用。已知某些 mixintos 不会从热加载清单中正确应用，因为游戏已经缓存了这些值（例如，向初始作物列表中添加新作物）。在大多数情况下，你必须将这些添加到你的主清单中。

应用/热加载的清单通常位于我们模组的某个子文件夹内，并且在其 "info" 部分内有

  "deferred_load": true

标志。

如何热加载模组？

为了热加载额外的清单，我们有两个选项：

  • 如果我们制作了自定义生物群系，我们可以在生物群系的 JSON 文件（具有主要信息的文件，而不是生成数据文件）的 "applied_manifests" 键中应用其他清单，这将在玩家选择在我们的生物群系中游玩时应用：

    "applied_manifests": {
   "desert": "file(desert/manifest.json)"
}

  • 如果我们想仅在选择特定王国或使用其他条件时应用其他清单，那么我们需要使用 client_init_script。

    在你的模组中创建一个 Lua 文件，并在你的清单中添加：

    "client_init_script": "file(rayyas_children_client)"

    在根级别（例如，在 "info" 部分下方）。

    注意在这种情况下，我们在 "file()" 中不包含 .lua 扩展名。对于 "server_init_script" 我们也不包含它。

    上面的示例来自 Rayya's Children 模组。现在，在你的 init 脚本中，你需要有这样的内容（继续使用相同的示例）：

    rayyas_children = {
   constants = require 'constants'
}
local player_service_trace = nil

local function check_override_ui(players, player_id)
   -- Load ui mod
   if not player_id then
      player_id = _radiant.client.get_player_id()
   end

   local client_player = players[player_id]
   if client_player then
      if client_player.kingdom == "rayyas_children:kingdoms:rayyas_children" then
         -- hot load rayyas children ui mod
         _radiant.res.apply_manifest("/rayyas_children/ui/manifest.json")
      end
   end
end

local function trace_player_service()
   _radiant.call('stonehearth:get_service', 'player')
      :done(function(r)
         local player_service = r.result
         check_override_ui(player_service:get_data().players)
         player_service_trace = player_service:trace('rayyas children ui change')
               :on_changed(function(o)
                     check_override_ui(player_service:get_data().players)
                  end)
         end)
end

radiant.events.listen(rayyas_children, 'radiant:init', function()
   radiant.events.listen(radiant, 'radiant:client:server_ready', function()
         trace_player_service()
      end)
   end)

return rayyas_children

    我们有几个本地函数，我们将使用它们来跟踪选择了哪个王国并应用额外的清单。

    在底部，我们监听此脚本被初始化，当完成后，我们监听 'radiant:client:server_ready' 事件。一旦准备好，我们就可以开始检查（使用 stonehearth 服务）在登船屏幕中选择了哪个王国。

    如果王国 URI 是 "rayyas_children:kingdoms:rayyas_children"，那么我们调用：

    _radiant.res.apply_manifest("/rayyas_children/ui/manifest.json")

    这有效地应用了更改 UI 外观的额外清单。延迟清单的 client_only 标志设置为 true（如上面所述，这是热加载清单的要求之一）。你也可能在应用的清单中看到 deferred_load 标志，但对于它们来说不再需要。主清单也不应该有它。
