战斗类职业

战斗由许多文件控制：combat 服务、combat_state / target_tables 组件、stonehearth/ai/actions/combat 下的 AI 行为等。

让我们创建一个新的战斗类职业。我们称它为 Crusader（十字军），他们将是从 Footman（步兵）提升的高级职业。

添加战斗类职业

这些是手动添加战斗类职业的 steps。我们将快速浏览它们，因为通用部分已在另一个页面中解释，并解释战斗类职业特有的功能。

  1. 我们将从复制 footman 职业开始（你可以复制任何其他战斗类职业，如果你想制作远程单位，你可能想复制 archer）。

  2. 我们用 crusader 替换 footman 来重命名文件。

  3. 我们在模组的清单中为我们的职业添加别名，以及为我们的职业索引添加别名，以便在 mixinto 中使用它：

    "aliases": {
   "jobs:index": "file(jobs/index.json)",
   "jobs:crusader": "file(jobs/crusader/crusader_description.json)"
}

  4. 我们将控制器的别名添加到清单的 "controllers" 部分：

    "controllers":{
   "class:crusader": "file(jobs/crusader/crusader.lua)"
}

  5. 我们向职业索引添加 mixinto，以将我们的 Crusader 职业添加到现有职业列表中：

    "mixintos": {
   "stonehearth:jobs:index": "crusader:jobs:index"
}

    我们的 index.json 文件看起来像这样：

    {
    "jobs": {
        "crusader:jobs:crusader": {
            "description": "crusader:jobs:crusader"
        }
    }
}

    职业的键恰好与别名匹配，这就是 "description" 字段指向的内容。

  6. 由于护身符对应于实际武器，我们在 my_mod/entities/weapons 下创建了它。我们编辑其相关文件（battle_hammer_talisman.json、battle_hammer.json、battle_hammer_iconic.json 及其所有模型和图像）。

    我们还为护身符和武器添加别名：

    "crusader:talisman": "file(entities/weapons/battle_hammer/battle_hammer_talisman.json)",
"weapons:battle_hammer": "file(entities/weapons/battle_hammer)"

  7. 现在我们编辑 crusader_description.json 文件。有许多字段对于任何类型的职业都是通用的。以下是如何为战斗类职业设置它们：

    • "abilities" -- 对于我们的示例，crusader_abilities.json 文件将如下所示：

      {
   "type": "entity",
   "components": {
      "stonehearth:equipment_piece": {
         "injected_ai": {
            "ai_packs": [
               "stonehearth:ai_pack:patrolling",
               "stonehearth:ai_pack:wimpy",
               "stonehearth:ai_pack:panic:flee",
               "stonehearth:ai_pack:combat_control"
            ]
         },
         "injected_buffs": [
            "stonehearth:buffs:combat_basics"
         ],
         "military_score": 20
      }
   },
   "entity_data": {
      "stonehearth:combat:melee_attacks": [
         {
            "name": "combat_1h_forehand_spin",
            "effect": "combat_1h_forehand_spin",
            "active_frame": 21,
            "cooldown": 8000,
            "priority": 1
         }
      ],
      "stonehearth:combat:melee_defenses": [
         {
            "name": "combat_1h_dodge",
            "effect": "combat_1h_dodge",
            "active_frame": 8,
            "cooldown": 12000,
            "priority": 1
         }
      ]
   }
}

      注意我们注入了 patrolling AI 包，以便 crusader 知道如何巡逻，一个对所有战斗单位通用的 buff，增加他们的属性，以及初始攻击和防御技能（动画效果）。

      equipment_piece 组件内的 "military_score" 属性将使城镇的军事分数每次有人提升到 Crusader 时增加那里定义的值。在 stonehearth 模组中，只有高级战斗单位有它。这样我们可以通过每次提升/降级 Hearthling 到战斗职业来自动增加/减少更多的分数。

      军事分数（你将在城镇概览菜单中看到）基于 Hearthlings 的武器/工具/盔甲，加上我们拥有的战斗单位（通过这些 abilities 文件）。游戏的某些方面将根据你的军事分数进行缩放。

      如果你查看 stonehearth 模组中的其他示例，你会看到我们可以在此处添加任何其他 AI 包，或不指定任何攻击（这通常取决于武器本身，以便动画与武器匹配）。Abilities 文件就像默认的equipment piece或职业附带的默认技能。

    • "task_groups": [] -- 对于我们的示例，此数组将如下所示：

      "task_groups": [
  "stonehearth:task_group:common_tasks",
  "stonehearth:task_group:restock",
  "stonehearth:task_groups:rescue"
]

      这里我们有很少的任务组，因为我们希望我们的单位专注于战斗。注意战斗单位没有 town_alert AI 包/任务组。对于 cleric，他们也有 healing 任务组。

    • "xp_rewards": {} -- 对于战斗单位，我们这样设置它们：

      "xp_rewards": {
   "town_protection": 1
}

      这是他们在保护城镇时（无论是巡逻还是站在防御区域）不时获得的经验值量。它通过监听 combat_job.lua 中的事件来控制。他们通过杀死敌人获得的经验值量取决于 monster_tuning 文件（它将随难度缩放，每个怪物将奖励不同数量的经验值）。

      所有队伍成员在战斗中获得相同的经验，无论谁杀死了敌人。

      Cleric 在他们的 "xp_rewards" 中还有几个更多设置。一般来说，你可以通过 job 组件的 add_exp 函数授予经验值。

    • "level_data": {} -- 战斗单位通常解锁永久 buff 和新的战斗技能。

      例如，Crusader 将有这两个特权：

      "level_data": {
   "1": {
      "perks": [
         {
            "type": "add_combat_action",
            "name": "i18n(crusader:jobs.crusader.crusader_description.level_1_data.perk_000_name)",
            "id": "crusader_holy_punch",
            "icon": "file(images/holy_punch.png)",
            "equipment": "crusader:crusader:holy_punch",
            "action_type": "stonehearth:combat:melee_attacks",
            "description": "i18n(crusader:jobs.crusader.crusader_description.level_1_data.perk_000_description)",
            "level": 1,
            "demote_fn": "remove_combat_action"
         }
      ]
   },
   "2": {
      "perks": [
         {
            "type": "apply_buff",
            "name": "i18n(crusader:jobs.crusader.crusader_description.level_2_data.perk_000_name)",
            "id": "crusader_heal_aura",
            "icon": "file(images/cleric_perk_healing_aura.png)",
            "buff_name": "crusader:buffs:crusader:heal_aura",
            "description": "i18n(crusader:jobs.crusader.crusader_description.level_2_data.perk_000_description)",
            "level": 2,
            "demote_fn": "remove_buff"
         }
      ]
   }
}

      • 第一个有 "type" : "add_combat_action"。这是在 combat_job.lua 中定义的函数的名称。注意我们有 "demote_fn" 字段，具有相反的函数，"remove_combat_action"。

        我们声明一个 "action_type"，以便游戏知道在哪里注入新的战斗技能（在这种情况下，"stonehearth:combat:melee_attacks"）和一个 "equipment"（在这种情况下，"crusader:crusader:holy_punch"），它是描述 equipment piece 的 JSON 文件的 URI，通常是一个不可见的，只添加新战斗技能（动画）或可施加 debuffs 等的。

        请记住，伤害是在武器中定义的，而不是在攻击本身中定义的，并且它将受到属性、buff 或攻击的一些属性（与动画信息一起添加）等因素的影响。对于治疗，我们有 "stonehearth:combat:healing_spells"，而不是近战攻击。它用于 cleric，除了动画属性外没有任何特殊属性，因为治疗量取决于武器的 entity_data：

        "stonehearth:combat:healing_data": {
   "base_healing": 5
}

        通常我们会将从特权解锁的技能的 "priority" 设置得稍高一些，以便它们比基本攻击更有机会运行。

      • 第二个有 "type": "apply_buff"（带有 "demote_fn": "remove_buff"）。这些函数在 base_job.lua 中定义。我们声明一个 "buff_name" 属性，链接到我们要应用的 buff 的 URI（通常是永久 buff）。

      base_job.lua 和 combat_job.lua 中还有一些更多我们可以用于特权的函数。

  8. 我们现在将继续编辑其余文件：

    • 我们编辑默认职业服装的文件，这与任何其他类型的盔甲相同，通常包括盔甲数据：

      "stonehearth:combat:armor_data": {
   "base_damage_reduction": 7
}

    • 我们编辑控制器文件。这非常重要。我们重命名的控制器文件仍将具有相同的代码，所以我们必须在其中重命名一些内容。

      游戏中的所有职业都继承自 base_job.lua。战斗类职业改为继承自 combat_job.lua，因为该文件已经继承自 base_job.lua。CombatJob 类在 combat_job.lua 中定义，负责在战斗单位提升时将它们添加到默认队伍中，在他们降级到非战斗职业时移除它们，并管理战斗单位在巡逻时的经验获取。

      在我们重命名 Footman 为 Crusader 后，我们的 crusader.lua 文件将如下所示：

      local CrusaderClass = class()

local CombatJob = radiant.mods.require 'stonehearth.jobs.combat_job'
radiant.mixin(CrusaderClass, CombatJob)

return CrusaderClass

      注意我们如何将 CombatJob 类 mixin 到我们的 CrusaderClass，以检索战斗单位的通用代码。上面的示例是战斗类应具有的最少代码。如果需要，我们可以在此 Lua 文件中添加更多自定义函数。例如，查看 cleric.lua。

      为了覆盖继承的函数，我们将这样做（在从 CombatJob 类继承之后并在返回我们的 CrusaderClass 类之前）：

      -- Overriding standard combat class functions looks like this
function CrusaderClass:promote(json_path, options)
   -- First call the combat class's version
   CombatJob.promote(self, json_path, options)
   -- Your custom code goes here
end

      这允许我们在需要向类添加额外变量或任何新行为时扩展功能。

  9. 现在我们为护身符添加制作配方，使其在游戏中可获得，或以其他方式使其可获得。

  10. 最后，我们可以在游戏中测试我们的新战斗类职业，并确保一切按预期工作。

    你可以使用 Item Stamper debugtool 生成所需的护身符，并通过晋升 UI 提升 Hearthlings，或者如果你为其制作了配方，让工匠制作护身符等。

    如果你启用了 debugtools 模组，你也可以使用默认控制台提升他们，例如：

      promote_to crusader:jobs:crusader

    然后检查他们是否被添加到队伍菜单中，巡逻，可以战斗等。

关于攻击和防御

我们可以轻松地使用 equipment pieces 添加新的战斗技能。早些时候我们已经解释了它们可以具有的基本属性。但除了战斗动画效果的信息外，还有更多。

你可以检查现有的战斗类职业、盔甲和武器以获取示例。

对于近战攻击：

  • "damage_multiplier" -- 使用此属性，当执行攻击时，总伤害将根据此乘数增加或减少。

  • "aggro_addition" -- 我们可以使此攻击增加对我们战斗单位的 aggro。

  • "aggro_multiplier" -- 与上述属性类似，但使用乘数而不是添加一些数量。如果两者都存在，将在添加之前应用乘数。

  • "minimum_damage" -- 我们可以限制此攻击应造成的最小伤害。如果伤害为负，则将其更改为 0 或 1。

  • "inflictable_debuffs" : {} -- 与其他 inflictable debuffs 相同的语法。我们可以使此攻击有机会对目标施加 debuff。

    "inflictable_debuffs": {
   "infection": {
      "uri": "stonehearth:buffs:knight:taunted_effect",
      "chance": 1.0
   }
}

  • "aoe_effect" : {} -- 此攻击将造成 aoe 伤害。它至少包括一个 "aoe_range" 属性（aoe 伤害将生效的世界单位范围，默认为近战攻击的范围）和一个 "aoe_target_limit" 属性（受 aoe 影响的最大目标数量，默认为 10）。它还可以包括上述任何字段，以便 aoe 攻击可以增加 aggro，对多个目标施加 debuffs 等。

对于远程攻击：

  • 除了正常的 "active_frame" 属性外，我们可以有一个 "active_frames" : [] 数组，以指定多个帧。它用于 archer double shot 技能。我们也可以将此属性用于近战攻击。
  • "target_armor_multiplier" -- 通常是 0 到 1 之间的值，这将减少目标的盔甲，因此伤害将更高。也可用于近战攻击。

对于近战防御：

  • "chance_of_success" -- 通常是 0 到 1 之间的值（默认为 1）。它是阻挡攻击的机会。
  • "defense_attribute" -- 我们可以用来增加成功机会的属性的名称，默认为 0。我们可以在此字段中使用任何属性，但要使其生效，Hearthlings 必须在其 attributes 组件中已有它。

使用 SHED

免责声明：这些步骤指的是在此页面编写时使用 SHED 创建战斗类职业。

  • 首先我们需要从 stonehearth 模组克隆一个职业节点（例如，'jobs:footman'）。

  • 右键单击它并点击 'Clone!'。选择你的模组作为克隆的目标模组，并在替换文本框中输入你的战斗类职业的名称。

  • 点击 clone 并审查依赖项。如果你不使用 buffs 等中的任何不需要的文件，请取消选中它们，如果你想为其余文件的名称添加更多替换文本，以便它们已经以你想要的名称克隆，请返回。

    确保取消选中控制器的别名（your_mod:class:your_job），因为它将在 aliases 节点内创建，这是我们不想要的。点击 'Accept'。

  • 一旦 SHED 刷新视图，查找 stonehearth 模组的 controllers 节点并找到（例如）'class:footman' 节点。克隆它，用你的自定义职业的名称替换 footman，但取消选中 Lua 文件，因为我们已在上一步中克隆了它。

    这将使控制器的别名克隆到你清单的 "controllers" 部分，这是我们想要的。

  • 现在在 stonehearth 模组中搜索 'weapons:wooden_sword' 节点并将其克隆到你的模组，以便你的护身符 JSON 文件检测到缺少的模型和图像。

  • 选择你克隆的职业节点并开始编辑文件：

    • 编辑 "alias"、"controller"、"talisman_uri" 和任何其他带有别名的字段，以便它们指向你模组的清单。如果 SHED 检测到它们指向的文件，它将以浅绿色突出显示它们。

    • 如果你的职业特权包括 buffs，编辑它们的别名，以便它们指向 stonehearth 模组（如果你计划重用它们）或你的模组（如果你已克隆它们）。战斗技能也是如此。

      请记住，效果的名称基于 stonehearth 模组。如果你计划添加自定义动画，请确保正确设置 overrides，以便 Hearthlings 检测到它们。

    • 保存更改。

  • 最后，继续编辑模型、特权数据、控制器等，如上述指南中所述，直到你克隆的职业准备好进行测试。你可以使用 'Localize This File' 按钮尝试自动本地化你的 JSON 文件（请注意，对于某些文件可能不起作用）。

    记住通过 SHED 外部的 mixinto 将你的职业添加到职业索引中，否则你将无法测试它。还要为护身符制作配方或以其他方式使其可获得。
