创建新职业

这些是创建新职业的通用步骤：

  1. 从 stonehearth 模组复制现有职业。最好选择一个在功能上与你正在创建的最接近的职业。现有职业位于 stonehearth/jobs 目录内。

  2. 照常重命名文件。

  3. 在清单中为你的自定义职业添加别名，指向包含职业描述的 JSON 文件。

  4. 在清单的 "controllers" 部分添加别名，指向你的职业的 Lua 文件。例如：

    "controllers": {
   "class:lamplighter": "file(jobs/lamplighter/lamplighter.lua)"
}

  5. 向 "stonehearth:jobs:index" 添加 mixinto，以将你的自定义职业添加到可用职业列表中。

  6. 从编辑护身符开始。适当更改模型并编辑相应的 JSON 文件以指向你的自定义职业的文件。

  7. 编辑职业描述 JSON 文件。此文件中的数据将根据职业类型（制作/战斗/等）而有所不同。你需要向清单中添加更多别名，至少包括护身符、服装和工具/武器。

  8. 编辑其余的模型、UI 和 JSON 文件（服装、工具、工作室皮肤等）。这些文件可能因职业类型而异。还可能涉及 Lua 文件，确保根据需要重命名文本。

护身符（Talisman）

Stonehearth 中的每个职业（除了 Worker，这是基础职业）都使用我们称为护身符的物品来解锁职业并提升 Hearthling。提升后，护身符被消耗并替换为其可装备版本（实际的工具/武器），Hearthling 在执行工作或战斗时使用它。

通常我们用职业名称加工具本身的名称来命名相关文件，例如 farmer_hoe.json、farmer_hoe_talisman.json、farmer_hoe_iconic.json。

  • job_tool_talisman.json 文件类似于其他实体的文件。在其中，我们有物品属性 mixin。

    对于 entity_data，我们有净资产和 catalog（确保类别是 "tools" 并且 material_tags 包括 "stockpile_tool"，以便正确地补货到存储中）。

    对于组件，我们有 model variants，指向 iconic 版本的模型，以及具有默认效果的 "effect_list"（在 stonehearth 模组中，护身符和工作台的 iconics 具有此粒子效果，以便在库存或地面上快速区分它们）。

    你可能还会在一些护身符文件中看到一个名为 "stonehearth:promotion_talisman" 的组件。这是一个已弃用且不再存在的旧组件，因此你可以为你的职业省略它。

  • 现在我们编辑 job_tool.json 文件。"model_variants" 组件中的模型是已装备工具的模型，所以记得在你的体素编辑器中定位它，以便 Hearthling 可以正确抓取它。

    Entity forms 组件仅指向 iconic 形式，因为此物品不可放置，所以我们不需要 ghost JSON 文件。

    Equipment_piece 组件通常有一个 "postures" 字段，因为这是他们在战斗中使用的武器（即使这不是战斗职业，他们在城镇警戒模式下仍然会战斗）。

    在 entity_data 中，除了通常的 "stonehearth:net_worth" 和 "stonehearth:catalog" 条目外，我们有 "stonehearth:combat:weapon_data"，用于定义工具在城镇警戒模式下用作武器时的范围以及应该造成多少伤害；以及 "stonehearth:combat:melee_attacks"，用于定义基本攻击。

    对于战斗单位，高级攻击通常在其他文件中指定，以便我们可以通过特定武器或在特定等级解锁它们。

  • 然后我们编辑 job_tool_iconic.json，并为护身符和工具（武器）添加别名。

  • 最后，不要忘记使护身符在游戏中可获得！通过配方、战利品、任务奖励，任何你想要的。

职业描述

这些是我们可以在所有职业的描述 JSON 文件中拥有的通用字段：

  • "type": "job" -- 此文件表示一个职业。

  • "enabled" -- 通常我们希望将其设置为 true。游戏中没有动态启用职业的功能，所以只有当你的职业正在进行中且你还不希望它出现在晋升树中时，才将其设置为 false。还有一种可能性是使用 "in-progress" 而不是 true/false，这将使其在晋升树中以 '?' 符号作为图标出现，并用更多问号隐藏名称和数据，以及防止玩家提升到该职业。

  • "job_id" -- 唯一标识符。

  • "display_order" -- 一个序数，使我们的职业图标在职业索引中出现在其他职业之前/之后。由于模组可以在不了解其他职业的情况下添加更多职业，因此重复的序数是正常的，在这种情况下顺序可能不如预期。

  • "alias" -- 我们职业的别名，前面加上命名空间和冒号。

  • "display_name" -- 职业的本地化显示名称，用于 UI 的不同部分。

  • "controller" -- 此职业控制器的 URI，前面加上命名空间和冒号。请记住，它必须在清单的 "controllers" 部分中定义，而不是在 "aliases" 下。

  • "description" -- 我们职业的本地化描述，将在我们选择职业时显示在晋升树的左下角。

  • "requirements" -- 描述此职业要求的本地化文本，将在我们选择时显示在晋升树的底部。

    所需护身符的图标和名称，以及所需的父职业/等级将自动显示。但我们可以在这里添加自定义描述，例如说明哪个职业可以制作此护身符，如何获得它，或一些风味文本（查看 worker 职业的要求）。

  • "abilities" -- 职业能力 JSON 文件的路径（它定义了一个equipment piece，我们可以向其中注入 AI / 等，以便 Hearthling 知道如何执行他们的新职业）。

  • "equipment" -- 包含引用装备槽指向 equipment pieces 的键。通常我们在 "torso" 槽中应用职业服装，在 "mainhand" 槽中应用工具/武器。我们也可以在这里告诉游戏为 Hearthling 选择随机 equipment piece。例如，查看 worker_description.json。

  • "roles" -- 由空格分隔的角色字符串或角色数组（这对模组兼容性更好）。通常一个角色将是职业名称加上 _job 后缀，根据职业类型，我们将添加更多角色，如 "crafter"、"combat"、"mail_wearer" 等。这样他们将能够装备在其 equipment_piece 组件中具有这些角色的装备。

  • "talisman_uri" -- 我们护身符实体的别名，前面加上我们模组的命名空间和冒号。

  • "default_stance" -- 可以是 "passive"（用于大多数职业）、"aggressive"（用于战斗职业）或 "defensive"（用于某些 NPC 职业，只有在被攻击时才会攻击）。

  • "icon" -- 将在 UI 的不同部分为我们的职业出现的图标的图像路径。晋升树中的图标不需要方形背景，那是自动添加的。

  • "promotion_activity_name" -- 现在未使用的标识符。它旨在用于城镇日志。

  • "task_groups": [] -- 任务组别名的数组，当 Hearthling 提升到此职业时将应用于他们，并在我们提升到另一个职业或降级时移除。

    根据职业类型，我们可能希望包含/不包含某些任务组。例如，如果我们不包含 "stonehearth:task_groups:build"，Hearthling 在提升后将无法建造，并且他们在市民面板中 Build 列下的复选框将被禁用。一些职业将不包括允许 harvesting 资源或移动物品的任务组，以便 Hearthling 更专注于他们的职业。

    没有任务组列表，但它们的别名在 stonehearth 清单中分组。你通常需要与你复制的职业相同的任务组。更多关于 AI 的内容见指南的高级部分。

  • "parent_job" -- 父职业的 URI。

  • "max_level" -- 如果此职业可以升级（除 Worker 外所有职业都可以），确定 Hearthling 可以达到的最大等级。

  • "xp_rewards": {} -- 此字段的内容可能因职业类型而异，它们用于定义 Hearthling 通过执行其职业中的不同任务将获得多少经验。

  • "level_data": {} -- 此职业的职业特权列表。见下文。

关于职业特权

job_description.json 文件中职业特权的工作方式是：

  • 它们都由 JSON 对象数组组成。

  • 最小字段是 "name"、"id"、"icon"、"description" 和 "level"。"level" 用于解锁它们。"id" 是一个标识符，用于在我们需要知道 Hearthling 是否已解锁特权时在 Lua 中执行检查。

  • 如果特权包括由其他文件自动控制的内容（如解锁配方 - 每个配方定义了在哪个等级解锁），则不需要额外字段。

  • 如果特权涉及更复杂的内容，它将有一个 "type" 字段以及任何额外的自定义数据字段。

    这个 "type" 是在职业控制器文件中定义的函数的名称。有时它将在 stonehearth/jobs/base_job.lua 中定义，因为所有职业都继承自它。额外字段可用于传递给该自定义函数的数据。

    这样我们可以拥有为 Hearthling 添加永久 buff 的特权，或装备添加自定义 AI 行为的隐形装备，使他们可以做更花哨的事情（如新的战斗技能），或增加城镇可以拥有的炮塔/魔像的最大数量，例如。

  • 与使用自定义函数的特权一起，你还将看到一个 "demote_fn" 字段。这指定了如果 Hearthling 拥有该特权并被降级时应运行什么。这是一种确保在 Hearthling 提升/降级到其他职业时移除永久 buff 和自定义能力的方法。如果你不添加它，Hearthling 将保留这些能力。

你还会看到许多职业在 6 级将获得一个头衔而不是/除了特权之外，这使我们能够在 UI 中知道他们处于最大等级。此头衔在 "perks" 数组之外。现在它不是必需的，所有 Hearthlings 将获得一个将在角色表中显示并随着他们升级而改变的头衔（Apprentice、Journeyman、Master）。

除了特权外，Hearthlings 每次升级时将获得 +10 生命值。这些将在提升/降级后保留。 当你提升/降级 Hearthling 时，职业经验将重置，以鼓励专业化。

关于职业 progression

在 base_human.json mixin 内的 job 组件中，我们可以找到所有职业通用的信息：

  "stonehearth:job": {
     "starting_level_title": "i18n(stonehearth:stonehearth_mixins.base_human.job.starting_level_title)",
     "mid_level_title": "i18n(stonehearth:stonehearth_mixins.base_human.job.mid_level_title)",
     "max_level_title": "i18n(stonehearth:stonehearth_mixins.base_human.job.max_level_title)",
     "default_level_title": "i18n(stonehearth:stonehearth_mixins.base_human.job.default_level_title)",
     "default_level_announcement": "i18n(stonehearth:stonehearth_mixins.base_human.job.default_level_announcement)",
     "default_promote_announcement": "i18n(stonehearth:stonehearth_mixins.base_human.job.default_promote_announcement)",
     "xp_equation_for_next_level": "curr_level * 115 + 135",
     "levels_between_attribute_increase": 10,
     "level_up_data": {
        "level_up_perk_description": "i18n(stonehearth:stonehearth_mixins.base_human.job.level_up_perk_description)",
        "level_up_perk_icon": "/stonehearth/data/images/race/human_HP_on_level.png"
     }
  }

我们可以通过 mixintos 自定义这些字段。例如，我们有基于等级的头衔（Apprentice、Journeyman、Master），以及用于计算达到下一级所需经验量的方程式。请注意，并非所有这些字段现在都在使用中。

创建其他类型的职业

只要它们继承自 base_job.lua，你就可以创建其他类型的职业。例如，在游戏中我们有生产者（农民、捕猎者和牧羊人），他们的工作基于使用区域。

但如果你不想，你不需要使用区域。仅使用自定义AI 就足够了。例如，在原版游戏中，一些工匠有其他职责（草药师治疗、厨师 farming、工程师修理炮塔），他们并不专门致力于制作。而 Workers 既不是战斗职业也不是制作职业。
