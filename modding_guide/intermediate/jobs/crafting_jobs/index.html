制作类职业

让我们创建一个新的制作类职业。我们称它为 Lamplighter（点灯人），他们将制作用于点亮游戏中路灯的燃料（这个示例职业来自 Silent Woods 模组）。

  • 添加制作类职业
  • 使用 SHED

添加制作类职业

这些是手动添加制作类职业的 steps。我们将快速浏览它们，因为通用部分已在另一个页面中解释，并解释制作类职业特有的功能。

  1. 我们将从复制 herbalist 职业开始（你可以复制任何其他制作类职业）。

  2. 我们用 lamplighter 替换 herbalist 来重命名文件。

  3. 我们在模组的清单中为我们的职业添加别名，以及为我们的职业索引添加别名，以便在 mixinto 中使用它：

    "aliases": {
   "jobs:index": "file(jobs/index.json)",
   "jobs:lamplighter": "file(jobs/lamplighter/lamplighter_description.json)"
}

  4. 我们将控制器的别名添加到清单的 "controllers" 部分：

    "controllers":{
   "class:lamplighter": "file(jobs/lamplighter/lamplighter.lua)"
}

  5. 我们向职业索引添加 mixinto，以将我们的 lamplighter 职业添加到现有职业列表中：

    "mixintos": {
   "stonehearth:jobs:index": "silent_woods:jobs:index"
}

    我们的 index.json 文件看起来像这样：

    {
    "jobs": {
        "silent_woods:jobs:lamplighter": {
            "description": "silent_woods:jobs:lamplighter"
        }
    }
}

    职业的键恰好与别名匹配，这就是 "description" 字段指向的内容。

  6. 我们编辑护身符文件（lamplighter_staff_talisman.json、lamplighter_staff.json、lamplighter_staff_iconic.json 及所有模型和图像）。

    我们为护身符和工具（武器）添加别名：

    "lamplighter:staff": "file(jobs/lamplighter/lamplighter_staff)",
"lamplighter:talisman": "file(jobs/lamplighter/lamplighter_staff/lamplighter_staff_talisman.json)"

  7. 现在我们编辑 lamplighter_description.json 文件。有许多字段对于任何类型的职业都是通用的。以下是制作类职业特有的字段以及如何设置一些通用字段：

    • "abilities" -- 对于我们的示例，lamplighter_abilities.json 文件将如下所示：

      {
   "type": "entity",
   "components": {
      "stonehearth:equipment_piece": {
         "injected_ai": {
            "ai_packs": [
               "stonehearth:ai_pack:crafting",
               "stonehearth:ai_pack:ladders",
               "stonehearth:ai_pack:place_item",
               "stonehearth:ai_pack:town_alert",
               "stonehearth:ai_pack:wimpy",
               "stonehearth:ai_pack:panic",
               "stonehearth:ai_pack:panic:flee",
               "stonehearth:ai_pack:panic:cower",
               "silent_woods:ai_pack:refueling"
            ]
         },
         "injected_commands": []
      }
   }
}

      注意我们注入了 crafting AI 包，以便 lamplighter 知道如何制作，town_alert 以便他们可以集结，以及其他一些基本的 AI 包，使他们能够放置梯子、物品和逃跑。我们还添加了一个在我们的模组中定义的自定义 AI 包（注意命名空间不同）。

      如果你查看 stonehearth 模组中的其他示例，你会看到我们可以在此处添加任何其他 AI 包，例如 Cook 有 farming AI 包，Herbalist 有 healing AI 包，Engineer 有 mining AI 包。

    • "task_groups": [] -- 对于我们的示例，此数组将如下所示：

      "task_groups": [
  "stonehearth:task_groups:common_tasks",
  "stonehearth:task_groups:restock",
  "stonehearth:task_groups:placement",
  "stonehearth:task_groups:build",
  "stonehearth:task_groups:rescue",
  "stonehearth:task_groups:crafting",
  "stonehearth:task_groups:town_alert"
]

      我们确保它包括 crafting 任务组，除了通常的任务组。在我们的情况下，我们还包括了 build 任务组，但不包括 mining 任务组，所以 Lamplighter 将能够建造但不能挖掘。

    • "xp_rewards": {} -- 对于工匠，我们这样设置它们：

      "xp_rewards": {
   "craft_level_0": 15,
   "craft_level_1": 17,
   "craft_level_2": 19,
   "craft_level_3": 21,
   "craft_level_4": 23,
   "craft_level_5": 25,
   "craft_level_6": 27
}

      工匠将根据配方的等级要求获得那么多经验值。数量还将取决于 Curiosity 属性，以及工匠等级和配方等级之间的差异（检查 crafter 组件）。

      升级所需的经验值量随着每个等级而增加，所以我们通常逐步设置经验奖励。craft_level_0 存在是因为在过去，职业的等级从 0 开始。现有职业可能仍在使用它。如果你将所有配方设置为具有 1 或更高的等级要求，你的自定义工匠可能不需要它。

    • "level_data": {} -- 大多数 stonehearth 工匠具有相同的特权，所以如果你想，你可以重用它们（解锁配方或制作 fine/excellent 质量物品的能力）。

      配方将在工匠达到其等级要求时自动解锁，所以只有当你的工匠具有该等级要求的配方时才添加那种特权。Fine/excellent 质量只有在生产的物品具有

      "stonehearth:item_quality": {
   "variable_quality": true
}

      在其 entity_data 中才有可能。所以只有当你的生产物品具有可变质量时才添加那些特权。对于所有工匠，他们将在 3 级开始制作 fine 物品（无论配方的等级），在 5 级制作 excellent 物品。

    • "crafter": {} -- 此字段专用于具有制作能力的职业。这里我们有两个属性：

      • "work_effect" -- 效果文件的名称（仅名称，不要以 .json 结尾！），工匠在工作时将播放。通常包括动画轨道，可能还有声音轨道。对于 Lamplighter，我们将使用 "fiddle"。
      • "recipe_list" -- 配方列表 JSON 文件的路径。对于我们的 Lamplighter 示例，它将是 "/silent_woods/jobs/lamplighter/recipes/recipes.json"。

      配方列表按配方组和配方文件的标识符组织。你可以查看如何设置它这里。

    • "workshop": {} -- 工作室的属性：

      • "portrait" -- 基本工作台图像的路径。现在可能未使用。

      • "workbench_type" -- 基本工作台路径的 URI。现在可能未使用。

      • "skin_class" -- 工作室的 CSS (Less) 类的标识符。在我们的情况下，"lamplighter"。我们必须在清单的 "ui" 部分声明我们工作室的 .less 文件，如下所示：

          "ui":{
     "less":[
        "file(jobs/lamplighter/skin/workshop.less)"
     ]
  }

        它将在 UI 中定义我们工匠工作室的皮肤。我们的 workshop.less 文件看起来像这样：

          #craftWindow.lamplighter {
     background: url('skin.png');

     .item {
        background-image: url('skin.png');
     }

     #craftButton {
        position: absolute;
        right: 10px;
        top: -50px;
        width: 90px;
        height: 151px;
        #craftButtonImage {
           background: url('saw.png');
        }
     }

     #orderListContainer {
        table{
           background: #1f1e1e;
        }

        .orderListRow {
           background: #1f1e1e;
        }
     }
  }

        这里我们可以嵌套 CSS 属性并使用自定义背景（上面示例中的 skin.png）和自定义制作按钮（上面示例中的 saw.png）更改工作室 UI 的外观。我们可以在支持透明度的插图程序中编辑图像。注意在主元素（#craftWindow.lamplighter）中使用的 lamplighter 类。

      • "open_sound" 和 "close_sound" -- 当我们打开和关闭包含配方列表的工作室 UI 时播放的声音的别名。我们通常重用 stonehearth 模组中的声音，如 "stonehearth:sounds:ui:carpenter_menu:menu_open" 和 "stonehearth:sounds:ui:carpenter_menu:menu_closed"。

  8. 我们现在将继续编辑其余文件：

    • 与工作台相关的文件。注意 workbench.json 文件具有 workshop 组件，指向职业别名：

      "stonehearth:workshop": {
   "job_alias": "silent_woods:jobs:lamplighter"
}

      这用于将物品识别为工匠工作台，并将自动为其添加一个命令到单位框架，以便我们可以从中打开该工匠的工作室 UI。

      在其 entity_data 中，你可能会注意到以下部分：

      "stonehearth:table": {
   "drop_effect": "carry_putdown_on_table",
   "drop_offset": { "x": 0, "y": 1.25, "z": 0 }
}

      这些属性用于定义工匠将播放以将原料放在工作台顶部的效果，以及在工匠拿起之前定位掉落的原料和生产物品的偏移量。

      我们也可以在工作台.json 文件的 entity_data 中定义工作台等效物。例如：

      "stonehearth:workshop": {
   "equivalents": [ "stonehearth:mason:pedestal:amberstone" ]
}

      这将使需要 Mason pedestal 的配方可以使用 Amberstone pedestal 制作。另一个示例是升级的 Potter's kiln 和工作台。确保在声明它们时使用正确的命名空间。

      在工作台的 ghost 文件的 entity_data 中，你可能会看到此字段：

      "stonehearth:hide_child_entities_from_pathfinder": true

      这是一个标志，以便 Hearthlings 在尝试为其他 AI 行为查找该类型的物品时不考虑已掉落在工作台顶部的原料。

      工作台的 iconic 文件通常具有与护身符相同的默认效果。

    • 我们编辑工匠服装的文件，这与任何其他类型的盔甲相同，尽管通常它们不包括 armor_data。

    • 我们编辑控制器文件。这非常重要。我们重命名的控制器文件仍将具有相同的代码，所以我们必须在其中重命名一些内容。

      游戏中的所有职业都继承自 base_job.lua。制作类职业改为继承自 crafting_job.lua，因为该文件已经继承自 base_job.lua。CraftingJob 类在 crafting_job.lua 中定义，负责向 Hearthling 添加 crafter 组件并管理工匠在制作配方时的经验获取。

      在我们重命名 Herbalist 为 Lamplighter 并删除所有不需要的代码后，我们的 lamplighter.lua 文件将如下所示：

      local CraftingJob = require 'stonehearth.jobs.crafting_job'

local LamplighterClass = class()
radiant.mixin(LamplighterClass, CraftingJob)

return LamplighterClass

      注意我们如何将 CraftingJob 类 mixin 到我们的 LamplighterClass，以检索工匠的通用代码。上面的示例是制作类应具有的最少代码。如果需要，我们可以在此 Lua 文件中添加更多自定义函数。例如，查看 engineer.lua、herbalist.lua 或 geomancer.lua。

      为了覆盖继承的函数，我们将这样做（在从 CraftingJob 类继承之后并在返回我们的 LamplighterClass 类之前）：

      -- Overriding standard crafting class functions looks like this
function LamplighterClass:promote(json_path, options)
   -- First call the crafting class's version
   CraftingJob.promote(self, json_path, options)
   -- Your custom code goes here
end

      这允许我们在需要向类添加额外变量或任何新行为时扩展功能。

  9. 现在我们将我们的制作类职业添加到开始菜单中的工作室列表中。我们需要添加此 mixinto：

    "mixintos" : {
   "stonehearth/data/ui/start_menu.json" : "file(data/ui/start_menu.json)"
}

    Mixinto 文件将如下所示：

    {
   "crafter_menu": {
      "items": {
         "lamplighter": {
            "name": "i18n(silent_woods:ui.game.menu.crafter_menu.items.lamplighter.display_name)",
            "description": "i18n(silent_woods:ui.game.menu.crafter_menu.items.lamplighter.description)",
            "class": "button",
            "required_job": "silent_woods:jobs:lamplighter",
            "required_job_text": "i18n(silent_woods:ui.game.menu.crafter_menu.items.lamplighter.required_job_text)",
            "icon": "/silent_woods/jobs/lamplighter/images/lamplighter_menu_icon.png",
            "menu_action": "show_crafter_ui",
            "hotkey_action": "craft:lamplighter"
         }
      }
   }
}

    我们为新职业添加一个键，用于工具提示的本地化 "name" 和 "description"，CSS "class"（通常与我们复制的相同），"required_job"，这是我们工匠的别名，前面加上我们模组的命名空间和冒号，本地化的 "required_job_text"，当玩家尚未提升任何人到我们的制作类职业时将在工具提示中显示，用于菜单项显示的 "icon" 的路径，"menu_action" : "show_crafter_ui"（这将使点击按钮打开我们工匠的工作室），以及 "hotkey_action"，它将指向 hotkeys.json 中的热键，以便玩家可以绑定键以用它们打开我们工匠的工作室。

    请注意，如果我们在此处添加热键操作，我们将必须向 stonehearth/data/hotkeys.json 添加 mixinto 并在那里也添加操作。

    确保为另一个工匠创建你的护身符的配方，使其在游戏中可获得，或以其他方式使其可获得。

  10. 最后，我们可以在游戏中测试我们的新制作类职业，并确保一切按预期工作。

    你可以使用 Item Stamper debugtool 生成所需的护身符，并通过晋升 UI 提升 Hearthlings，或者如果你为其制作了配方，让另一个工匠制作护身符等。

    如果你启用了 debugtools 模组，你也可以使用默认控制台提升他们，例如：

      promote_to silent_woods:jobs:lamplighter

    然后检查工作室 UI 看起来是否正常，配方是否可以制作等。

使用 SHED

免责声明：这些步骤指的是在此页面编写时使用 SHED 创建制作类职业。

  • 首先我们需要从 stonehearth 模组克隆一个职业节点（例如，'jobs:carpenter'）。

  • 右键单击它并点击 'Clone!'。选择你的模组作为克隆的目标模组，并在替换文本框中输入你的工匠的名称。

  • 点击 clone 并审查依赖项。取消选中控制器的别名（your_mod:class:your_job），因为它将在 aliases 节点内创建，这是我们不想要的。点击 'Accept'。

  • 一旦 SHED 刷新视图，查找 stonehearth 模组的 controllers 节点并找到（例如）'class:carpenter' 节点。克隆它，用你的自定义职业的名称替换 carpenter，但取消选中 Lua 文件，因为我们已在上一步中克隆了它。

    这将使控制器的别名克隆到你清单的 "controllers" 部分，这是我们想要的。

  • 选择你克隆的职业节点并开始编辑文件：

    • 编辑 "alias"、"controller"、"talisman_uri" 和任何其他带有别名的字段，以便它们指向你模组的清单。如果 SHED 检测到它们指向的文件，它将以浅绿色突出显示它们。
    • 编辑特权图像的路径，使它们正确。对于制作类职业，如果你打算重用 stonehearth 模组中的图标，你可以将它们重写为绝对路径（删除最后一个括号并编辑路径的第一部分为 /stonehearth/jobs/common）。
    • 保存更改。

  • 你会注意到配方索引尚未被克隆。我们必须像我们对控制器别名所做的那样单独克隆它。

    • 在 stonehearth 模组的 'jobs:carpenter' 节点下搜索 'recipes' 节点，右键单击它，并点击 'Clone'。

      为了更容易克隆，你可以尝试使用 engineer 配方列表，它具有较少的项目并且更容易编辑。

    • 在克隆对话框中，将 Target mod 设置为你的模组。

    • 将 'recipes' 的替换文本设置为 'recipes' 并使用 + 按钮添加新的替换文本，以用你的职业名称替换 carpenter（如果你克隆了那个，则用 engineer 替换 carpenter）。这是为了使克隆的配方列表的路径与我们新职业的路径匹配。

    • 现在点击 'Clone!'。在预览窗口中，点击 'Uncheck All'，因为它试图克隆所有生产的物品，而我们不想要那样。

    • 仅选中 recipes.json 文件的路径（确保它指向你克隆的职业所在的文件夹），如果你愿意，也克隆一个配方 JSON 文件（例如，工作台的配方，但不要克隆其别名，因为我们在上面的某些步骤中已经有了）。点击 'Accept'。

  • SHED 刷新后，回到你克隆的职业文件并编辑配方列表的路径，使其指向你的模组。SHED 在编辑文件路径时可能会挂起几秒钟，但希望不会太糟糕。现在你必须转到 File -> Reload，以便在树视图中检测到配方列表。请耐心等待。

  • 现在你可以点击你克隆的配方的节点，删除所有 "recipes" 条目下的内容（如果你想重用它们或添加你的自定义类别，可以保留类别）。如果你在上面的几步中克隆了工作台的配方，你现在应该看到它以绿色突出显示，并被 SHED 在树视图中识别。

  • 最后，继续编辑模型、数据等，如上述指南中所述，直到你克隆的职业准备好进行测试。你可以使用 'Localize This File' 按钮尝试自动本地化你的 JSON 文件（请注意，对于某些文件可能不起作用）。工作室的 UI 文件必须在 SHED 外部添加到清单中，就像所有 mixintos 一样。
