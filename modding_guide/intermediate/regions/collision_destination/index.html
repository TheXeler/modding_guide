碰撞类型

通常我们会为我们的实体定义一个或多个碰撞区域（如果它们需要碰撞），但对于有知觉的实体，我们通常使用预定义的碰撞类型。如果 Hearthlings 应该能够穿过该实体行走，我们可能不需要添加 region_collision_shape 组件。

要使用默认碰撞类型，我们必须在 mob 组件内定义它们。例如：

  "mob": {
     "mob_collision_type": "clutter"
  }

可能的类型有：

  • "none" -- 用于火坑座位（它们是不可见的，Hearthlings 应该能够穿过它们坐下）和泰坦。

  • "clutter" -- 此类型将使 Hearthlings 能够穿过物品行走。用于小型植物、窗户等。

    具有此碰撞类型的物品将在其支撑方块被移除时被摧毁。这是为了防止在挖掘地面时产生大量随机杂物（如果物品没有定义碰撞类型但有碰撞区域，那么 Hearthlings 将无法挖掘其碰撞区域下方的方块）。

  • "tiny" -- 此类型通常用于生物和一些小物体。实际上，它与 "clutter" 类型相同（想象一个 1x1x1 的区域）。

  • "humanoid" -- 这是 Hearthlings 和怪物使用的类型。想象一个 1x3x1 的区域。

  • "titan" -- 这是一个未使用的类型。想象一个 7x6x7 区域加上一个 5x8x5 区域。物品和有知觉的生物在与如此大的区域交互时可能会漂浮得太高，反之亦然，因此不推荐使用。

我们无法使用 debugtools 看到这些区域类型，但请注意 Hearthlings 如何无法挖掘短于 3 个方块的隧道，也无法到达那么小的洞中的物品，因为他们会与天花板碰撞。

也可以从 QB 模型派生区域，但如果它是一个更简单的形状，性能会更好。并且它将从原始模型以世界单位翻译，无论游戏中模型的缩放比例如何，因此建议定义你自己的碰撞区域或使用上述类型之一。

例如，这来自 large_boulder_1.json：

  "region_collision_shape": {
     "region_from_model": {
        "model": "file(large_boulder_1.qb)"
     }
  }

像树木一样，巨石的缩放比例与地形相同（模型中的 1 个体素是 1 个世界单位），因此碰撞区域将匹配。但如果我们在 JSON 文件中将它们的缩放比例更改为更大或更小，碰撞区域仍将保持这么大（因为我们无法定义具有小数值的区域）。

如上所述，最好尽可能使用简单的矩形形状。当可以在同一位置放置许多这些实体时，"clutter" 类型也很好。

碰撞区域

碰撞区域告诉游戏 Hearthling 不能穿过对象行走，并且我们不能在该区域放置任何其他物品。如果没有碰撞区域，Hearthlings 将能够直接穿过对象！

region_collision_shape 组件根据其自己的坐标系定义碰撞区域。+Y 与世界坐标系相同，但 +X 和 +Z 相反。这样定义是为了使大多数值为正（X 或 Z 方向大于 2 个方块的对象可能不会所有值都为正）。

如果你按 F11 并选择一个实体，你应该能够看到其碰撞区域作为一个带有浅蓝色边缘的透明框。它通常与目标区域重叠，所以有时很难看到一个或另一个。

以下是从 stone_chair.json 中获取的示例：

  "region_collision_shape" : {
     "region": [
        {
           "min" : { "x" : 0, "y" : 0, "z" : 0 },
           "max" : { "x" : 1, "y" : 2, "z" : 1 }
        }
     ]
  }

在此示例中，我们可以看到对象在 X 和 Z 方向上宽 1 个方块，在 Y 方向上高 2 个方块，从我们之前定义的 region_origin 开始。如前所述，Z 方向朝北，X 方向朝西。

你可以在 "region" 数组内添加多个碰撞形状。确保每个区域的 "min" 值低于 "max" 值。

你不能有覆盖世界单位分数的区域。值必须以整块定义。如果你使用小数值，或者如果区域未与方块边缘对齐，游戏内部仍将使用离散值，所以你可能会注意到 Hearthlings 没有在你定义的确切位置交互。

默认情况下，Hearthlings 可以走过高度为 1 单位的任何对象，所以如果你的碰撞立方体太低，Hearthlings 将能够在其顶部行走。

为了手动调整我们的碰撞区域，我们可以简单地在我们的 JSON 文件（根实体的文件，而不是 ghost！）中添加

  "region_collision_shape": {}

然后打开游戏并使用我们的物品的Entity editor。带 + 号的铅笔允许添加新的碰撞区域，X 是移除它们，我们可以使用每个维度的上下箭头来调整它们：

一旦你对碰撞区域满意，点击 Collision 部分旁边的卷轴图标，点击 Copy 按钮，并将数据粘贴到你的 JSON 文件中（再次记住，碰撞和目标区域不放 在 ghost JSON 文件中）。

区域碰撞类型

除了碰撞类型外，还有区域碰撞类型，但在大多数情况下模组制作者不需要关心它们。

  • "none" -- 这用于像墙饰（挂毯、绘画...）、地毯之类的物品和门。它们需要一个碰撞区域，以便在建造者中我们有一个可以拖动来移动物品的框（并且建造者知道它们应该按什么顺序建造），但同时我们不希望 Hearthlings 与它们碰撞（因为区域是完整方块大小，而这些是薄物品）。

    对于门，它有助于防止敌人进入房屋，同时仍允许 Hearthlings 使用它们（尽管这也涉及 player_id）。

  • "solid" -- 大多数物品都有此类型。当我们不定义 "region_collision_type" 时，默认为 solid。

  • "platform" -- 这用于湿石、干石，更重要的是，脚手架。使用此类型，Hearthlings 可以站在脚手架顶部，也能够穿过其侧面。

来自 throw_rug.json 的示例：

  "region_collision_shape": {
     "region_collision_type": "none",
     "region": [
        {
           "min": { "x": -2, "y": 0, "z": -2 },
           "max": { "x": 1, "y": 0.1, "z": 1 }
        }
     ]
  }

目标区域

如果 Hearthlings 通过走到对象旁边来与我们的对象交互，我们需要定义一个 destination 组件来声明站在对象旁边意味着什么。

Destination 组件确定对象周围在与对象交互时可以接受的站立距离区域。在 JSON 文件中声明它时，将对象周围的区域作为具有起始和结束对角坐标的矩形传递。结束坐标是排他的。

默认情况下，即使我们没有指定任何相邻标志，游戏也会生成 4 个相邻区域作为指定目标区域的边缘。这通常意味着我们希望目标区域是整个对象周围的盒子（很多时候它与碰撞区域相同）。我们很少想站在对象的斜角，因为这通常离模型不够近或看起来不好。当没有目标时，游戏只是假设它与碰撞相同。

与碰撞区域一样，如果我们首先添加

  "destination": {}

在根实体 JSON 文件的组件中（不是 ghost！），我们可以在游戏中可视化地添加、移除和调整目标区域。

现在将你的物品放置在世界上，选择它并打开 Entity editor。添加一个目标区域并确保也选中 "Auto update Adjacency" 复选框。现在你可以点击相邻标志的任何方块。

你应该看到一个带有紫色边缘的透明框（目标区域），以及与该区域相邻的带有红色边缘的透明框。这些红框定义了 Hearthling 可以站在哪些网格方块上被视为"靠近"对象。目标是让红框包含对象边缘周围的网格（它们必须对齐，否则可能会发生不好的事情！）。

Hearthlings 将走到相邻区域中的任何方块开始与对象交互。如果我们的目标区域很高，相邻区域也将很高（但通常它只有一块厚，因为 Hearthlings 只需要 1 个方块站在地面上）。

通过调整目标区域和相邻标志，我们可以使 Hearthlings 仅从对象的一侧交互。例如，陶工仅从前面使用窑炉，只有当较长侧旁边的方块未被其他物品的碰撞框占据时才能到达床。

可能的相邻标志有：

  "adjacency_flags": [
     "front",
     "left",
     "back",
     "right",
     "front_left",
     "front_right",
     "back_left",
     "back_right"
  ]

来自 clay_bed.json 的 destination 组件示例：

  "destination": {
     "region": [
        {
           "min": {
              "x": -1,
              "y": 0,
              "z": -1
           },
           "max": {
              "x": 2,
              "y": 1,
              "z": 2
           }
        }
     ],
     "adjacency_flags": [
        "left",
        "right"
     ]
  }

与原点和碰撞区域一样，点击 Entity Editor 中 Destination 旁边的 "Show Json" 按钮以获取目标和碰撞信息。将该文本粘贴到实际 JSON 文件的组件部分（记住，不在 ghost JSON 文件中！那会导致错误）。

始终确保你的区域在对象旋转时有效！还要测试以确保 Hearthlings 可以走到你的对象旁边，并且他们的动画在旁边看起来正确。
