Mixintos 和 Overrides

使用 Mixintos 和 Overrides，你将能够修改 Stonehearth 模组（或任何模组）中的内容，而无需篡改模组的内部文件。Mixintos 和 Overrides 都在你的模组清单中指定。

记得仅在需要时添加逗号，否则游戏将无法读取你的清单并且无法加载你的模组。查看这些其他提示以进行调试。

Mixinto

"Mixinto" 允许将内容推送到来自不同模组的现有文件中。外部模组（如你的模组！）指定要将什么内容推送到目标数据文件中。Mixintos 仅适用于 JSON 文件，我们无法使用它们添加或修改 Lua/HTML/Javascript/其他类型的文件。

让我们看看 rayyas_children 模组的清单，它使用 mixintos 将内容注入到 stonehearth 模组中。这是其清单中的 "mixintos" 节点（位于与 "info" 相同的级别）：

  "mixintos": {
     "stonehearth:playable_kingdom_index": "file(data/rc_playable_kingdom_index.json)",
     "stonehearth:data:gm_index": "rayyas_children:data:gm_index",
     "stonehearth:farmer:initial_crops": "file(data/rc_initial_crops.json)",
     "stonehearth:building_templates:index": "file(data/saved_objects/building_templates_index.json)",
     "stonehearth:data:sound_constants": "rayyas_children:data:sound_constants",
     "stonehearth:gm:campaigns:arcs:ambient_threats": "file(data/mixintos/encounters/rc_ambient_threats_arc.json)"
  }

正如你所见，左侧的键是对 stonehearth 模组中资源的引用，而右侧的值指的是 rayyas_children 模组中包含我们要混入到这些资源中的数据的文件。在 "mixintos" 节点中，我们可以对键和值使用别名或路径。

对于 mixintos，我们总是需要用其他模组的命名空间加上冒号作为左侧键的前缀，在这种情况下是 "stonehearth:"。如果是路径，它应该是 "/<other_mod_namespace>/..." 的形式（在上面的示例中，我们可以使用路径 "/stonehearth/data/gm/gm_index.json" 而不是别名 "stonehearth:data:gm_index"）。

这样游戏就能准确知道我们要修改哪个模组的哪个文件，因为在不同模组中很容易有同名的文件。

如果你在 "mixintos" 节点中对值（右侧）使用别名，请确保它们也始终以你的模组命名空间为前缀。如果你使用路径，它们可以是相对或"绝对"路径。

我们的 mixintos 不应该是其他模组原始文件的精确副本，它们应该只包含我们要添加或修改的部分。例如，这是 rc_playable_kingdom_index.json mixinto 的样子：

  {
     "kingdoms": {
        "rayyas_children" : "rayyas_children:kingdoms:rayyas_children"
     }
  }

这是 stonehearth 模组中原始的 playable_kingdom_index.json：

  {
     "type": "index",
     "kingdoms": {
        "the_ascendancy": "stonehearth:kingdoms:ascendancy"
     }
  }

在 mixinto 中，我们需要指定父节点（在这种情况下是 "kingdoms"），以便游戏确切知道在 JSON 节点的哪个级别注入我们的内容。应用来自 rayyas_children 的 mixinto 后，stonehearth 的 playable_kingdom_index 将被游戏读取为以下内容，有效地在可用选择列表中添加了一个新王国：

  {
     "type": "index",
     "kingdoms": {
        "the_ascendancy": "stonehearth:kingdoms:ascendancy",
        "rayyas_children" : "rayyas_children:kingdoms:rayyas_children"
     }
  }

在大多数情况下，我们将为我们的新实体创建别名，然后通过 mixinto 将它们注入到 stonehearth 模组的相应文件中。示例包括添加新的制作配方、新派系、新生物群系、新特性、新命令等。

使用 Mixintos 修改内容

我们可以使用 mixintos 更改现有内容，如果我们要更改的值不是对象或数组。例如，我们可以使用这样的 mixinto 更改现有实体的缩放比例：

  {
     "components": {
        "render_info": {
           "scale": 0.8
        }
     }
  }

在这种情况下，文件合并后的最终值将是我们 mixinto 中的值。

Override

Mixintos 用于修改或增强模组中的内容。Override 将完全覆盖该资源。这是在清单中添加 override 的语法（"overrides" 节点位于与 "info" 相同的级别）：

  "overrides": {
     "/stonehearth/entities/critters/poyo/poyo.qb": "file(sandy_poyo/sandy_poyo_tan.qb)"
  }

就像 mixintos 一样，我们指定一个别名（前面加上我们要修改的模组命名空间和冒号）或"绝对"路径（如果例如特定资源没有别名），然后我们告诉游戏要用我们模组的哪个文件覆盖它，可以使用别名（前面加上我们的模组命名空间和冒号），或使用路径（相对或"绝对"）。

虽然 mixintos 允许多个模组向同一文件添加内容，但 overrides 会完全替换该文件。这意味着如果两个模组覆盖同一个文件，只有一个会胜出，这可能导致兼容性问题。因此，不建议对代码文件使用 override，但在某些情况下它对交换模型或图像很有用。

例如，上面的 override（属于延迟清单）在我们选择在沙漠生物群系中游玩时应用，并将 poyos 的 .qb 模型更改为更适合沙漠主题。

使用 Overrides 注入内容

我们也可以使用 overrides 添加某些类型的内容。当我们需要向 Hearthlings 添加动画时，这很有用，因为这些效果列表不是索引，只是指向 stonehearth 模组内文件夹的路径，所以没有我们可以 mixinto 的文件。

为了做到这一点，我们可以指定一个在 stonehearth 模组中不存在的路径，指向我们模组中的文件。动画在效果文件中指定，所以我们会有一个引用我们模组文件夹中动画的效果文件。这在清单的 overrides 节点中看起来是这样的：

  "overrides" : {
     "stonehearth/data/rigs/entities/humans/effects/my_effect.json" : "/my_mod_namespace/my_effect.json",
     "stonehearth/data/rigs/entities/humans/animations/female/my_anim.json": "/my_mod_namespace/animations/female/my_anim.json",
     "stonehearth/data/rigs/entities/humans/animations/male/my_anim.json": "/my_mod_namespace/animations/male/my_anim.json"
  }

当然，如果你愿意，你可以在模组中使用相同的路径，以便你的文件组织方式与 stonehearth 模组相同。

在这种情况下，"my_effect.json" 在 stonehearth 模组中不存在，只存在于我们的模组中，但通过在清单中这样指定，游戏会认为它确实有这个不存在的文件，并用我们的内容覆盖它。这个效果/动画何时播放是另一个问题。

注意这些 overrides 中的路径（指向 stonehearth 模组中不存在的文件的路径）不以正斜杠开头。对于这种类型的 overrides，必须以这种方式声明，否则它们将无法工作。
清单和其他文件中的其他完整路径必须以正斜杠开头，除非另有说明。其中一些无论有没有斜杠都能正常工作，但为了保持一致性，我们将在示例中使用相同的语法。

模组兼容性

你可以猜到，当多个模组 mixinto 同一个文件时，所有新内容都会被读取。如果 mixintos 更改特定值，情况将与 overrides 相同，只有一个模组会胜出，所以要注意你对文件的更改，以通知用户可能与触及与你相同文件的模组存在不兼容性。
