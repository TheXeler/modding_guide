你的第一个模组

要开始制作你的模组，请在 Stonehearth/mods 目录中创建一个文件夹。你可以随意命名文件夹，但它只能包含字母数字字符、连字符和下划线。我们也倾向于仅使用小写字母作为文件夹名称。

如果你打算将模组上传到 Steam 创意工坊，你应该通过模组菜单创建你的模组文件夹。或者，你可以在 Stonehearth/mods 文件夹中开发你的模组，当它准备好后，通过模组菜单生成模组文件夹，复制粘贴你的模组内容，并编辑自动生成的清单文件内容以包含你的添加内容。

从那时起，你应该继续在自动生成的模组文件夹中开发你的模组，或者禁用该模组并在 mods 文件夹中开发那个模组，每当需要将更新推送到创意工坊时重复上述步骤。本指南中的步骤将引用 mods 文件夹（本地模组），但如果你愿意，你可以在为创意工坊自动生成的文件夹中开发你的模组。

在你的模组文件夹中创建一个文本文件并将其命名为 manifest.json（确保扩展名是 .json 而不是 .json.txt 或类似格式。启用显示文件扩展名以便你可以看到文件的实际格式）。 现在，用你选择的文本编辑器打开它并复制/粘贴以下数据： 

  {
     "info": {
        "name": "My First Mod",
        "namespace": "my_first_mod",
        "version": 3
     }
  }

在这里，你可以将名称更改为任何你想要的名称（例如："Candy Land"）。你必须将版本号保持为 3，如下文所述。 至于命名空间，它只能包含字母数字字符和下划线。通常命名空间与文件夹名称匹配。

就是这样。你的第一个模组完成了。运行 Stonehearth，打开模组菜单，你应该会看到你的模组按名称列出，旁边用小字显示其命名空间。这个模组目前还不做任何事情，但首先我们需要学习清单文件的结构以及如何处理文件。

什么是清单文件？

清单文件是模组中所有资源和代码的列表。

清单文件中最低要求的字段位于名为 "info" 的部分中：

  1. 模组的命名空间。
    这将用于区分你的模组内容的标识符与其他模组内容的标识符。你将在文件中使用它，所以尽量保持简短且唯一。它只能包含字母数字字符和下划线，不允许使用空格或其他字符。

  2. 版本。
    "info" 部分的 "version" 键指的是游戏本身的模组 API 版本，而不是你的模组版本。它用于在 API 版本更改时自动使旧模组过时。要使你的模组与当前版本的游戏兼容，请始终确保它具有与当前 stonehearth 模组中相同的值，现在是 3。

  3. 模组的名称。
    你的模组的名称或标题。只要你在别名中使用命名空间，你可以对名称使用任何字符。

可选字段

在清单文件的 "info" 部分中还可以有其他可选键，它们会影响模组在模组菜单中的显示方式：

  "required" : true

这仅用于运行游戏所需的主要模组（radiant、stonehearth）。它在模组选项卡中将它们显示为已勾选且禁用状态，以便玩家不会意外取消勾选它们。其他默认模组也有 "is_base_mod" 键，以在模组菜单中高亮显示它们。这两个键允许区分基础游戏内容和外部模组，因此你不应该使用它们。

  "client_only": true

这是一个标签，用于防止在玩家加入多人游戏时比较该模组。游戏不会尝试同步仅限客户端的模组（如本地化模组）。

  "is_debug_mod": true

这将把模组标记为用于调试的模组（例如，游戏附带的 "debugtools" 模组）。它将在模组菜单中标记为"调试模组"。

  "default_enabled": false

如果你希望你的模组默认禁用，请在 "info" 部分中使用此标签。

  "author" : "Your name"

这是一个可选字段。它在游戏中不会产生任何效果，但你可以在此处包含你的姓名或昵称以记住谁制作了这个模组。

JSON 允许编写任何你想要的键/值对，因此只要游戏不将其视为特殊关键字，你就可以在此处添加其他键。

清单文件的结构

在 stonehearth 的清单文件中，你可以看到 "info" 下面的其他部分。它们用于引用游戏中的所有资源。现在我们将简要介绍它们各自的作用，并在指南的特定部分中引用如何使用它们的示例。

别名（Aliases）

别名是引用游戏对象的昵称。我们可以声明一个别名并将其与文件路径关联，以获取描述该对象的数据文件。稍后，在我们模组的文件中，我们可以通过使用其别名（始终以我们模组的命名空间加冒号为前缀）来引用该游戏对象，这比必须写入整个文件路径要短得多。让我们看一个来自 stonehearth 清单文件的简短示例：

  "aliases": {
     "medium_oak_tree" : "file(entities/trees/oak_tree/medium_oak_tree)"
  }

你可能已经注意到路径看起来有点奇怪。有几种方法可以指定路径。这些都是等效的：

 "medium_oak_tree" : "file(entities/trees/oak_tree/medium_oak_tree)"
 "medium_oak_tree" : "file(entities/trees/oak_tree/medium_oak_tree/medium_oak_tree.json)"
 "medium_oak_tree" : "/stonehearth/entities/trees/oak_tree/medium_oak_tree/medium_oak_tree.json"

第一个语法是第二个语法的简写形式，但仅适用于在与该文件同名的文件夹中的 JSON 文件。如果文件是 Lua 文件（或 .png、.qb 等），你将必须指定整个路径。

用于包装路径的 "file()" 将从当前文件夹开始搜索。了解这一点很重要，因为我们可以使用此语法从其他文件中引用文件，但这些文件的路径将根据我们所在的文件夹而不同。稍后会有更多示例。

并非我们模组中的所有文件都需要有别名，但有时为某些文件（不仅仅是实体）拥有额外的别名会很方便。

别名（键）应仅包含字母、数字、下划线和冒号。你可以使用大写和小写，但 stonehearth 的约定是仅使用小写并用冒号分隔部分（用于组织），然后使用下划线分隔单词（你也可以使用连字符，但约定是使用下划线）。混合使用大写和小写会造成混淆，并且你有拼写错误的风险，因此默认情况下我们仅对小写的 JSON 键使用小写。也不建议使用空格。

有关模组的完整示例，请参阅入门模组部分。

Mixintos 和覆盖（Overrides）

这些部分不会出现在 stonehearth 清单文件中，但它们是模组制作的基础。它们在指南的下一部分中进行介绍。

函数（Functions）

"functions" 部分中列出的键是函数的别名。它们都列出了具有函数定义的控制器的路径，以及一个端点，该端点指的是客户端或服务器，具体取决于函数 intended 在哪里运行。

控制器（Controllers）

控制器是一个包含实用程序类的 Lua 文件，该类存储数据和函数（并且不是组件、服务、调用处理器等）。它具有一些使初始化/加载/销毁更容易的行为。在 stonehearth 清单文件中，你可以找到不同职业、特性、场景、库存跟踪器等的控制器的别名。

组件（Components）

组件是一段可重用的代码块，描述实体的相关细节集。组件包括 ai/actions、实体的渲染装备、携带的物品、统计数据等。如果任何类型的游戏逻辑与实体相关联，则可能有附加到该实体的组件涵盖该游戏逻辑。如果你创建新组件，则需要将其添加到清单文件的此部分中。

用户界面（UI）

清单文件的此部分声明定义不同 UI 视图的文件。它包含所有 HTML 文件、Javascript (.js) 文件和 Less 文件（CSS 的变体）列表的键。

区域设置（Locales）

  "default_locale":"en"

此键允许进行本地化（有关如何本地化你的模组的更多信息，请参阅指南的特定部分）。

依赖关系（Dependencies）

我们还可以指定我们的模组应该在其他模组之后加载。当我们想要添加与其他模组的兼容性时，这很有用。 在与 "info"、"default_locale" 等相同的级别，我们可以添加此数组，列出我们依赖的模组的命名空间： 

  "dependencies" : [ "mod_a", "mod_b"]

这样，如果我们的模组覆盖了与 "mod_a" 或 "mod_b" 相同的文件，我们的覆盖将获胜，因为它们将最后加载。

这不会强制依赖关系，也不会在玩家缺少依赖关系时警告他们。模组制作者应该相互讨论以决定哪个模组应该添加依赖关系，并告诉玩家安装他们需要的模组。

基础模组将首先加载。在多人游戏中，仅限客户端的模组将单独排序，因为主机可能没有它们。"dependencies" 字段不会影响热加载的清单文件。

如果游戏检测到循环依赖（如 mod_a 依赖于 mod_b，mod_b 依赖于 mod_c，mod_c 依赖于 mod_a），这些模组将被临时禁用（它不会影响 user_settings 禁用设置，也不会更改模组菜单中的复选框）。如果依赖关系可能导致游戏中的错误，它们将被禁用。模组加载后会出现一个弹出窗口，告知哪些模组具有循环依赖关系并将因此被禁用。

同样，如果我们有一个具有依赖关系的模组，但所依赖的模组未安装或已禁用，游戏将忽略该依赖关系并像没有依赖关系一样加载该模组。

已弃用的别名

已弃用的别名用于开发者想要替换而玩家已经拥有资产版本的资产。如果他们只是删除资产，那么他们会破坏保存/加载兼容性——或者可能会加载，但事物会损坏。因此，通过将原始 URI 移动到此弃用列表，原始资产继续存在但不再在新游戏中填充。例如：'red_fox' 最初使用兔子的装备，但狐狸的移动方式并不像兔子。他们决定将狐狸切换到狼的装备，但当他们第一次这样做时，所有预先存在的存档都显示了严重损坏的狐狸生物。因此，他们改为弃用该资产并制作了一个新的狐狸，问题解决了。另一个例子是他们更新了许多木匠的资产，但这意味着许多形状和颜色的变化，人们可能不希望立即在家中被替换。

模组制作者应该在他们的模组已经向公众发布但想要对艺术资产进行足够剧烈的更改以至于存档文件的旧数据会导致问题，或者视觉效果明显不同到足以保证不强迫某人立即拥有新视觉效果时使用此功能。

其他部分

清单文件中还可以有其他部分，如 "game"、"client_init_script"/"server_init_script"、"component_renderers"、"invariant_renderers"、"client_controllers"、"client_components" 等，但我们不会在此解释它们。它们将在指南的其他部分中提到。
