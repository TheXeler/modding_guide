生命周期函数

组件和控制器都是类，并共享类似的属性和函数。

可以在每个类上实现的特殊函数有：

  • __init() -- 这是类的默认 Lua 构造函数。这将首先被调用，总是如此。在此构造函数期间，不会有特殊变量或任何可用的内容。建议不要包含它，而是将初始化信息放在特殊的 initialize() 函数中。

  • initialize() -- 此函数总是在 __init 之后调用。在 Lua 组件中，特殊的 self._entity 字段将被设置，您可以通过调用 radiant.entities.get_json(self) 获取组件的 JSON 数据。您不能在 initialize() 中访问服务或公共字段（如 stonehearth.constants），因为在恢复期间，它们可能尚不存在。

    重要提示：您必须在 initialize() 函数中声明所有保存变量字段，包括 nil 变量。您可以在 initialize 中询问实体是否有其他组件，但请注意，那些其他组件可能尚未调用它们的 initialize()。

  • create() -- 这在组件首次创建时调用一次，无论是在实体创建时还是动态添加时。对于控制器，create 函数可以接受从控制器创建者传入的参数。

    对于控制器，create() 在 initialize() 之后立即调用。对于通过 JSON 指定的组件，create() 在实体上的所有其他组件都已 initialized 之后调用。

  • restore() -- 这在从存档游戏加载组件/控制器时调用。它在 initialize() 和 fixup_post_load() 之后调用。

  • activate() -- 这每次都会被调用。它在 create() / restore() 之后调用。对于创建时的控制器，这在 create() 之后立即调用。对于创建时的组件，这在所有组件都已调用 create() 之后调用。对于加载时的所有内容，这在所有组件/控制器都已调用 restore() 之后调用。

  • post_activate() -- 这在所有组件/控制器都已调用 activate() 之后调用。在刚刚创建的实体上，它将使实体 JSON 中指定的所有组件在调用其 post_activate() 之前先被 activated。在恢复时，世界中的每个组件/控制器都将先被调用 activate()，然后世界中的每个组件/控制器才会被调用 post_activate()。

  • destroy() -- 这在实体被销毁时调用。在这里，我们通常添加代码来销毁监听器和任何其他需要安全清理的变量。

总结一下，这些函数的调用顺序是：

  __init() -> initialize() -> {create(optional_controller_variables) OR restore()} -> activate() -> post_activate()

示例

组件中生命周期函数的示例：

  local MyComponent = class()

  function MyComponent:initialize()
     local json = radiant.entities.get_json(self)
     self._sv.first_saved_variable = nil
     self._sv.second_saved_variable = 'test string'
     self._sv._unremoted_save_variable = json.my_json_data_variable
     self._sv._created_linked_entity = nil

     -- 我们还不知道这是加载还是新创建，
     -- 所以 self._sv 将具有在 initialize 中设置的值。
     -- 如果这是加载，self._sv 变量将在 restore 函数中具有它们的加载值
  end

  function MyComponent:create()
     self._sv._created_linked_entity = radiant.entities.create_entity('my_mod:sub_entity')
  end

  function MyComponent:restore()
     -- 如果这是从存档游戏加载的，self._sv 中的变量
     -- 现在将具有它们的保存值
  end

  function MyComponent:activate()
     self._log = radiant.log.create_logger(self._sv.second_saved_variable)
  end

  function MyComponent:post_activate()
     -- 可以在此处访问其他组件并让它们执行操作
     local component2 = self._entity:get_component('stonehearth:my_component_2)
     component2:some_setter_function(self._sv._unremoted_save_variable)
  end

  return MyComponent

控制器和服务也可以使用这些函数。

关于保存变量

saved_variables（实际上是字段 __saved_variables）是一个存在于 Stonehearth Lua 代码中每个控制器、组件和服务上的对象。

它实际上是一个 C++ 数据存储，有两个用途：作为存储/加载持久数据的地方，以及作为数据从服务器传输到客户端的手段。

一般来说，您唯一需要直接访问它的时候是当您在服务器上更改了一些数据，现在您需要客户端（或 UI）知道您已进行了更改。在这种情况下，您调用 self.__saved_variables.mark_changed()，更改的数据将被序列化并通过网络传输到客户端，并可能触发您在那里安装的任何事件处理程序。

数据存储的字段在传输到客户端之前会被"diffed"，因此如果您只更改一个字段，您只需支付该字段的传输费用。

要将要保存的字段添加到控制器、组件或服务，只需在您的 initialize Lua 例程中将其添加到 ._sv 字段。作为一个说明性示例，考虑 AI 组件：

  function AIComponent:initialize()
     self._log = radiant.log.create_logger('ai.component')
                                :set_entity(self._entity)

     self._sv.spin_count = 0
     self._sv.status_text_key = nil
     self._sv.status_text_data = nil
     self._num_unyielded_spins = 0
  end

在这里，我们看到添加了一些字段，一些添加到 self，其他添加到 ._sv。直接在 self 上的字段将不会被保存，也不会被传输到客户端。它们实际上是服务器组件私有的。添加到 ._sv 的字段将被保存，并传输到客户端。

另一个重要功能是，如果您命名一个以 underscore 开头的 ._sv 字段（self._sv._foo），虽然该字段将被保存，但它不会被传输到客户端。当您有重要的可持久化数据但在网络上可能非常昂贵时，这非常有用。

维护存档兼容性

默认情况下，_sv 有一个版本，指定保存的数据存储的版本。版本值默认为 0。如果您需要更新控制器到不同版本，因为您重命名了变量、添加了需要在 create() 中设置的变量，或者需要修复旧版本控制器的任何其他情况，您可以在控制器/组件上实现 version 函数，以告诉持久化系统代码版本已更改。

首先向控制器/组件添加一个 versions 变量，以便您可以记录每个版本添加的内容。来自 job 组件的示例：

  local VERSIONS = {
     ZERO = 0,
     CLEAN_UP_SAVED_FIELDS = 1,
     JOB_CONTROLLER_JSON_PATHS = 2,
     LEVEL_0_REMOVED = 3
  }

然后实现 get_version 函数并使其指向最新版本：

  function JobComponent:get_version()
     return VERSIONS.LEVEL_0_REMOVED
  end

最后实现 fixup_post_load 函数。此函数仅在加载时调用，如果保存的版本与当前版本不匹配，在 initialize 之后和 restore 之前调用（_sv 中的变量已经被填充）。然后您可以修改保存数据中的任何内容。

例如，在 CLEAN_UP_SAVED_FIELDS 版本中，我们将 _sv.equipment 变量重命名为 _sv._job_equipment。请注意，如果需要，我们可以从 old_save_data 参数中检索保存的数据：

  function JobComponent:fixup_post_load(old_save_data)
     if old_save_data.version < VERSIONS.CLEAN_UP_SAVED_FIELDS then
        self._sv._job_equipment = old_save_data.equipment
     end

     if old_save_data.version < VERSIONS.JOB_CONTROLLER_JSON_PATHS then
        self._fix_up_job_controller_json_paths = true
     end

     if old_save_data.version < VERSIONS.LEVEL_0_REMOVED then
        self._needs_level_0_update = true
     end
  end

在后续版本中，我们设置了一些标志，以便我们可以在 restore/activate/post_activate 中使用它们来适当修补控制器，使玩家可以使用新版 mod 继续玩他们的旧存档文件。

关于观察者

观察者监视实体上的变量，并在该变量更改时执行操作。它有点像组件，但它是被动的而不是主动的。例如，参见 stonehearth\ai\observers 中的睡眠观察者，它监视困倦分数的降低，然后触发一系列动作。

如果您希望控制器持有公开访问的状态，您通常需要组件。当您需要将行为注入到无法由动作封装的实体上时，观察者更有用。

一般来说，您可以使用观察者做的事情，也可以使用组件完成。组件比观察者更受欢迎，因为这是我们在整个代码库中使用的，并且它们的行为更可预测。

关于验证器

在 radiant/modules/validator.lua 中，我们可以找到验证 Lua 代码中参数的函数。

您会在代码中的许多地方看到我们在文件顶部获取验证器：

  local validator = radiant.validator

然后在函数的开头，调用其函数之一来验证传递的参数：

  function CatalogCallHandler:get_catalog_data(session, response, uri)
     validator.expect_argument_types({'string'}, uri)
     -- ...
  end

这对于防止多人游戏中的不良数据很有用。

在 stonehearth/ui/common/validator/validator.js 中也有一些用于 UI 的验证函数：

  var townName = App.stonehearth.validator.enforceStringLength(self.$('#name'), 28);

在 Lua 中引入文件

如果我们需要在 Lua 脚本中引入文件以使用其中的函数或常量，我们会在脚本顶部、开始我们的代码之前执行此操作：

  local constants = require 'constants'

但是，在从您的 mod 引入 stonehearth 文件时，请确保在其前面加上 mods.stonehearth：

  local constants = require 'mods.stonehearth.constants'

否则游戏将不知道在哪里找到该文件。通常路径是用点分隔的目录名称。

对于引入 C++ 内容，我们省略 require 和引号：

  local Point2 = _radiant.csg.Point2

我们还可以为服务或其他内容创建快捷方式：

  local calendar = stonehearth.calendar
