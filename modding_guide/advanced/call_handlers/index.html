什么是调用处理器？

游戏中的大多数操作都是通过炉民 AI 或通过操作对象上的值来完成的，这会导致一系列导致游戏状态变化的动作。然而，有时游戏中的操作不属于单个炉民或对象，这些操作由 farming 或 town 等服务管理。

调用处理器帮助 UI 调用这些高级服务之一的函数，触发一系列动作。有关示例，请参阅 farming 或 unit control 服务。

Lua 客户端和服务器概述

Lua 在两个地方运行——在 Lua 服务器上，大部分游戏在此处理和运行，以及在较小的 Lua 客户端中，它以用户可交互的方式处理将实体渲染到屏幕。

例如，当用户选择一个要放入世界的对象时，附加到用户鼠标的临时物品在 Lua 客户端中渲染，并且在数据本身（对象将被放置的位置、旋转角度等）被发送到服务器进行处理之前，点击将其放下由该客户端捕获。存在于用户放下对象和工人实际实例化对象之间的 ghost 物品也仅存在于 Lua 客户端上；因此，世界中的人看不到它也无法与之交互。

在某些方面，您可以将 Lua 客户端渲染器视为玩家可以看到并与之交互以玩游戏的所有事物的领域，但这些事物实际上并不存在于世界模拟中。

作为一般规则，您可以假设 Lua 客户端具有与服务器相同的所有数据，但它是只读的：不能创建新组件，也不能将数据保存到现有数据结构中。某些东西，如效果，根本不能在客户端创建。

以下是使用调用处理器在客户端和服务器之间通信的一些示例：

示例 1：通过用户交互的客户端渲染器

假设您想由于用户与 UI 的某些交互而在 lua 客户端中绘制某些内容（例如，用户已点击他们想要将新物品放入世界）。

首先，在 mod 清单的 "functions" 部分（与"aliases"或"mixintos"同级）添加一个带有"client"端点的调用处理器：

  "functions": {
     "choose_place_item_location": {
        "controller": "file(call_handlers/place_item_call_handler.lua)",
        "endpoint": "client"
     }
  }

这意味着如果用户从 UI 调用 'stonehearth:choose_place_item_location' 地址，将调用 call_handlers/place_item_call_handler.lua 中的 choose_place_item_location 函数。并且此函数将存在于 Lua 客户端的上下文中，而不是 Lua 服务器中。

UI 调用可能如下所示：

  radiant.call('stonehearth:choose_place_item_location', data_function_is_expecting)
     .done(function(o){
      // 做一些事情
  });

而 place_item_call_handler.lua 内部的函数声明本身可能如下所示：

  function PlaceItemCallHandler:choose_place_item_location(session, response, item_uri)
     self._cursor_entity = radiant.entities.create_entity(item_uri)
     local re = _radiant.client.create_render_entity(RenderRootNode, self._cursor_entity)
     -- ...计算物品应该出现的位置...
     self._cursor_entity:add_component('mob'):set_location_grid_aligned(pt)
  end

这段代码将在渲染器中创建一个实体并将其放置在关联点的世界中。稍后，可以使用以下命令销毁实体：

  _radiant.client.destroy_authoring_entity(self._cursor_entity:get_id())

有关捕获鼠标和鼠标点击的更多信息，请参见 move_unit_call_handler.lua。

有关实体的客户端渲染，请参见组件渲染器。

示例 2：区域

区域是玩家（或 DM） earmarked 用于特定目的的游戏区域。区域可能仅出现在区域视图中，或者它们也可能具有游戏内外观。例如：农场、库存区、陷阱区等。

区域由玩家从 UI 创建。通常，一旦它们被创建，它们的关联组件就会发出任务，指示任务组中的任何可用人员照料该区域（或其中的任何内容）。

向 UI 添加区域

这些是通过 mod 添加牧场区域的说明。

开始菜单在 stonehearth/data/ui/start_menu.json 中定义。首先，我们需要创建一个 mixinto 以向区域部分添加新条目（为了保持一致性，这是最合适的位置）：

  "mixintos" : {
     "/stonehearth/data/ui/start_menu.json" : "file(data/ui/start_menu.json)"
  }

我们的 mixinto 看起来像这样：

  {
     "zone_menu": {
        "items": {
           "create_pasture": {
              "name": "i18n(stonehearth:ui.game.menu.zone_menu.items.create_pasture.display_name)",
              "description": "i18n(stonehearth:ui.game.menu.zone_menu.items.create_pasture.description)",
              "required_job": "stonehearth:jobs:shepherd",
              "required_job_text": "i18n(stonehearth:ui.game.menu.zone_menu.items.create_pasture.required_job_text)",
              "class": "button",
              "icon": "/stonehearth/ui/game/start_menu/images/zone_pasture.png",
              "sticky": true,
              "hotkey_action": "zone:pasture"
           }
        }
     }
  }

请注意，我们可以 mixinto start_menu.json 文件以向开始菜单添加任何新选项。有一些可选字段，如 "required_job" 和 "required_job_text"，它们仅用于区域，以在玩家将某人晋升为该职业之前禁用它们。对于制作菜单，它们会立即生效，但对于其余选项来说很棘手，因为它涉及 UI 文件。

条目的名称（上面示例中的 create_pasture）将成为选择该项目时调用的操作（或者，某些菜单有一个 "menu_action" 字段，其中包含要调用的操作）。这些操作的处理程序在 stonehearth/ui/game/start_menu/start_menu.js 中：

  menuActions: {
     // ...
     create_pasture : function () {
        App.stonehearthClient.createPasture();
     },
     // ...

在这种情况下，create_pasture 函数调用在 stonehearth/ui/root/js/stonehearth/stonehearth_client.js 中定义的另一个函数：

  createPasture: function() {
     var self = this;

     App.setGameMode('zones');
     var tip = self.showTip('stonehearth:ui.game.menu.zone_menu.items.create_pasture.tip_title',
           'stonehearth:ui.game.menu.zone_menu.items.create_pasture.tip_description', { i18n: true });

     // 我们调用一个客户端操作，允许玩家绘制区域
     return this._callTool('createPasture', function() {
        // 我们调用一个客户端函数来创建区域并返回它
        return radiant.call('stonehearth:choose_pasture_location')
           .done(function(response) {
              radiant.call('radiant:play_sound', {'track' : 'stonehearth:sounds:place_structure'} );
              // 然后 UI 将选择它
              radiant.call('stonehearth:select_entity', response.pasture);
              // 再次调用工具，以防玩家想要继续绘制牧场
              self.createPasture();
           })
           .fail(function(response) {
              self.hideTip(tip);
           });
     });
  },

现在您可能想知道，如果函数是从 start_menu.js 读取的，我们如何从我们的 mod 中为我们的新菜单添加自定义函数？请参阅此页面以解决您的疑问。

在客户端指定区域

UI 调用了一个客户端 Lua 函数来绘制区域。此函数在清单的 "functions" 部分中声明：

  "functions": {
     "choose_pasture_location": {
        "controller" : "file(call_handlers/shepherd_call_handler.lua)",
        "endpoint" : "client"
     }
  }

调用处理器函数看起来像这样：

  function ShepherdCallHandler:choose_pasture_location(session, response)
     stonehearth.selection:select_designation_region(stonehearth.constants.xz_region_reasons.PASTURE)
        :set_min_size(10)
        :set_max_size(50)
        :require_unblocked(false)
        :use_designation_marquee(Color4(56, 80, 0, 255))
        :set_find_support_filter(stonehearth.selection.valid_terrain_blocks_only_xz_region_support_filter({
              grass = true,
              dirt = true
           }))
        :set_can_contain_entity_filter(function(entity)
              -- 避免其他指定。
              if radiant.entities.get_entity_data(entity, 'stonehearth:designation') then
                 return false
              end
              if entity:get_component('terrain') then
                 return false
              end
              return true
           end)
        :set_cursor('stonehearth:cursors:zone_pasture')
        :done(
           function(selector, box)
              local size = {
                 x = box.max.x - box.min.x,
                 z = box.max.z - box.min.z,
              }
              _radiant.call('stonehearth:create_pasture', box.min, size)
                 :done(
                    function(r)
                       response:resolve({ pasture = r.pasture })
                    end
                 )
                 :always(
                    function()
                       selector:destroy()
                    end
                 )
           end
        )
        :fail(
           function(selector)
              selector:destroy()
              response:reject('no region')
           end
        )
        :go()
  end

请注意，它在客户端（客户端端点）上运行，并且只能访问客户端可用的 Lua 项目。

该函数允许玩家在世界中点击并指定一个区域。它将根据您传入的 Color4(r, g, b, a) 绘制区域。

一旦玩家释放鼠标，此函数将调用服务器函数以实际创建区域。由于我们在客户端和服务器之间通信，它也必须在调用处理器中定义。

请注意，对于这些示例，我们使用的是 stonehearth 命名空间，您在代码中编写时将使用您的 mod 命名空间。如果您正在制作自定义类型的区域，请确保更改任何应指向您的 mod 的引用。

创建区域

创建区域的函数位于服务器上：

_radiant.call('stonehearth:create_pasture', box.min, size)。

它也在清单的 "functions" 下声明，但具有服务器端点：

  "create_pasture": {
     "controller" : "file(call_handlers/shepherd_call_handler.lua)",
     "endpoint" : "server"
  },

它看起来像这样：

  function ShepherdCallHandler:create_pasture(session, response, location, size)
     validator.expect_argument_types({'Point3', 'table'}, location, size)
     validator.expect.num.range(size.x, 10, 50)
     validator.expect.num.range(size.z, 10, 50)

     local entity = stonehearth.shepherd:create_new_pasture(session, location, size)
     return { pasture = entity }
  end

通常，实际区域由适当的服务创建（在 stonehearth/stonehearth_server.lua 中初始化）。由于它是一个区域，它将需要用于碰撞和大小的区域组件。在这种情况下，函数在 shepherd 服务中，看起来像这样：

  function ShepherdService:create_new_pasture(session, location, size)
     -- 一些清理：我们从客户端得到的不完全是 Point3
     location = Point3(location.x, location.y, location.z)
     local entity = radiant.entities.create_entity('stonehearth:shepherd:shepherd_pasture', { owner = session.player_id })
     self:_add_region_components(entity, size)

     local pasture_component = entity:get_component('stonehearth:shepherd_pasture')
     pasture_component:set_size(size.x, size.z)
     radiant.terrain.place_entity_at_exact_location(entity, location)

     return entity
  end

（您可以检查如何创建服务这里）。

您会注意到牧场有一个组件。此组件对于赋予区域功能是必要的。区域还需要一个渲染器才能在世界中正确显示。

为区域添加行为

首先，让我们看看区域的组件。这是一个非常标准的组件。它将重要数据保存到其 _sv 变量中。通常，它还会发出交给负责区域的任务组的任务。

区域的组件在 manifest.json 的 "components" 下声明：

  "shepherd_pasture" : "file(components/shepherd_pasture/shepherd_pasture_component.lua)"

渲染区域

如果有一个与组件同名的渲染器，则该渲染器将与该组件关联。在这种情况下，由于区域没有 Qubicle 模型，它依赖于渲染器来绘制自己。

在上面的示例中，组件名为 shepherd_pasture，因此我们可以在清单的component_renderer部分中声明一个看起来像这样的渲染器：

  "component_renderers" : {
     "shepherd_pasture" : "file(renderers/shepherd_pasture/shepherd_pasture_renderer.lua)"
  }

渲染器可以访问保存在组件的 _sv 变量中的任何数据。有关示例，请参见 stonehearth/renderers/shepherd_pasture/shepherd_pasture_renderer.lua。

选择区域

当 UI 从创建过程中获取区域时，它应该选择该区域，以便可以进行任何区域相关的配置。为此，请确保 UI 调用：

  radiant.call('stonehearth:select_entity', response.zone_name);

在 stonehearth/ui/game/modes/zones_mode/zones_mode.js 中有 _examineEntity 函数。您需要向其中添加另一个 if，以便在选择新区域时显示您的自定义 UI 视图。确保您的 UI 视图扩展了 StonehearthBaseZonesModeView，以便它具有区域的通用功能。

此外，请确保在 stonehearth/ui/game/modes/mode_manager.js 的 _getModeForEntity 函数中将区域类型注册为区域。

在编写本指南时，ACE mod 覆盖了 mode_manager.js 以使其更容易根据所选实体更改 UI 模式。如果您基于该代码，玩家需要安装并启用 ACE mod 才能玩您的 mod。
