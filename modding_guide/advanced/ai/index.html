AI 系统概述

Stonehearth 中的智能实体运行分层任务网络，通过制定、比较和执行由动作组成的计划来执行活动。通常会同时考虑许多潜在的计划，其中大多数尚未准备好执行或不是最优的，AI 会不断选择在任何给定时间执行的最佳计划。

让我们 unpack 这个正式定义并将其映射到实现中使用的术语：

  • 智能实体 -- 具有 stonehearth:ai 组件的实体。此组件运行一个 Lua 线程（由线程类管理的协程），它持续评估计划树并执行其中的一个分支。智能实体包括炉民、敌对生物、宠物、小动物，甚至炮塔，尽管本页面其余部分使用"角色"来指代这些实体中的任何一个。

  • 活动 -- 可以有多种实现的行为。例如，如果"获取食物"是一项活动，"从库存区获取食物"、"从背包获取食物"和"狩猎野生动物"都可能是"获取食物"活动的实现。活动表示为字符串，通常采用 mod_name:verb 的形式，例如 stonehearth:get_food。可以在 stonehearth/ai/activities.lua 中找到不一定最新的活动列表。活动可以接受参数。例如，stonehearth:find_path_to_entity 活动将要寻路的实体作为参数。核心来说，AI 组件所做的就是不断尝试执行 stonehearth:top 活动。

  • 动作 -- 活动的特定实现。在代码中，我们通过说动作"执行"给定的活动字符串来表达这一点。动作负责告诉 AI 系统 (1) 其"效用值"，即它有多好，(2) 如何"思考"它，即决定它是否准备好执行以及如果执行它期望改变哪些相关状态，(3) 以及如何"运行"它，即实际实时执行它。有几种类型的动作：

    • 叶子动作 -- 最简单的动作类型。一个具有执行动作逻辑的 run() 方法的 Lua 文件。例如 heal_entity_adjacent_action。
    • 复合动作 -- 由步骤列表组成的动作，每个步骤都是具有可选参数绑定的活动。执行复合动作涉及找到执行每个步骤活动的有效动作并按顺序执行所有这些动作。例如，heal_self_action 包括 (1) 获取治疗物品（一个 stonehearth:goto_item_made_of 活动，材料参数设置为 "healing_item"），(2) 保留该物品以便其他角色不会拿走它（一个 stonehearth:reserve_entity 活动，将上一步选择的物品作为参数传递），最后 (3) 使用该物品自我治疗（一个 stonehearth:heal_entity_adjacent 活动，将实体本身作为目标，将上一步保留的物品作为要应用的物品）。
    • 任务组动作 -- 运行声明的任务（活动）列表之一的动作。AI 控制之外的系统决定在给定时间哪些任务处于活动状态，任务组允许这些系统要求多个角色执行某些操作。这是不同活动相互竞争的唯一地方，因此任务组声明允许您重新映射它们的效用值或范围。例如，farming_task_group 涉及耕作（stonehearth:till_entire_field）、种植（stonehearth:plant_crop）或收获（stonehearth:harvest_field），并指定这三种活动在效用方面如何相互竞争（参见下面的"比较计划"）。
    • 任务动作 -- 运行任务组中每个任务实例的动作。这是一个实现细节，只有一个 TaskAction 类，因此基本上可以忽略它。

    角色行为通常由他们"知道"的动作集配置，通过"AI 包"指定，在本指南的后面部分中有更详细的描述。

  • 制定计划 -- 决定动作是否准备好运行。这在代码中称为"思考"。当 AI 组件想要执行活动时，它会询问所有实现该活动的动作开始思考。然后，每个动作可以（通常是异步地）决定何时准备好运行，并提供"思考输出"，如果此动作正在执行序列的一部分（即它正在执行复合动作中的一个步骤），则可以将其用作后续步骤的输入。例如，find_path_to_entity 启动异步寻路器，当找到有效路径时，声明它已准备好并将路径作为其思考输出；此路径通常作为参数传递给后续的 follow_path 活动）。对于由多个步骤组成的复合动作，每个步骤按顺序"思考"，并且所有步骤都必须声明自己准备好后，复合动作本身才被视为准备好。重要的是要注意，除非它们所在的分支已完成执行（或被中止），否则不会重新思考动作，因此动作编写者不应依赖于保证在任何时间重新启动思考，如果动作的有效性应响应世界状态而变化，动作应在思考时监听事件并相应调整其有效性。

  • 比较计划 -- 从所有可用动作中选择要执行活动的动作。动作通过宣布其"效用值"告诉 AI 系统它们有多好。这可以是一个对所有动作实例始终相同的静态值，也可以是一个范围，动作实例在思考或运行时指定它在该范围内的位置。请注意，选择是针对每个有多个实现的活动本地发生的，并且是即时发生的。活动的所有实现同时开始思考，并并发地进行（嗯，异步地），每当动作准备好时，如果其效用值高于当前为该活动选择的"最佳"动作，它将替换该动作。如果动作一直在运行，其效用值会获得"沉没成本"提升以减少中断，但如果它仍然低于新准备好的动作，它会被中断并且新动作开始。

    例如，healing_task_group 执行 stonehearth:work 活动并具有 0.87 的恒定效用值；crafting_task_group 也执行 stonehearth:work 活动但具有 0.81 的恒定效用值，因为我们希望治疗优先于制作。因此，当草药师决定工作时，他们将开始考虑治疗（通过寻找受伤的市民）和制作（通过查看任何可用的草药师制作任务）。如果他们开始制作但后来发现受伤的市民，他们将中断制作任务组动作，转而开始执行治疗任务组。

  • 执行计划 -- 在执行 stonehearth:top 的最佳准备好动作上调用 run()。这几乎总是一个复合动作，因此执行它涉及调用其序列中每个动作的 run()，其中许多本身将是复合或任务组动作，因此最终它包括按顺序运行一系列叶子动作。

以下是 AI 检查器中典型 AI 树的外观截图。活动（=帧）以尖括号呈现。其余节点是动作。叶节点是叶子动作。内部节点是复合、任务组或任务动作。根始终是 stonehearth:top 活动。

编写动作

动作是在 mod 清单中注册并"注入"到角色 AI 中的 Lua 脚本。本节探讨如何声明动作。下一节探讨如何将动作注入到角色的 AI 中。

叶子动作示例

（假设的）draw_animal_action.lua

  local DrawAnimalAction = class()

  DrawAnimalAction.name = 'draw an animal' -- 必需
  DrawAnimalAction.does = 'stonehearth:paint' -- 必需
  DrawAnimalAction.args = { -- 必需
     subject = Entity,
     effect_name = {
        type = 'string',
        default = 'draw_in_notebook',
     }
  }
  DrawAnimalAction.think_output = {}
  DrawAnimalAction.priority = {0.2, 0.4} -- 必需
  DrawAnimalAction.sunk_cost_boost = 0.2
  DrawAnimalAction.weight = 1
  DrawAnimalAction.scheduler_priority = 4
  DrawAnimalAction.status_text_key = 'stonehearth:ai.actions.status_text.drawing'

  function DrawAnimalAction:start_thinking(ai, entity, args)
     if is_animal(args.subject) then
        ai:set_utility(get_photogenicity(args.subject))
        self._listener = radiant.events.listen(args.subject, 'stonehearth:ready_to_pose',
                                                function()
                                                   ai:set_think_output({})
                                                end)
     else
        ai:reject('not an animal!')
     end
  end

  function DrawAnimalAction:stop_thinking(ai, entity, args)
     if self._listener then
        self._listener:destroy()
        self._listener = nil
     end
  end

  function DrawAnimalAction:start(ai, entity, args)
     if not is_ready_to_pose(args.subject)
        ai:abort('animal stopped being ready to pose by the time the action started')
     end
  end

  function DrawAnimalAction:run(ai, entity, args)
     if not is_ready_to_pose(args.subject)
        ai:abort('animal stopped being ready to pose by the time the action ran')
     end

     freeze_animal(args.subject)
     ai:execute('stonehearth:run_effect', {effect = args.effect_name})
     radiant.entities.add_thought(entity, 'stonehearth:thoughts:drew_an_animal')
  end

  function DrawAnimalAction:stop(ai, entity, args)
     if is_frozen(args.subject) then
        unfreeze_animal(args.subject)
     end
  end

  return DrawAnimalAction

让我们看看该动作的每个部分：

  • 元数据

    • name -- 动作在调试工具中显示的名称。

    • does -- 此动作执行的活动。

      在这种情况下，我们的动作是执行 stonehearth:paint 的一种方式。

    • args -- 此活动的参数类型；所有执行相同活动的动作必须采用相同的参数。如果参数是可选的，它应该声明一个默认值。这通常为空。

      在这种情况下，我们将要绘制的主体作为参数，这可能来自复合动作中的前一步骤或任务，以及一个可选的效果名称参数，以防我们想在绘制时播放不同的动画。

    • think_output -- 可选。传递给 set_think_output() 的对象的格式。如果没有传递对象或传递空对象，则不需要声明此项。

      在这种情况下，我们没有思考输出。

    • priority -- 动作的效用值，即此动作执行活动的效果如何，从 0.0 到 1.0。如果动作始终具有相同的效用值，这可以是一个数字，或者如果动作将根据状态或输入通过调用 ai:set_utility() 更改其效用值，则可以是一个范围，如 {0.2, 0.6}。动作的效用值用于在准备好的动作中选择执行活动的最佳动作。

      在这种情况下，我们指定将动态设置效用值，并且我们传递给 set_utility() 的值应线性重新映射到 {0.2, 0.4} 范围，因此 0 将重新映射到 0.2，0.5 到 0.3，1.0 到 0.4，等等。

    • sunk_cost_boost -- 可选，罕见。对动作运行一段时间后添加到其效用值的量的覆盖。这控制了动作被中断的难易程度。默认值为 0.05。值为 1.0 意味着没有执行相同活动的动作可以中断它（但将此动作作为步骤运行的复合动作仍然可以被中断）。

      在这种情况下，我们指定了 0.2，这意味着要使另一个绘画任务中断此任务，它的效用值必须比我们的效用值高 0.2 以上。

    • weight -- 可选，罕见。对于具有相同效用值的动作，权重用于随机化哪个将运行。

      在这种情况下，我们指定了 1，这是默认值，所以这什么都不做，仅用于说明目的。

    • scheduler_priority -- 可选，罕见。运行此动作时为此实体分配的寻路等资源量的乘数。截至 2017 年 9 月，这仅用于帮助空闲角色找到其他动作。

      在这种情况下，我们指定了 4，我们没有充分的理由这样做，只是为了演示此选项的存在。

    • status_text_key -- 可选。用于查找面向用户的此动作描述的 i18n 键。这在市民菜单中显示以描述给定角色正在做什么。这也可以通过在动作的 run() 方法中调用 ai:set_status_text_key() 来设置。

      在这种情况下，我们指定了一个可能翻译为"正在绘制动物"的键。

  • 方法

    • 注意：Action 接口中的所有方法都采用相同的参数：

      • ai -- 一个AIContext对象，用于与 AI 系统通信关于当前动作的信息。
      • entity -- 正在思考或运行动作的实体。请注意，每个"知道"动作的实体都有自己的实例。
      • args -- 传递给活动的参数。

    • start_thinking() -- 当 AI 系统开始规划活动（例如，由复合或任务组动作提示）时，所有执行该活动的动作都将收到对此方法的调用。只有在调用 ai:set_think_output() 后（或者如果没有 start_thinking() 方法），动作才被视为准备好。这可以在 start_thinking() 运行时同步发生，也可以作为事件的结果在稍后时间发生——在这种情况下，start_thinking() 是开始监听该事件的地方。

      在这种情况下，我们查看主体参数。如果它不是动物，我们立即拒绝并给出解释。否则，我们根据动物的上镜程度更新我们的效用值（因此，例如，如果有一个 draw_landscape_behind_subject 动作在恒定优先级 0.3 下"执行" stonehearth:paint，只有更上镜的动物才会优于风景）。接下来，我们设置一个监听器以等待动物准备好摆姿势，当它准备好时，我们调用 set_think_output() 将此动作标记为准备好运行。

    • stop_thinking() -- 每当 AI 系统决定停止规划此动作时，都会调用此方法来清理在 start_thinking() 中完成的任何设置，例如销毁事件监听器和跟踪。如果动作已成功完成规划（即它调用了 set_think_output()，在这种情况下，这是在 start() 之后调用的，因此可以使用动作状态来避免清理事运行动作所需的内容），或者如果 AI 决定不执行该动作（例如，如果更高优先级的动作已准备好，或者如果 AI 完成了它正在做的事情并现在重新启动所有规划），就会发生这种情况。保证如果动作收到了对 start_thinking() 的调用，它将在任何其他 Action 接口方法之前收到对 stop_thinking() 的调用。

      在这种情况下，我们销毁了在 start_thinking() 中创建的监听器。

    • should_start_thinking() -- 在执行活动的每个动作上的 start_thinking() 之前调用，并允许动作决定此活动是否允许运行。如果执行活动的任何动作从 should_start_thinking() 返回 false，则该活动的所有动作都被阻止思考或运行。这很少使用，当使用时，通常用于特定的"过滤"动作，它们自己不执行任何操作，只是在某些情况下阻止活动运行。例如，为了防止木匠建造非木制结构，我们给所有角色通用的 fabricate_structure 动作，它执行 stonehearth:fabricate_structure，但只给木匠 fabricate_wooden_structure 过滤动作，它也执行 stonehearth:fabricate_structure。因此，如果结构不是木制的，木匠将不允许开始他们的常规制造动作，因为过滤器将返回 false。

      在这种情况下，我们不需要此方法。

    • start() -- 当选择动作用于运行时，调用此方法。通常，您可以在 run() 的开头执行您可能在此处执行的任何操作，但在两种情况下很有用：(1) 对于复合动作，它们自己没有 run() 方法，它们可以在 start() 中进行初始化，(2) 当动作本身是复合动作中的步骤时，在任何 run() 方法之前对序列中的每个动作调用 start()，因此它们可以在开始执行之前在那里进行健全性检查（尽管它们可能需要在 run() 中重复它们，因为时间已经过去）。

      在这种情况下，我们检查动物在我们完成思考和动作实际开始之间是否不再准备好摆姿势。

    • run() -- 这是实际实现执行此动作含义的方法。与所有其他 Action 方法不同，它可以 yield/suspend 线程，并且在动作完成之前不应返回。使用中的经典示例是 run_effect 动作，它播放效果（通常是动画）并在效果持续期间 yield 线程，仅在效果完成后恢复并返回。如果在某个时刻动作确定它无法成功完成，它可以调用 ai:abort()，这将取消当前计划（停止所有进行中的动作）并从头开始重新启动规划。

      在这种情况下，我们重复在 start() 中的检查（如果此动作是复合动作中的后期步骤，这可能是很久以前）。然后我们冻结动物，并调用 ai:execute()，它立即思考并运行执行给定活动的动作（思考必须是同步的，否则我们会在思考时挂起角色的 AI 线程）。在这种情况下，我们执行 run_effect，它播放请求的动画效果（draw_in_notebook），一旦完成就恢复线程并将控制权返回给我们。最后，我们添加一个关于绘制动物的想法，并返回，这告诉 AI 系统我们已经成功完成运行此动作，可以继续下一步（如果没有更多步骤，则完成当前计划）。

    • stop() -- 每当动作停止时，都会调用此方法。这可以通过完成其 run()、被中止或被更高效用值的动作中断来实现。保证如果动作收到了对 start() 的调用，它将收到对 stop() 的调用。该方法应用于清理在 start() 或 run() 中设置的任何状态或资源。与 start() 一样，如果此动作是复合动作中的步骤，其 stop() 方法将在所有步骤完成运行后调用（以及其他步骤的 stop() 方法，按顺序）。

      在这种情况下，我们解冻我们在 run() 期间可能冻结的动物。

复合动作示例

（假设的）find_and_paint_subject_action.lua

  local FindAndPaintSubjectAction = class()

  FindAndPaintSubjectAction.name = 'find and paint subject'
  FindAndPaintSubjectAction.does = 'stonehearth:free_time'
  FindAndPaintSubjectAction.args = {}
  FindAndPaintSubjectAction.priority = {0.3, 0.7}

  function FindAndPaintSubjectAction:start_thinking(ai, entity, args)
     local traits = entity:get_component('stonehearth:traits')
     if traits and traits:has_trait('stonehearth:traits:artist') then
        ai:set_utility(0.5)
     else
        ai:set_utility(0.0)
     end
     ai:set_think_output()
  end

  function FindAndPaintSubjectAction:compose_utility(entity, self_utility, child_utilities, current_activity)
     return self_utility + child_utilities:get('stonehearth:follow_path') * 0.1 + child_utilities:get('stonehearth:paint') * 0.4
  end

  local ai = stonehearth.ai
  return ai:create_compound_action(FindAndPaintSubjectAction)
              :execute('stonehearth:find_willing_painting_subject')
              :execute('stonehearth:find_path_to_entity', { destination = ai.PREV.subject })
              :execute('stonehearth:follow_path', { path = ai.PREV.path })
              :execute('stonehearth:paint', { subject = ai.BACK(3).subject })

  return FindAndPaintSubjectAction

让我们看看该动作的每个部分：

  • 元数据

    • 复合动作的所有元数据与叶子动作相同。

  • 方法

    • 复合动作具有叶子动作具有的所有方法，但有 2 个例外：

      • 复合动作没有 run() 方法。它们的 run() 隐式包括调用每个步骤的 run()。

      • 复合动作可以声明一个 compose_utility() 方法，从其步骤的效用值派生其效用值。每当任何步骤的效用值发生变化，或者我们在运行时从一个步骤转换到另一个步骤时，都会调用此方法。与其他方法不同，它有一组不同的参数（实体、由 set_utility() 设置的复合动作的效用值、动作步骤效用值的访问器，以及当前正在运行的活动的名称，如果有）。

        在此示例中，我们在 start_thinking() 中设置的此动作的效用值增加了绘画步骤的效用值（在叶子动作示例中，我们看到它代表了主体的上镜程度），以及路径跟随步骤的效用值（代表距离）。也许更好的实现方式是让 stonehearth:find_willing_painting_subject 活动始终选择最佳可能的主体，并使其效用值代表主体有多好。

      请注意，复合动作的思考在任何步骤之前完成，并且只有在复合动作成功完成其思考后，步骤才会按顺序被要求思考。其 start() 发生在任何步骤的 start() 之前，其 stop() 发生在所有步骤的 stop() 之后。

      在此示例中，我们仅使用 start_thinking() 使此动作在实体具有假设的艺术家特性时更有可能运行。

  • 步骤声明

    • 复合动作通过在 create_compound_action() 返回的对象上调用 execute() 来声明它们由哪些步骤组成。请注意，这与前面提到的 ai:execute() 方法无关。步骤是将按顺序执行以运行动作的活动。每个步骤可以指定要传递给活动的参数。参数可以是常规值，也可以是"占位符"，它们是动态引用运行动作的上下文或前一步骤结果的方式。截至 2017 年 9 月，支持的占位符是：

      • ai.ENTITY -- 运行动作的实体。

      • ai.ARGS.foo -- 复合动作的参数之一，在这种情况下是 foo。

      • ai.CALL(fn, ai.XXX, ai, ai.YYY) -- 这将所有参数评估为占位符，然后调用第一个参数中指定的函数，传递其他参数，其返回值是此占位符评估的结果。在这种情况下，它将使用两个参数调用 fn()，即评估 ai.XXX 和 ai.YYY 的结果，并使用其返回值作为结果。

      • ai.PREV.foo -- 前一步骤的"思考输出"的字段之一。例如，在这种情况下，前一步骤传递给其 set_think_output() 调用的任何对象的 foo 字段。

      • ai.BACK(n).foo -- 第 n 个前步骤的"思考输出"的字段之一。例如，在这种情况下，N 步之前的前一步骤的任何对象的 foo 字段。ai.BACK(1) 等同于 ai.PREV。

      • ai.NOT(...) -- 评估其参数并否定它。例如，ai.NOT(ai.ARGS.foo) 与此动作开始思考时 foo 参数的值的相反。

      • ai.CURRENT -- 运行此动作的实体的"预期状态"。这个有点棘手，是当其他占位符都不足以满足您的需求时的回退。任何动作都可以在思考时写入 ai.CURRENT 条目（通过设置 ai.CURRENT.foo = value），以表示他们认为动作运行后实体的状态将会是什么，随后作为步骤出现的动作将获得该修改后的对象。例如，如果拾取动作修改了角色背包的预期状态，而后续动作想要检查角色背包中有什么以决定他们是否或应该如何运行，这就很有用。以下是 ai.CURRENT 中支持的条目：

        • location -- 角色的 X,Y,Z 位置。各种路径跟随和追逐动作会修改此项。
        • carrying -- 角色携带的物品。
        • storage -- 角色背包的内容。具有一个 items 属性，它是从实体 ID 到实体的表，一个 full 属性指示背包是否已满，以及 add_item 和 remove_item 方法。
        • self_reserved -- 由前面的 reserve_entity 动作保留的对象，从实体 ID 到实体的表。
        • self_region_reserved -- 由前面的 reserve_entity_destination 动作保留的对象，从实体 ID 到从偏移键到 XYZ 偏移的表的表。
        • location_changed -- 一个只读标志，指示 ai.CURRENT.location 是否在 AI 分支中的任何点发生了变化。
        • carrying_changed -- 一个只读标志，指示 ai.CURRENT.carrying 是否在 AI 分支中的任何点发生了变化。

      请注意，每个步骤的 start() 方法在执行第一个步骤的 run() 方法之前按顺序运行，每个步骤的 stop() 方法在最后一个步骤的 run() 方法返回后按顺序运行。这意味着，例如，如果您正在编写一个想要创建小动物、将角色走到小动物面前然后销毁小动物的复合动作，您可以在创建动作的 stop 函数中调用销毁，它将仅在行走发生后执行。

      在此示例中，我们的步骤如下：

      • 首先，找到一个愿意被绘制的主体。
      • 接下来，找到到达该主体的路径。在这里，我们使用占位符将前一步骤中找到的主体作为目的地传递以找到路径。
      • 然后，实际跟随路径（同样，我们使用占位符将前一步骤中找到的路径传递给此活动）。
      • 最后，实际绘制主体（我们使用 ai.BACK 占位符一直回溯到第一步中找到的主体）。这可能会运行上面描述的 draw_animal 动作，或另一个执行 stonehearth:paint 的动作，例如 draw_human 或 draw_building，具体取决于第一步中选择了什么实体，以及每个动作的效用值是多少（始终选择最高效用值的动作）。

任务组示例

solo_basic_needs_task_group.lua

  local SoloBasicNeedsTaskGroup = class()

  SoloBasicNeedsTaskGroup.name = 'solo basic needs'
  SoloBasicNeedsTaskGroup.does = 'stonehearth:top'
  SoloBasicNeedsTaskGroup.priority = {0, 0.3}

  return stonehearth.ai:create_task_group(SoloBasicNeedsTaskGroup)
                          :declare_task('stonehearth:sleep', 0.6)
                          :declare_multiple_tasks('stonehearth:eat', {0.4, 1.0})
                          :declare_permanent_task('stonehearth:rest_when_injured', {}, 0.2)

让我们看看该动作的每个部分：

  • 元数据

    • 任务组动作的所有元数据与叶子动作相同，除了它们不能有参数（仅由于缺乏用例）。

  • 方法

    • 任务组动作仅支持一种可选方法 compose_utility()，它允许您拥有自定义逻辑以从它正在考虑或运行的任务的效用值派生组的效用值，而不是默认只是将声明的任务效用值（见下文）重新映射到为任务组动作声明的范围（如果它是一个范围而不是静态数字）。与其他方法不同，它有一组不同的参数（实体和任务的效用值）。

      在这种情况下，我们不需要任何自定义效用值计算逻辑，因此我们不声明 compose_utility()。

  • 任务声明

    • 任务组的目的有两个：(1) 允许 AI 系统外部的代码控制在给定时间哪些活动处于活动状态，它们应该使用什么参数，以及哪些角色应该执行它们；(2) 从一组可能的活动中选择最佳活动来运行。第一个目的在本页的后面部分中探讨，但第二个是我们在 AI 上下文中关注的。

    • 任务组声明它们可以运行哪些活动，以及是否可以有多个该活动的实例尝试运行（例如，我可能想要两个具有不同目标且相互竞争的"攻击目标"任务）。任务可以通过调用从 create_task_group() 返回的对象上的方法来声明。有 3 种变体：

      • declare_task() -- 这声明管理任务组的人最多可以创建一个（一次）尝试执行指定活动（参数在运行时指定）的任务。第一个参数是此任务将尝试执行的活动。第二个是任务的效用值。第二个参数可以是一个数字，这意味着任务忽略执行活动的动作的效用值，并且始终认为任务在任务组的上下文中具有指定的效用值。或者，它可以是一个范围，如 {0.2, 0.7}，这意味着执行请求活动的动作的效用值将从其 0 到 1 的范围线性重新映射到指定范围。

        请记住，这些任务效用值声明是在任务组动作的上下文中，这意味着它们用于选择任务组内的哪些任务要执行。任务组本身在其活动上下文中的效用值（其在元数据中声明的"does"）是通过将此任务效用值再次重新映射到任务组元数据中指定的范围（如果声明了则通过 compose_utility()）来计算的。

      • declare_multiple_tasks() -- 这类似于 declare_task()，除了它允许创建同一任务的多个实例（可能具有不同的参数）。

      • declare_permanent_task() -- 这类似于 declare_task()，除了它接受一组参数，并在创建任务组时立即创建该任务，而不是等待某些外部代码创建该任务。

    • 在此示例中，我们声明一个处理炉民饮食、睡眠和恢复任务的"基本需求"任务组。（请注意，这是此任务组的最早、最简单版本，到您阅读此时可能不是游戏正在使用的版本）。

      • 我们声明一个 stonehearth:sleep 任务，具有 0.6 的恒定效用值。这意味着某些外部系统（在这种情况下是 sleepiness_observer）将在某个时候要求 AI 开始考虑睡眠。我们还表示我们不关心执行睡眠活动的动作的效用值，并且将始终认为该任务相对于睡眠和恢复具有 0.6 的效用值。请记住，任务组本身的效用值是任务的效用值，重新映射到任务组声明的效用值范围。因此，由于此任务具有 0.6 的恒定效用值，并且任务组声明其效用值范围为 {0, 0.3}，如果选择了睡眠任务，任务组的效用值为 0 + (0.6 * (0.3 - 0)) = 0.18。
      • 我们声明一个 stonehearth:eat 任务，效用值范围为 {0.4, 1.0}，并允许由某些外部系统（在这种情况下是消费组件）创建多个进食任务。由于我们指定了效用值范围，我们将使用执行 stonehearth:eat 的动作的效用值，并将其重新映射到此范围。因此，如果我们的进食动作说其效用值为 0.7（例如，基于饥饿度或食物质量），则任务的效用值为 0.4 + (0.7 * (1.0 - 0.4)) = 0.82，与组内的睡眠、恢复和其他进食任务相比。如果选择了该任务，任务组的效用值将为 0 + (0.82 * (0.3 - 0)) = 0.246。
      • 我们声明一个永久的 stonehearth:rest_when_injured 任务，具有空参数列表和 0.2 的效用值。这意味着具有此任务组的角色将始终拥有此任务，并且其效用值相对于进食、睡眠和恢复将始终为 0.2，使其成为最不重要的。如果选择了该任务，任务组的效用值将为 0 + (0.2 * (0.3 - 0)) = 0.06。

AI 上下文

如上所述，所有 Action 方法都传递一个"AI 上下文"，用于与该动作的 AI 系统通信。以下是 AI 上下文上可用的方法：

  • 常用方法

    • set_think_output() -- 必须在动作思考时调用以将动作标记为准备好，并设置动作的"思考输出"，这是一个表，复合动作序列中的任何后续步骤都可以访问。一个例子是 find_path_to_entity 动作将计算的路径设置为"思考输出"，以便它可以在复合动作步骤序列中与 follow_path 动作一起使用。如果没有参数传递给 set_think_output()，思考输出默认为动作的原始参数。请注意，在任何其他状态（例如，已经准备好）下调用 set_think_output() 是无效的。
    • clear_think_output() -- 撤销 set_think_output() 的效果。使用此方法将动作标记为不再准备好，将其返回到思考状态。
    • set_utility() -- 更新动作的效用值。接受 0 到 1 之间的数字，将其线性重新映射到动作声明的效用值范围，并将动作的效用值设置为该值。这可以在思考和运行时调用，但动作必须声明效用值范围而不是静态效用值。
    • abort() -- 用于宣布动作已进入不可恢复的状态，通常在运行时。例如，它正在运行到一个物品，但该物品已被销毁。这迫使 AI 取消当前动作，并从头开始重新计算新计划。这应该仅在极端情况下使用，因为它可能导致角色结巴或快速切换计划，因为它从头开始重新考虑一切。如果 AI 继续选择一个动作并且它不断中止，它最终（目前在连续 10 次中止后）将放弃并杀死角色的 AI 线程。此方法接受一个字符串解释取消的原因，该原因显示在 AI 检查器和错误消息中。
    • execute() -- 立即思考并运行执行给定活动的动作（思考必须是同步的，否则我们会在思考时挂起角色的 AI 线程）。一般来说，如果您发现自己使用此方法，请考虑您的动作是否应该是将此作为步骤的复合动作。这允许思考是异步的，并且使要执行的活动即使在动作未运行时也显示在 AI 检查器中。

  • 不常用方法

    • set_status_text_key() -- 设置 i18n 键和参数，用于查找和渲染此动作的面向用户描述，该描述显示在市民菜单中以描述给定角色正在做什么。这应该仅在您需要动态更改文本或传递参数时使用。否则，只需将其声明为 status_text_key 元数据字段。
    • reject() -- 信号表明动作收到的参数不再有效。这个有点棘手！如果执行给定活动的所有动作都拒绝，并且该活动是复合动作的一部分，则该复合动作将在（20ms 延迟后）重新思考自己。这意味着 reject() 应该仅在调用者期望成为复合动作中重新思考的步骤时调用，该步骤现在将选择不同的参数。经典的例子是 reserve_entity 动作——如果其目标在开始思考时已被保留，它会拒绝以要求父动作重新评估实体的选择（它假设它们将检查保留）。另一方面，如果您在重新思考其父动作不更改其参数时调用 reject()，这将导致无限循环的拒绝、重新思考、拒绝等，因此这是危险且高度依赖上下文的。
    • suspend() -- 暂停实体的 AI 线程直到恢复，并且只能在运行时调用。几乎专门用于在 run_effect 等动作中等待效果。如果您暂停 AI 线程，您必须稍后从同一动作（在另一个线程上；通常在回调中）调用 resume()；否则线程可能会永远挂起。
    • resume() -- 恢复 AI 线程。应该仅在运行时调用，如果此同一动作之前调用了 suspend()。
    • protect_argument() -- 将实体标记为受保护。如果在此动作思考时该实体被销毁，其思考将被取消。默认情况下，传递给动作的所有参数都受保护，因此这仅在您需要保护与您的动作相关的其他实体时有用。
    • unprotect_argument() -- 撤销由 protect_argument() 设置的保护，意味着销毁给定实体不再停止动作思考。例如，如果您需要销毁作为参数传递给动作（或之前受保护）的实体作为您动作的一部分，则使用此方法。

  • 调试方法

    • set_debug_progress() -- 设置在 AI 检查器中显示的字符串，描述动作的当前状态。例如，"查找物品"动作可以显示它选择了什么物品。
    • get_log() -- 返回一个（按需创建）配置为打印有关此动作的额外上下文信息（实体和调试路由）的记录器。这只是一个快捷方式。
    • get_debug_info_datastore() -- 返回包含有关此动作的调试元数据的数据存储。可用于添加您自己的调试元数据，但通常应避免使用。

  • 特殊方法

    • spawn()：创建一个新的执行帧，用于管理和运行给定活动。调用者负责管理返回的帧，在正确的时间调用其方法等。这仅用于实现低级 AI 系统功能，通常没有用处。
    • get_parent_node()：返回此动作在 AI 图中的父节点，可能是动作、执行帧或 AI 上下文。这仅用于实现低级 AI 系统功能，通常没有用处。

效用值

在决定选择哪个动作来执行给定活动或任务组时，通过比较它们的效用值来判断动作，效用值是 0 到 1 之间的值，表示该动作在当前情况下执行其活动的效果如何。

如果动作始终具有相同的效用值，动作可以将效用值指定为数字，或者如果动作将根据状态或输入在某个时刻更改其效用值，则指定为范围，如 {0.2, 0.6}。在后一种情况下，默认假设为最小值，动作可以在思考或运行时通过调用 ai:set_utility() 并使用 0 到 1 之间的数字（将重新缩放到声明的范围）来调整它，或者对于复合和任务组动作，通过从其 compose_utility() 方法返回新的效用值，如上所述。

请注意，动作的效用值仅在其活动上下文中才有意义。执行不同活动的动作的效用值不能直接比较。

除了动作报告的效用值外，我们还有一个"沉没成本提升"功能，它在动作运行一段时间后增加其效用值。默认情况下，它需要相当于真实时间 500ms 的游戏时间才能生效，默认情况下它增加 0.05 效用值，尽管后者数字可以被动作的元数据覆盖。

注入动作

即使动作已编写并在清单的 "aliases" 部分中注册，如果游戏中没有角色"知道"该动作，它也没有用处。为了让角色"知道"动作，必须使用 AI 包将其注入到他们的 AI 中。AI 包是 JSON 文件，通常在 stonehearth/ai/packs 下，具有 "type" : "ai_pack" 并包含以下列表：

  • 动作，上面已经解释过，以及组件。

  • 观察者，它们是类似组件的对象，通常用于在满足某些条件时创建任务，并且大多正在被淘汰， favoring 动作本身在思考期间等待条件。

    您可以在 pet_collar.json 中看到动作、观察者和组件的示例，请参见下面的链接。

  • 个人任务组，它们是仅包含此角色的组的任务组动作。上述基本需求任务组就是一个例子。下面有关任务组的更多信息。语法将是这样的（来自 celebration_ai_pack.json 的示例）：

    "task_groups": [
   "stonehearth:task_groups:solo:celebration",
   "stonehearth:task_groups:solo:building_celebration",
   "stonehearth:task_groups:solo:unit_control"
]

  • 对它们作为依赖项需要的其他 AI 包的引用。这些是递归遍历的。

有时 AI 包内联在其他 JSON 文件中而不是被引用。

实体以几种不同的方式注入 AI 包：

  • 实体可以在其 entity_data 中的 stonehearth:ai_packs 下指定 AI 包列表。例如，在 base_human.json 中，炉民的基础 mixin。

  • 具有 equipment_piece 组件的物品可以在 injected_ai 下包含内联 AI 包。例如，在 pet_collar.json 中，给予所有宠物的物品，我们有这个注入的 AI。

    职业也通过使用这条路径，通过向获得职业的角色提供"abilities"物品。例如，在 trapper_abilities.json 中。

  • 增益效果可以在 injected_ai 下包含内联 AI 包。例如，在 snared_buff.json 中，由 snare 能力应用的减益效果。

因此，在您创建动作后，您应该将其添加到 AI 包中，并确保应该知道此动作的角色在某处注入了该 AI 包。

任务

虽然让角色执行某些行为的最简单方法是创建动作，将其添加到 AI 包中，并让它履行顶级活动 stonehearth:top，但这确实有一些限制：

  • 动作将始终被视为一个选项。
  • 动作总是在知道该动作的每个角色上独立运行。
  • 无法将行为分配给尚未在游戏中的人（让他们在进入游戏后立即执行）。

任务是您超越这些限制的方式。任务是一种调度活动的结构；将其视为个人助手，其唯一工作就是催促某人（或一群人）做某事。

为了将任务与人匹配，我们将任务 housed 在任务组中。任务组实例包含一组待处理任务和一组要将这些任务给予的人。每个任务恰好属于一个任务组，而每个角色可以属于任意数量的任务组。当角色被添加到任务组时，其任务组动作被注入到他们的 AI 中。当创建新任务时，任务动作被注入到属于该组的每个角色的 AI 中，并在组的任务组动作下运行。

个人任务组

每个角色可以（并且通常确实）有仅包括角色自己的"个人"任务组。例如，这用于发布基本需求任务（饮食/睡眠/恢复），如上所示，或向军事单位发布移动命令。按照惯例，旨在成为个人的任务组以"solo"为前缀。

个人任务组在 AI 包中声明，如上所述，并通过 stonehearth:ai 组件的 get_task_group() 方法访问。

城镇任务组

除了个人任务组外，还有由城镇管理的任务组，可能包括多个角色。建筑和农场任务组就是这方面的例子。其他系统可以从城镇获取这些任务组并在其中发布任务。这些任务组在任务组 JSON 文件（例如 player_task_groups.json）中声明，并在人口派系 JSON（例如 ascendancy_population.json）中引用。如果角色的职业描述在 task_groups 下指定了组（例如 farmer_description.json），则角色会被添加到这些任务组中。

临时任务组

某些系统创建临时任务组以将任务分派给炉民的临时分组。这方面的一个例子是战斗小队，它创建一个任务组以向小队中的每个人分派命令。

发布任务

想要发布任务的系统需要获取任务组（来自 AI 组件的个人任务组、来自 Town 的城镇任务组，或在其他地方创建的临时任务组），然后调用 create_task()，传递活动和参数。仅允许该任务组动作声明的活动（如本页前面所述），如果任务组未将任务声明为"multiple"，则一次只能发布一个任务实例。当创建任务时，它以暂停状态开始，调用者可以在调用 start() 之前配置它，这将把它分派给属于其组的角色。配置方法包括：

  • times(n) -- 指定任务在完成之前应完成这么多次。默认情况下，任务将无限期地继续执行。
  • once() -- times(1) 的简写。
  • set_annotation(text) -- 指定与任务关联的文本用于调试。
  • set_source(object) -- 指定被认为是任务源的对象用于调试。
  • notify_completed(cb)、notify_interrupted(cb)、notify_destroyed(cb) -- 分别为任务完成、中断或销毁时添加回调。

任务可以随时添加到任务组中，即使任务组中还没有人。

例如，牧羊人牧场组件将为城镇中的所有牧羊人创建一个任务，当动物变得饥饿时喂养动物。来自 shepherd_pasture_component.lua：

  local feed_task = town:get_task_group('stonehearth:task_groups:herding')
                          :create_task('stonehearth:feed_pasture_animals', {pasture = self._entity})
                          :set_source(self._entity)
                          :start()

请记住在清单的 "aliases" 部分中声明您的任务组。

持久化

没有任何 AI 状态被持久化到存档游戏中，这意味着几件事，包括：

  • 角色在您重新加载存档游戏时不会恢复他们正在做的事情。
  • 如果 AI 线程进入不良状态，它将在重新加载时重置（对玩游戏有利，对调试不利）。
  • 重新加载时，您负责重新创建游戏保存时处于活动状态的任何任务。
