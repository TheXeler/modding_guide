关于服务器和客户端

服务器渲染游戏的逻辑，是 Stonehearth 世界中发生的所有事情的真相来源：有关炉民的数据、他们周围世界中的实体，以及操作这些信息的功能。服务器主要用 C++ 和 Lua 编写。

客户端渲染仅与单个玩家相关的事物：渲染器、屏幕 UI、音乐和声音，以及用于物品放置等事物的游戏内 UI。

这种结构在您进行多人游戏时很有用：服务器在一台计算机上运行，客户端在每个玩家的计算机上运行，客户端使用服务器上的状态更新自己，并在玩家想要执行某些操作时向服务器发送命令。如果来自 2 个玩家的 2 个命令冲突，服务器会处理问题并告诉所有客户端世界的新状态是什么。当玩 Stonehearth 的单人游戏时，您基本上是在一台计算机上同时运行服务器和客户端。

作为 modder，当您处理 UI 时，您显然是在客户端工作。当您处理 AI 或组件时，您显然是在服务器端工作。变得棘手的地方是当您：

a.) 尝试将数据从服务器获取到 UI 中。为此，您需要创建一个Handlebars 模板来吸收数据并渲染它，然后创建一个对象来为您从服务器提取相关数据。参见 workshop 或 citizens 视图以获取示例。

b.) 尝试将 ghost 或其他仅客户端的游戏对象渲染到世界中以帮助玩家选择某物的位置。有关此的更多数据，请阅读此页面。

向游戏添加新服务

有关添加新服务的示例，请参阅 stonehearth_tests mod（随游戏捆绑）中的 stonehearth_tests_server.lua 和 stonehearth_tests_client.lua。

如果您的服务是客户端服务，请创建客户端初始化脚本（您可以在此处看到一个示例）。如果是服务器服务，则创建服务器初始化脚本。

为此，在您的清单中，您需要添加以下内容之一（或者如果您同时有客户端和服务器服务，则两者都添加）：

  "server_init_script" : "file(my_server_init_script)",
  "client_init_script" : "file(my_client_init_script)"

它们与"aliases"或"mixintos"同级。

请注意，对于这些，我们在指向它们时不包含".lua"扩展名。请遵循此语法。

例如，在我们的服务器初始化脚本中，我们有这个：

  ui_mod = {}

  local service_creation_order = {
     { example = 'ui_mod.services.server.example.example_service' },
  }

  radiant.events.listen(ui_mod, 'radiant:init', function()
        radiant.service_creator.create_services(ui_mod, 'ui_mod', service_creation_order)
  end)

  return ui_mod

您希望添加一个监听器，监听您的 mod 上的 radiant:init 事件（上面示例中的 ui_mod）。此事件在游戏创建您的 mod 时立即触发。

在回调函数内部，您可以调用 radiant.service_creator.create_services 并传入您的 mod、您的 mod 命名空间以及您要创建的服务列表。

一旦您创建了 mod，您就可以通过 mod 命名空间访问您的服务，就像 stonehearth 服务一样。在上面的示例中，我们将这样调用它：

  ui_mod.example:function_from_the_example_service()

它是 mod 的命名空间、一个点，以及我们在 service_creation_order 变量中为服务定义的键。您可以在 stonehearth/data/constants.json 中找到现有客户端/服务器服务的列表。

覆盖现有服务

要覆盖现有服务，只需在清单中添加一个覆盖，覆盖现有服务的 Lua 文件。
